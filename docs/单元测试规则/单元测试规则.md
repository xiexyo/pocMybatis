# 🏦 零售银行 POC 项目单元测试规范

## 一、总体原则

1. **测试金字塔**：以纯单元测试为主
2. **快速反馈**：单元测试必须运行极快，不依赖外部环境（如真实数据库、网络），通过 Mock 隔离外部依赖。
3. **单一职责**：一个测试方法只验证一个业务场景或一个分支。
4. **自律性**：测试代码与生产代码同等重要，需保持整洁、无冗余、易读懂。

## 二、技术栈与依赖规范

1. **测试框架**：统一使用 **TestNG**（移除 JUnit 5 依赖）。
2. **Mock 框架**：统一使用 **Mockito**（包含在 `spring-boot-starter-test` 中）。
3. **断言工具**：优先使用 TestNG 原生断言（`assertEquals`, `assertNotNull`, `assertTrue` 等）。
4. **Spring 集成**：仅在需要启动 Spring 上下文时使用 `AbstractTestNGSpringContextTests` 或 `@SpringBootTest`。

## 三、命名规范

### 1. 类命名
- **Service 层（纯单元测试）**：`{ClassName}Test.java`（例如：`AccountServiceImplTest.java`）
- **Repository/Mapper 层（集成测试）**：`{ClassName}IntegrationTest.java`（例如：`AccountRepositoryImplIntegrationTest.java`）
- **Controller 层（接口测试）**：`{ClassName}ControllerTest.java`

### 2. 方法命名
采用 `test{MethodName}_{Scenario}_{ExpectedResult}` 的格式，并配合 TestNG 的 `description` 属性提供中文描述。

```java
@Test(description = "开户失败-客户不存在")
public void testOpenAccount_CustomerNotFound_ThrowBizException() {
    // ...
}
```

## 四、分层测试策略

### 1. Service 层（纯单元测试 - 核心）
- **原则**：**绝对不能启动 Spring 容器**，不连接数据库。所有的外部依赖（Repository、其他 Service）全部通过 Mockito Mock。
- **编写模式**：
    - 继承或手动初始化 `MockitoAnnotations.openMocks(this)`。
    - 使用 `@InjectMocks` 注入被测 Service。
    - 使用 `@Mock` 注入其依赖的 Repository。
    - 遵循 **Given-When-Then** 模式。

```java
// 标准示例
@Test(description = "正常开户-二类户")
public void testOpenAccount_Success() {
    // Given: 准备测试数据和Mock行为
    when(customerRepository.findByCustomerId("C001")).thenReturn(mockCustomer);
    
    // When: 执行被测方法
    OpenAccountResponse response = accountService.openAccount(request);
    
    // Then: 验证结果和交互
    assertEquals(response.getAccountStatus(), "ACTIVE");
    verify(accountRepository, times(1)).save(any());
}
```

### 2. 异常测试规范
对于自定义业务异常 `BizException`，必须同时校验异常**被抛出**以及**错误码/错误信息**正确。

```java
@Test(description = "开户失败-客户不存在", expectedExceptions = BizException.class)
public void testOpenAccount_CustomerNotFound() {
    // Given
    when(customerRepository.findByCustomerId(anyString())).thenReturn(null);
    
    // When & Then: 捕获异常验证Code后，重新抛出以匹配 expectedExceptions
    try {
        accountService.openAccount(request);
    } catch (BizException e) {
        assertEquals(e.getCode(), "CUST404"); // 必须断言错误码
        throw e; 
    }
}
```

### 3. Repository/Mapper 层（集成测试）
- **原则**：需要真实数据库交互，必须启动 Spring 容器，使用项目自带的 H2 内存数据库。
- **编写模式**：
    - 继承 `AbstractTestNGSpringContextTests`。
    - 使用 `@SpringBootTest` 注解。
    - 使用 `@Sql` 注解在测试前清理和准备数据。

```java
@SpringBootTest
public class AccountRepositoryImplIntegrationTest extends AbstractTestNGSpringContextTests {
    
    @Autowired
    private AccountRepository accountRepository;

    @Test
    @Sql(scripts = "/cleanup.sql", executionPhase = Sql.ExecutionPhase.BEFORE_TEST_METHOD)
    public void testSaveAndFind() {
        // ...
    }
}
```

### 4. Controller 层（API 集成测试）
- **原则**：验证 HTTP 请求路由、参数校验（`@Valid`）、全局异常处理及 JSON 序列化。
- **编写模式**：使用 `MockMvc` 结合 `AbstractTestNGSpringContextTests`。

## 五、Mock 与数据规范

### 1. Mock 数据构建
- 禁止在测试方法中手写多行 `set` 方法构建复杂对象，必须使用 Lombok 的 `@Builder` 模式。
- 核心字段必须赋明确值，非核心字段可省略，但禁止赋无意义的乱码（如 `asdfgh`）。

```java
// ✅ 推荐
AccountEntity entity = AccountEntity.builder()
        .accountNo("622588123456")
        .customerId("C001")
        .accountStatus("ACTIVE")
        .build();

// ❌ 禁止
AccountEntity entity = new AccountEntity();
entity.setAccountNo("asdfghqwer"); // 无意义数据
entity.setCustomerId("1"); // 边界不清数据
```

### 2. 生命周期管理
使用 TestNG 的注解进行资源初始化和清理，防止测试间数据污染：

```java
private AutoCloseable closeable;

@BeforeMethod
public void setUp() {
    closeable = MockitoAnnotations.openMocks(this); // 初始化 Mockito
    // 其他初始化操作...
}

@AfterMethod
public void tearDown() throws Exception {
    closeable.close(); // 释放 Mockito 资源
}
```

### 3. Matcher 规范
- 当 Mock 方法时，要么全部使用**确切参数**，要么全部使用 **Matcher**（如 `any()`、`anyString()`），绝对禁止混用。
- 对于需要精确捕获的参数，使用 `ArgumentCaptor`。

```java
// ❌ 禁止混用
when(mapper.find("exactString", any())).thenReturn(...);

// ✅ 推荐
when(mapper.find(eq("exactString"), any())).thenReturn(...);
```

## 六、单元测试覆盖率要求

1. **Service 核心业务逻辑**：代码覆盖率必须达到 **80%** 以上，分支覆盖率（如 if/else）必须达到 **100%**。
2. **异常分支测试**：每一个 `throw new BizException` 都必须有对应的单元测试用例覆盖。
3. **边界值测试**：金额类（`BigDecimal`）测试必须覆盖 `0`、`负数`（如果业务允许）、`极大值` 等边界情况。
