# 一、H2 表修改的核心原则

## 1. 你的项目配置回顾

```yaml
spring:
  jpa:
    hibernate:
      ddl-auto: none   # 关键：Hibernate 不会自动改表
  sql:
    init:
      mode: always     # 每次启动都会执行 schema.sql / data.sql
```

这意味着：
- **表结构完全由 schema.sql 控制**
- Hibernate 只是校验实体映射与表是否匹配（若校验严格会报错）
- 你修改表后，必须同步改 schema.sql 和 Entity

# 二、常见场景与操作方法

## 场景1：新增字段（最简单）

比如你要给 `t_account` 增加 `freeze_reason` 字段。

### 步骤：

1. **修改 schema.sql**  
   在 `CREATE TABLE t_account` 中添加新列（注意保留已有列）：
   ```sql
   ALTER TABLE t_account ADD COLUMN freeze_reason VARCHAR(128);
   ```
   或者直接在表定义中加上，但因为表已存在，用 `ALTER` 更安全。

2. **修改对应 Entity**
   ```java
   @Column(name = "freeze_reason", length = 128)
   private String freezeReason;
   ```

3. **重启应用**  
   H2 会自动执行 `ALTER TABLE`（因为 schema.sql 里写了），实体也同步了，启动正常。

> ✅ 这是最理想的增量变更，不会丢数据。

---

## 场景2：修改字段类型 / 长度

比如把 `t_customer.mobile` 从 `VARCHAR(20)` 改成 `VARCHAR(30)`。

### 步骤：

1. **手动备份数据（可选，POC 数据少可不备）**  
   H2 控制台执行：
   ```sql
   CREATE TABLE t_customer_backup AS SELECT * FROM t_customer;
   ```

2. **修改 schema.sql**  
   H2 不支持直接改列类型？实际上支持 `ALTER COLUMN`：
   ```sql
   ALTER TABLE t_customer ALTER COLUMN mobile VARCHAR(30);
   ```

3. **重启应用**  
   H2 会执行该 DDL，实体中字段长度注解也可同步改（但 JPA 的 `length` 只用于 DDL 生成，这里不生成，所以非必须，但建议保持一致）。

> ⚠️ 注意：H2 的 `ALTER COLUMN` 可能不兼容所有类型转换，比如 `VARCHAR` 改 `INT` 会失败。若不行，用场景3的“重建表”方法。

---

## 场景3：删除字段 / 重命名 / 复杂变更（需要重建表）

比如你要删除 `t_transaction` 的 `remark` 字段。

### 步骤（保留数据）：

1. **备份数据**
   ```sql
   CREATE TABLE t_transaction_old AS SELECT * FROM t_transaction;
   ```

2. **删除原表**
   ```sql
   DROP TABLE t_transaction;
   ```

3. **在 schema.sql 中写新的表定义（不含 remark 字段）**  
   注意：`CREATE TABLE IF NOT EXISTS` 会重新建表，数据已丢失，所以要先备份。

4. **将备份数据插回新表**
   ```sql
   INSERT INTO t_transaction (txn_id, request_id, ...)
   SELECT txn_id, request_id, ... FROM t_transaction_old;
   ```

5. **删除备份表**
   ```sql
   DROP TABLE t_transaction_old;
   ```

6. **重启应用**  
   建议在 H2 Console 中一次性执行完所有 SQL。

> 对于 POC，数据量很小，这完全可以接受。

# 三、最简单的“偷懒”方法（适合 POC，不保留数据）

如果你不关心现有测试数据（比如刚起步，还没多少数据），可以直接：

1. **删除项目根目录下的 `data/` 文件夹**（里面是 H2 的数据库文件）
2. **修改 `schema.sql` 和 `data.sql` 到你想要的新结构**
3. **重启应用**  
   H2 会重新创建数据库，执行全新的 schema.sql 和 data.sql。

✅ 这是最快、最干净的“表结构重构”方式，**强烈推荐用于 POC 早期**。

# 四、修改表后，Entity 与 Repository 的联动修改

## 1. 必须同步修改的地方
- **Entity**：增删改字段、注解
- **DTO**：如果接口要暴露新字段，修改对应 Request/Response
- **Service**：如果新字段影响业务逻辑（例如新增校验、计算）
- **Repository**：如果新增字段需要作为查询条件，可增加方法（例如 `findByFreezeReason`）

## 2. 不需要额外配置的地方
- `application.yml` 不用动
- 事务、异常处理不受影响

# 五、利用 AI 插件帮你做表修改

你现在已经有详细的需求文档和设计文档，可以这样让 AI 辅助：

> 当前工程使用 H2 文件模式，数据库表结构由 schema.sql 管理。  
> 请帮我为 `t_account` 表增加 `freeze_reason VARCHAR(128)` 字段，并同步更新：
> 1. `schema.sql` 中添加 ALTER TABLE 语句（或修改 CREATE TABLE）
> 2. `AccountEntity` 中增加对应属性
> 3. 账户详情查询响应 DTO `AccountDetailResponse` 中增加该字段
> 4. 开户时该字段默认为 null，无需额外逻辑
> 5. 注意保持 H2 兼容的 DDL 语法。

AI 可以一次性生成你需要的代码和 SQL 变更。

# 六、推荐的工作流程（POC 阶段）

| 变更类型 | 推荐做法 | 是否保留数据 |
|---------|----------|--------------|
| 新增字段（允许 NULL） | 直接 ALTER TABLE + 改 Entity | ✅ 保留 |
| 修改字段长度/类型 | 先备份，再 ALTER，不行就重建 | 可能保留 |
| 删除字段 | 重建表，或直接删 data 目录（数据不重要时） | ❌ 丢失 |
| 多表结构调整 | 删 data 目录，全新初始化 | ❌ 丢失 |
| 添加索引 | 直接 ALTER TABLE ADD INDEX | ✅ 保留 |

**对于 POC，我强烈建议：只要不是必须保留的测试数据，就删 data 目录重建**。  
这样你永远不会有“表结构不一致”的烦恼。

# 七、万一启动报错“表和实体不匹配”怎么办？

错误示例：
```
Schema-validation: missing column [freeze_reason] in table [t_account]
```

**原因**：
- `ddl-auto=none`，Hibernate 启动时会校验实体字段是否在表中存在
- 你改了实体，但 schema.sql 没同步

**解决**：
- 按上述方法执行 ALTER TABLE 或重建表
- 或者临时设置 `ddl-auto=update` 让 Hibernate 自动改表（不推荐长期，但可应急）

# 八、一句总结

## 对于 H2 + 文件模式的 POC：
- **开发阶段，优先用“删 data 目录 + 改 schema.sql + 重启”**，简单可靠
- **如果测试数据必须保留，用 ALTER TABLE 增量修改**
- **所有表结构变更必须同步更新 Entity、DTO、Service 和 schema.sql**

这样你就可以放心地继续用 AI 插件生成新功能了。
