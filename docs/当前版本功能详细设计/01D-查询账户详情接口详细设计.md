## 2. 功能描述

查询账户详情接口用于根据账户号查询账户信息，并聚合返回：

1. 账户主信息
2. 账户对应客户信息
3. 账户对应产品信息

该接口是当前已实现能力中最主要的查询类接口，也是后续余额、冻结状态、流水聚合等扩展能力的重要基础接口。

## 6. 业务规则设计

## 6.1 规则一：账户必须存在
首先根据 `accountNo` 查询账户：

```java
accountRepository.findByAccountNo(accountNo)
```

若不存在，则抛出：
- 错误码：`ACCT404`
- 错误信息：`账户不存在`

---

## 6.2 规则二：账户对应客户必须存在
获取到账户后，根据 `account.customerId` 查询客户：

```java
customerRepository.findByCustomerId(account.getCustomerId())
```

若不存在，则抛出：
- 错误码：`CUST404`
- 错误信息：`账户对应客户不存在`

---

## 6.3 规则三：账户对应产品必须存在
根据 `account.productCode` 查询产品：

```java
productRepository.findByProductCode(account.getProductCode())
```

若不存在，则抛出：
- 错误码：`PROD404`
- 错误信息：`账户对应产品不存在`

---

## 7. 处理流程设计

1. 客户端调用 `GET /api/v1/accounts/{accountNo}`
2. Controller 接收 `accountNo`
3. 调用 `accountService.getAccountDetail(accountNo)`
4. Service 调用 `accountRepository.findByAccountNo(accountNo)` 查询账户
5. 若账户不存在，抛 `ACCT404`
6. Service 根据账户中的 `customerId` 调用 `customerRepository.findByCustomerId(...)`
7. 若客户不存在，抛 `CUST404`
8. Service 根据账户中的 `productCode` 调用 `productRepository.findByProductCode(...)`
9. 若产品不存在，抛 `PROD404`
10. 聚合组装 `AccountDetailResponse`
11. 返回 `ApiResponse.success(response)`

## 8. 聚合返回设计说明

本接口并非只返回账户表数据，而是聚合了多张表的关键信息：

### 来自账户表的信息
- accountNo
- customerId
- accountName
- accountType
- productCode
- currency
- balance
- accountStatus
- branchCode
- openDate
- createdTime
- updatedTime

### 来自客户表的信息
- customerName

### 来自产品表的信息
- productName

这使得调用方无需分别查询客户和产品信息即可获得完整账户详情。

## 9. 当前查询实现说明

当前查询账户详情接口采用“分步查询 + Service 聚合”的实现方式：

1. 通过 `AccountRepository` 查询账户主数据
2. 通过 `CustomerRepository` 查询客户信息
3. 通过 `ProductRepository` 查询产品信息
4. 在 Service 层组装为 `AccountDetailResponse`

### 当前实现优点
1. 分层职责清晰
2. SQL 简单直观
3. Repository 对 Mapper 做了语义封装
4. 适合当前 POC 与演示阶段

### 当前实现特点
- 未将详情查询收敛为单条联表 SQL
- 未单独抽出 QueryRepository
- 当前更偏向可读性与演示友好性，而非极致查询性能

## 12. 当前实现特点

### 优点
1. 查询逻辑清晰
2. 聚合信息完整
3. Service 不直接依赖 Mapper
4. Repository 层职责明确
5. 适合作为后续详情类能力的基础接口
6. 易于演示账户主数据查询能力
7. 适合后续扩展为更复杂的 MyBatis XML 查询场景

### 当前特点说明
- 查询通过 `Repository + Mapper + XML` 分步加载三张表数据
- Entity 为普通 POJO
- 代码直观，适合 POC 场景
