# 当前已实现功能详细设计 - 健康检查接口

## 1. 文档目的

本文档用于说明当前工程中健康检查接口的设计与实现情况。

在阅读本文档前，请先参考：
- `docs/01-00-当前工程基线与公共设计.md`

---

## 2. 功能描述

健康检查接口用于快速验证服务是否已正常启动，适合作为：

1. 本地启动验证接口
2. 演示环境存活性检查接口
3. 后续 Swagger 调试前的连通性验证接口

该接口不依赖数据库业务逻辑，也不依赖客户、账户、产品等业务数据。

---

## 3. 接口设计

## 3.1 请求信息

### 请求方法
```http
GET
```

### 请求路径
```http
/health
```

### 请求参数
无

---

## 3.2 响应设计

### 当前返回类型
当前实现为：
```java
Map<String, Object>
```

说明：
- 当前未使用 `ApiResponse`
- 属于一个轻量化基础接口

### 当前返回字段
| 字段 | 类型 | 说明 |
|---|---|---|
| code | String | 固定 `B0000` |
| message | String | 固定 `success` |
| service | String | 服务名称 `retail-bank-aicoding-poc` |

### 返回示例
```json
{
  "code": "B0000",
  "message": "success",
  "service": "retail-bank-aicoding-poc"
}
```

---

## 4. 代码设计

## 4.1 Controller 设计

### 类名
`HealthController`

### 方法
建议当前方法形式类似：

```java
@GetMapping("/health")
public Map<String, Object> health()
```

### 职责
- 构造简单响应对象
- 返回固定服务状态信息
- 不调用 Service
- 不访问数据库

---

## 4.2 Service 设计
当前无独立 Service 层实现。

说明：
- 该接口逻辑极简
- 直接在 Controller 中处理即可
- 当前设计符合 POC 阶段需要

---

## 4.3 Repository 设计
无

---

## 5. 处理流程

1. 客户端调用 `GET /health`
2. Controller 直接构造返回结果
3. 返回固定健康状态信息

---

## 6. 当前设计特点

### 优点
1. 简单直接
2. 无外部依赖
3. 启动后即可验证服务是否可访问
4. 适合作为演示前快速探活接口

### 当前特点说明
- 没有走统一 `ApiResponse`
- 没有参数校验
- 没有业务异常处理分支
- 不涉及事务和数据库

---

## 7. 当前实现边界

当前未扩展以下能力：

- 版本号返回
- 构建时间返回
- 数据库连通性检查
- 外部依赖可用性检查
- JVM / 线程 / 内存健康信息
- Spring Boot Actuator 集成

---

## 8. 后续可选优化方向

如果后续需要增强，可考虑：

1. 统一改为 `ApiResponse`
2. 增加数据库状态检查
3. 增加应用版本信息
4. 增加启动时间信息
5. 增加外围依赖探测信息

---

## 9. 一句话总结

健康检查接口是当前工程中最基础的存活性接口，设计轻量、无依赖，主要用于验证服务启动成功和接口访问正常。
