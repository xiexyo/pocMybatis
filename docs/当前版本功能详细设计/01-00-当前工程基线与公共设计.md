## 4. 当前工程架构

项目当前采用分层架构：

```text
Controller -> Service -> Repository -> Mapper -> MyBatis XML -> H2 Database
```

### 各层职责

#### Controller 层
- 提供 REST API
- 接收请求参数
- 调用 Service
- 返回接口响应

#### Service 层
- 承担业务逻辑
- 校验业务规则
- 控制事务边界
- 调用 Repository 访问数据

#### Repository 层
- 对上提供仓储语义封装
- 屏蔽底层 Mapper 和 SQL 访问细节
- 可对多个 Mapper 的访问进行统一封装
- 对外暴露 `save`、`findBy...`、`existsBy...` 等领域语义方法

#### Mapper 层
- 定义数据访问接口
- 由 MyBatis 负责接口代理实现
- 与 XML 中的 SQL 语句对应

#### Mapper XML 层
- 编写自定义 SQL
- 承担查询、插入、更新等数据库访问逻辑
- 与 Mapper 接口方法一一映射

#### Entity 层
- 作为数据库表结构的映射对象
- 当前为普通 POJO
- 不使用 JPA 注解

#### Common / Config 层
- 提供统一返回结构
- 提供统一业务异常
- 提供全局异常处理

## 8. 当前公共能力设计

# 8.1 统一返回结构

## 类名
`ApiResponse<T>`

## 设计目的
统一业务接口响应结构，便于前后端和后续 AI 代码生成保持一致风格。

## 字段
| 字段 | 类型 | 说明 |
|---|---|---|
| code | String | 响应码 |
| message | String | 响应描述 |
| data | T | 响应数据 |

## 当前工厂方法
- `ApiResponse.success(T data)`
- `ApiResponse.success()`
- `ApiResponse.fail(String code, String message)`

## 当前成功码
- `B0000`

---

# 8.2 业务异常

## 类名
`BizException`

## 设计目的
用于在 Service 层统一表达业务错误。

## 使用方式
```java
throw new BizException("错误码", "错误信息");
```

---

# 8.3 全局异常处理

## 类名
`GlobalExceptionHandler`

## 当前处理范围

### 1）BizException
- 返回 HTTP 400
- 返回 `ApiResponse.fail(ex.getCode(), ex.getMessage())`

### 2）MethodArgumentNotValidException
- 返回 HTTP 400
- 返回码：`P0001`
- 返回信息：字段名 + 校验失败原因

### 3）Exception
- 返回 HTTP 500
- 返回码：`S9999`

## 9. 当前 MyBatis 基线说明

### 9.1 Mapper 扫描
当前工程通过启动类进行 Mapper 扫描：

```java
@MapperScan("com.bank.poc.mapper")
```

### 9.2 XML 位置
当前 Mapper XML 统一放置在：

```text
classpath:mapper/*.xml
```

### 9.3 字段映射策略
当前 MyBatis 配置启用了：

```yaml
map-underscore-to-camel-case: true
```

因此数据库字段与 Java 字段映射规则为：

| 数据库字段 | Java字段 |
|---|---|
| customer_id | customerId |
| customer_name | customerName |
| created_time | createdTime |

### 9.4 当前实现约束
- 所有数据库访问统一通过 `Mapper + XML`
- Service 层不直接调用 Mapper
- 数据访问统一通过 Repository 对 Mapper 进行封装
- 不使用 `Repository` 的 JPA 实现方式
- 不使用 JPA 实体注解
- 不在 Java 代码中拼接 SQL

## 10. 当前 Repository 层设计说明

### 10.1 设计目的
在 MyBatis Mapper 之上增加一层 Repository，用于：

- 统一封装数据访问语义
- 降低 Service 对底层 Mapper 的直接依赖
- 隐藏 SQL 查询细节
- 为后续复杂查询、聚合读取、数据源切换预留扩展空间

### 10.2 当前职责边界
Repository 层当前主要负责：

- 单表保存操作封装
- 单表主键/业务键查询封装
- 存在性判断封装
- 与账户校验相关的简单聚合查询封装

### 10.3 当前示例方法
以当前工程为例，Repository 层对外可提供如下语义方法：

- `CustomerRepository.save(...)`
- `CustomerRepository.findByCustomerId(...)`
- `CustomerRepository.existsByCertNo(...)`
- `ProductRepository.findByProductCode(...)`
- `AccountRepository.save(...)`
- `AccountRepository.findByAccountNo(...)`
- `AccountRepository.existsActiveClassOneAccount(...)`

说明：
- Repository 是当前工程的数据访问封装层
- Mapper 仍然保留，作为底层 SQL 访问接口
- Repository 并非 Spring Data JPA Repository，而是当前工程自定义仓储层

- `ACCT404`：账户不存在

- 后续扩展功能会新增更多错误码
