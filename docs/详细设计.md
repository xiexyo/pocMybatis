
# 零售银行 AICoding POC - 后续接口详细设计

## 1. 文档目的

本文档用于描述“账户入金、行内转账、交易流水查询、幂等控制”的技术实现方案，为 AI Coding 插件生成 Controller、DTO、Service、Repository、Entity、SQL 变更提供明确设计依据。

---

## 2. 设计原则

### 2.1 保持现有工程风格
- 不改变当前分层结构
- 不引入额外复杂框架
- 使用现有 ApiResponse / BizException / GlobalExceptionHandler

### 2.2 事务一致性优先
资金类操作必须确保：
- 账户余额更新成功
- 交易流水落库成功
- 幂等状态更新符合预期

### 2.3 设计以 POC 可实现为主
- 以单体应用 + 单数据库事务为基础
- 不处理分布式场景
- 不接入消息队列

### 2.4 代码生成友好
设计尽量清晰、直白、低耦合，方便 AI 插件一次性生成完整代码。

## 4. 数据模型设计

## 4.1 现有表复用原则

本轮尽量复用现有表：

- t_account：账户余额维护
- t_transaction：交易流水
- t_idempotent_record：幂等控制

---

## 4.2 t_transaction 表建议增强

为了支持：
- 幂等成功后的结果重放
- 从查询账户视角返回余额变化后结果

建议对交易表新增两个字段：

```sql
ALTER TABLE t_transaction ADD COLUMN debit_balance_after DECIMAL(18,2);
ALTER TABLE t_transaction ADD COLUMN credit_balance_after DECIMAL(18,2);
```

### 字段含义
- debit_balance_after：借方账户交易后的余额
- credit_balance_after：贷方账户交易后的余额

### 设计原因
1. 转账成功后，需要返回转出与转入账户交易后余额
2. 重复 requestId 请求成功时，需要能够返回“首次成功结果”
3. 查询交易流水时，需要能按查询账户视角返回 balanceAfter

---

## 4.3 索引建议

为便于交易查询，可增加如下索引：

```sql
CREATE INDEX IF NOT EXISTS idx_transaction_debit_account_no
    ON t_transaction(debit_account_no);

CREATE INDEX IF NOT EXISTS idx_transaction_credit_account_no
    ON t_transaction(credit_account_no);

CREATE INDEX IF NOT EXISTS idx_transaction_txn_time
    ON t_transaction(txn_time);
```

说明：
- H2 POC 阶段不是必须，但建议补齐
- 若 AI 插件修改 schema.sql，应保持 IF NOT EXISTS 风格

## 4.4 TransactionEntity 变更

TransactionEntity 需同步新增字段：

- debitBalanceAfter：BigDecimal
- creditBalanceAfter：BigDecimal

示意：

@Column(name = "debit_balance_after", precision = 18, scale = 2)

@Column(name = "credit_balance_after", precision = 18, scale = 2)

## 4.5 t_idempotent_record 复用设计

当前表结构足以满足本轮要求，不强制变更。

## 7.3 幂等服务设计

建议新增：

```java
public interface IdempotentService {
    void init(String requestId, String businessType, String businessKey);
    void markSuccess(String requestId, String businessType, String responseCode, String responseMessage);
    void markFailed(String requestId, String businessType, String responseCode, String responseMessage);
}
```

实现类：

```java
IdempotentServiceImpl
```

### 设计说明
为避免主业务事务回滚导致幂等记录状态丢失，建议 `IdempotentServiceImpl` 的 init / markSuccess / markFailed 使用独立事务（REQUIRES_NEW）。

### 重复请求处理策略
在主业务执行前：
1. 先查询是否已存在幂等记录
2. 若不存在，则初始化 PROCESSING
3. 若存在：
    - SUCCESS：查询交易表并返回原结果
    - PROCESSING：抛出 BizException("IDEMP001", "请求正在处理中")
    - FAILED：抛出 BizException("IDEMP003", "请求已失败，请更换requestId重试")

说明：
- “返回原结果”依赖 requestId 查询交易流水
- 交易表新增余额快照字段后，可精确重放成功响应

## 8. Repository 设计

## 8.1 AccountRepository 增强

为解决并发转账、并发入金的余额一致性问题，建议增加悲观锁查询方法：

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select a from AccountEntity a where a.accountNo = :accountNo")
Optional<AccountEntity> lockByAccountNo(@Param("accountNo") String accountNo);
```

### 说明
- 入金时锁定目标账户
- 转账时锁定转出、转入账户

### 设计目的

## 9. 详细流程设计

## 9.1 入金处理流程

### 9.1.1 主流程
1. Controller 接收请求
2. 校验 requestId、accountNo、amount
3. 查询幂等记录：
    - SUCCESS -> 返回原结果
    - PROCESSING -> 抛处理中异常
    - FAILED -> 抛失败异常
    - 不存在 -> 初始化幂等记录为 PROCESSING
4. 使用悲观锁查询账户
5. 校验账户存在且状态为 ACTIVE
6. 校验 amount > 0
7. 计算新余额：balanceAfter = balance + amount
8. 更新账户余额与 updatedTime
9. 新增交易流水：
    - txnType = DEPOSIT
    - debitAccountNo = null
    - creditAccountNo = accountNo
    - creditBalanceAfter = balanceAfter
    - txnStatus = SUCCESS
10. 更新幂等记录为 SUCCESS
11. 返回响应

### 9.1.2 异常流程
若业务校验失败或系统异常：
1. 更新幂等记录为 FAILED
2. 抛出异常，由 GlobalExceptionHandler 统一返回

---

## 9.2 行内转账处理流程

### 9.2.1 主流程
1. Controller 接收请求
2. 校验 requestId、两个账户号、amount
3. 查询幂等记录：
    - SUCCESS -> 返回原结果
    - PROCESSING -> 抛处理中异常
    - FAILED -> 抛失败异常
    - 不存在 -> 初始化为 PROCESSING
4. 校验 debitAccountNo != creditAccountNo
5. 为避免死锁，按账户号字典序排序后依次加锁查询两个账户
6. 获取实际转出账户、转入账户对象
7. 校验两个账户均存在
8. 校验两个账户状态均为 ACTIVE
9. 校验币种一致
10. 校验转出账户余额充足
11. 计算：
- debitBalanceAfter = debitBalance - amount
- creditBalanceAfter = creditBalance + amount
12. 更新两个账户余额与 updatedTime
13. 新增交易流水：
- txnType = INTERNAL_TRANSFER
- debitAccountNo = 转出账户
- creditAccountNo = 转入账户
- debitBalanceAfter = 转出后余额
- creditBalanceAfter = 转入后余额
- txnStatus = SUCCESS
14. 更新幂等记录为 SUCCESS
15. 返回响应

### 9.2.2 异常流程
若任一步失败：
1. 更新幂等记录为 FAILED
2. 抛出 BizException 或系统异常

---

## 9.3 交易查询流程

### 9.3.1 主流程
1. Controller 接收查询参数
2. 校验 accountNo 必填
3. 校验 accountNo 存在
4. 校验时间范围合法
5. 通过 Specification 构造查询条件
6. 查询满足条件的分页结果，按 txnTime desc 排序
7. 将每条交易转换为 TransactionItemResponse：
    - 若 accountNo = debitAccountNo，则 direction = DEBIT，balanceAfter = debitBalanceAfter
    - 若 accountNo = creditAccountNo，则 direction = CREDIT，balanceAfter = creditBalanceAfter
    - counterpartyAccountNo 为另一侧账户号
8. 返回分页结构

## 13. 并发控制设计

## 13.1 入金
锁定目标账户：

- lockByAccountNo(accountNo)

防止并发入金导致余额覆盖。

---

## 13.2 转账
需锁定两个账户。

### 死锁规避方案
按账户号排序后加锁：
1. first = min(debitAccountNo, creditAccountNo)
2. second = max(debitAccountNo, creditAccountNo)

然后按顺序获取锁。

### 目的
避免两个并发转账请求交叉锁定顺序不一致造成死锁。

## 7.3 幂等服务设计

建议新增：

```java
public interface IdempotentService {
    void init(String requestId, String businessType, String businessKey);
    void markSuccess(String requestId, String businessType, String responseCode, String responseMessage);
    void markFailed(String requestId, String businessType, String responseCode, String responseMessage);
}
```

实现类：

```java

```

### 设计说明
为避免主业务事务回滚导致幂等记录状态丢失，建议 `IdempotentServiceImpl` 的 init / markSuccess / markFailed 使用独立事务（REQUIRES_NEW）。

### 重复请求处理策略
在主业务执行前：
1. 先查询是否已存在幂等记录
2. 若不存在，则初始化 PROCESSING
3. 若存在：
    - SUCCESS：查询交易表并返回原结果
    - PROCESSING：抛出 BizException("IDEMP001", "请求正在处理中")
    - FAILED：抛出 BizException("IDEMP003", "请求已失败，请更换requestId重试")

说明：
- “返回原结果”依赖 requestId 查询交易流水
- 交易表新增余额快照字段后，可精确重放成功响应

## 9. 详细流程设计

## 9.1 入金处理流程

### 9.1.1 主流程
1. Controller 接收请求
2. 校验 requestId、accountNo、amount
3. 查询幂等记录：
    - SUCCESS -> 返回原结果
    - PROCESSING -> 抛处理中异常
    - FAILED -> 抛失败异常
    - 不存在 -> 初始化幂等记录为 PROCESSING
4. 使用悲观锁查询账户
5. 校验账户存在且状态为 ACTIVE
6. 校验 amount > 0
7. 计算新余额：balanceAfter = balance + amount
8. 更新账户余额与 updatedTime
9. 新增交易流水：
    - txnType = DEPOSIT
    - debitAccountNo = null
    - creditAccountNo = accountNo
    - creditBalanceAfter = balanceAfter
    - txnStatus = SUCCESS
10. 更新幂等记录为 SUCCESS
11. 返回响应

### 9.1.2 异常流程
若业务校验失败或系统异常：
1. 更新幂等记录为 FAILED
2. 抛出异常，由 GlobalExceptionHandler 统一返回

---

## 9.2 行内转账处理流程

### 9.2.1 主流程
1. Controller 接收请求
2. 校验 requestId、两个账户号、amount
3. 查询幂等记录：
    - SUCCESS -> 返回原结果
    - PROCESSING -> 抛处理中异常
    - FAILED -> 抛失败异常
    - 不存在 -> 初始化为 PROCESSING
4. 校验 debitAccountNo != creditAccountNo
5. 为避免死锁，按账户号字典序排序后依次加锁查询两个账户
6. 获取实际转出账户、转入账户对象
7. 校验两个账户均存在
8. 校验两个账户状态均为 ACTIVE
9. 校验币种一致
10. 校验转出账户余额充足
11. 计算：
- debitBalanceAfter = debitBalance - amount
- creditBalanceAfter = creditBalance + amount
12. 更新两个账户余额与 updatedTime
13. 新增交易流水：
- txnType = INTERNAL_TRANSFER
- debitAccountNo = 转出账户
- creditAccountNo = 转入账户
- debitBalanceAfter = 转出后余额
- creditBalanceAfter = 转入后余额
- txnStatus = SUCCESS
14. 更新幂等记录为 SUCCESS
15. 返回响应

### 9.2.2 异常流程
若任一步失败：
1. 更新幂等记录为 FAILED
2. 抛出 BizException 或系统异常

---

## 9.3 交易查询流程

### 9.3.1 主流程
1. Controller 接收查询参数
2. 校验 accountNo 必填
3. 校验 accountNo 存在
4. 校验时间范围合法
5. 通过 Specification 构造查询条件
6. 查询满足条件的分页结果，按 txnTime desc 排序
7. 将每条交易转换为 TransactionItemResponse：
    - 若 accountNo = debitAccountNo，则 direction = DEBIT，balanceAfter = debitBalanceAfter
    - 若 accountNo = creditAccountNo，则 direction = CREDIT，balanceAfter = creditBalanceAfter
    - counterpartyAccountNo 为另一侧账户号
8. 返回分页结构

- 若重复成功请求直接返回原结果，则不需要抛 IDEMP002
- IDEMP002 可作为日志或提示性码保留，不强制对外使用

## 12. 事务设计

## 12.1 TransactionServiceImpl

```

### 设计说明

## 12.2 IdempotentServiceImpl

```java

```

### 设计原因

## 13. 并发控制设计

## 13.1 入金
锁定目标账户：

- lockByAccountNo(accountNo)

防止并发入金导致余额覆盖。

---

## 13.2 转账
需锁定两个账户。

### 死锁规避方案
按账户号排序后加锁：
1. first = min(debitAccountNo, creditAccountNo)
2. second = max(debitAccountNo, creditAccountNo)

然后按顺序获取锁。

### 目的
避免两个并发转账请求交叉锁定顺序不一致造成死锁。

## 14. 交易流水映射规则

## 14.1 入金
- debitAccountNo = null

- debitBalanceAfter = null
- creditBalanceAfter = 入金后余额
- txnType = DEPOSIT

## 14.2 行内转账
- debitAccountNo = 转出账户
- creditAccountNo = 转入账户
- debitBalanceAfter = 转出后余额
- creditBalanceAfter = 转入后余额
- txnType = INTERNAL_TRANSFER
