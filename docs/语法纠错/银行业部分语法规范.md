# 银行业代码优化规则配置方案


## 一、配置总则

本规则配置方案基于银行业监管要求及行业最佳实践制定，适用于银行业应用系统的代码开发与质量管控。配置遵循以下核心原则：

- **安全优先**：以数据安全和系统安全为最高准则，所有规则配置均以防范安全风险为首要目标
- **合规驱动**：全面遵循中国人民银行及国家金融监督管理总局发布的相关法规和标准
- **标准先行**：依托国家标准（GB/T）、金融行业标准（JR/T）及企业规范，建立统一的代码规则体系
- **全链路覆盖**：覆盖编码、构建、测试、部署全生命周期，实现质量内建
- **可量化可执行**：所有规则均具备可自动化检测的能力，支持质量门禁卡点管控

## 二、规则配置矩阵

规则按优先级分为三级：**阻断级（Block）** —— 必须修复方可进入下一环节；**严重级（Critical）** —— 必须在版本发布前修复；**建议级（Info）** —— 建议修复，非强制。


### 2.1 命名规范规则

**优先级：阻断级（Block）**

| 规则ID | 规则描述 | 反例示例 | 修正建议 |
|--------|----------|----------|----------|
| NAM-001 | 代码中的命名不能以下划线或美元符号开始或结束 | `_name`、`$value` | 删除前导或末尾的下划线/美元符号 |
| NAM-002 | 严禁使用拼音与英文混合的方式，更不允许直接使用中文 | `userMingCheng`、`用户名称` | 使用完整英文单词组合，如`userName` |
| NAM-003 | 类名使用UpperCamelCase风格，遵从驼峰形式 | `userController` | 改为`UserController` |
| NAM-004 | 方法名、参数名、成员变量、局部变量使用lowerCamelCase | `GetUserInfo` | 改为`getUserInfo` |
| NAM-005 | 常量命名全部大写，单词间用下划线隔开 | `maxCount` | 改为`MAX_COUNT` |
| NAM-006 | 抽象类命名以Abstract或Base开头；异常类以Exception结尾 | `User`（抽象类） | 改为`AbstractUser`或`BaseUser` |
| NAM-007 | 包名统一使用小写，单数形式，单词间以点分隔 | `com.Bank.Service` | 改为`com.bank.service` |
| NAM-008 | POJO类中布尔类型变量不加is前缀 | `Boolean isSuccess` | 改为`Boolean success` |
| NAM-009 | Service/DAO层方法命名规约：get前缀获取单对象，list前缀获取多对象，save/insert前缀插入，remove/delete前缀删除，update前缀修改 | `fetchUserById()` | 改为`getUserById()` |

### 2.2 注释规范规则

**优先级：建议级（Info），但关键模块文档化注释为阻断级（Block）**

| 规则ID | 规则描述 | 检查要点 | 修正建议 |
|--------|----------|----------|----------|
| CMT-001 | 所有公共类、接口、方法必须有Javadoc注释 | 类级别缺少Javadoc | 添加类功能描述和作者信息 |
| CMT-002 | 接口类中的方法和属性不加任何修饰符号（public也不要加），并加上有效的Javadoc注释 | 接口方法有多余修饰符 | 删除多余修饰符，保留简洁性 |
| CMT-003 | 复杂业务逻辑必须有行内注释说明 | 复杂if/switch/循环无注释 | 添加逻辑说明注释 |
| CMT-004 | 代码维护注释：任何修改必须记录变更原因和日期 | 无变更记录 | 添加修改人和修改日期注释 |
| CMT-005 | TODO/FIXME等标记必须在规定时间内处理 | 长期未处理的待办标记 | 创建工单并关联处理 |

### 2.3 代码安全规范规则

**优先级：阻断级（Block）**

依据《中国人民银行业务领域数据安全管理办法》（中国人民银行令〔2025〕第3号），数据处理者应当履行数据安全保护义务，防范业务数据被篡改、破坏、泄露或者非法获取、非法利用等风险。同时，依据《银行保险机构数据安全管理办法》（金规〔2024〕24号），银行保险机构应当将数据安全保护纳入信息系统开发生命周期框架。

| 规则ID | 规则描述 | 检查要点 | 修正建议 |
|--------|----------|----------|----------|
| SEC-001 | 日志输出禁止包含敏感信息（身份证号、银行卡号、手机号等） | log.info/error中包含敏感字段 | 对敏感信息进行脱敏处理或禁止输出 |
| SEC-002 | 硬编码密钥、密码、token、证书等禁止出现在代码中 | 代码中存在明文密码 | 使用配置中心或密钥管理系统（KMS） |
| SEC-003 | SQL语句必须使用参数化查询或预编译，禁止字符串拼接 | `"SELECT * FROM user WHERE id=" + userId` | 改用PreparedStatement |
| SEC-004 | 从用户输入获取的数据必须进行校验和过滤 | 直接使用request.getParameter() | 添加参数校验和XSS过滤 |
| SEC-005 | 敏感数据（身份证、银行卡、手机号）必须加密存储 | 明文存储敏感字段 | 使用国密算法SM4等加密 |
| SEC-006 | 金融交易金额必须使用BigDecimal，禁止使用float/double | `double amount = 0.1 + 0.2` | 改用`BigDecimal`并进行精度校验 |
| SEC-007 | 禁止使用不安全的反序列化方式 | ObjectInputStream直接反序列化 | 使用安全的序列化方案或进行白名单校验 |
| SEC-008 | 随机数生成使用SecureRandom，禁止使用Random | `new Random().nextInt()` | 改用`SecureRandom` |
| SEC-009 | 错误信息不得向客户端暴露敏感技术细节 | 异常堆栈直接返回前端 | 统一异常处理，返回通用错误码 |

### 2.4 性能与资源管理规则

**优先级：严重级（Critical）**

| 规则ID | 规则描述 | 检查要点 | 修正建议 |
|--------|----------|----------|----------|
| PERF-001 | 数据库连接、文件流、网络连接必须在finally块或try-with-resources中关闭 | Connection/Stream未关闭 | 使用try-with-resources或finally关闭 |
| PERF-002 | 禁止在循环中进行数据库查询 | for循环内执行SQL | 批量查询或使用JOIN/IN优化 |
| PERF-003 | 大对象（如byte[]超过1MB）禁止频繁创建 | 循环内创建大数组 | 对象池化或复用 |
| PERF-004 | 字符串拼接在循环中使用StringBuilder，禁止使用String的+拼接 | `for(int i=0;i<1000;i++){ s += i; }` | 改用`StringBuilder.append()` |
| PERF-005 | 集合初始化时指定合理初始容量 | `new ArrayList()` | 指定初始容量，如`new ArrayList<>(16)` |
| PERF-006 | 事务边界控制：事务粒度不宜过大，禁止在事务中执行RPC调用 | @Transactional中调用远程服务 | 缩小事务范围，异步处理非核心操作 |

### 2.5 并发与多线程规则

**优先级：严重级（Critical）**

| 规则ID | 规则描述 | 检查要点 | 修正建议 |
|--------|----------|----------|----------|
| CON-001 | 共享变量访问必须使用同步机制 | 多线程读写未同步的实例变量 | 使用synchronized、Lock或Atomic类 |
| CON-002 | 线程池必须自定义，禁止使用Executors创建无界线程池 | `Executors.newCachedThreadPool()` | 使用`ThreadPoolExecutor`明确参数 |
| CON-003 | 避免在锁内调用外部方法（可能导致死锁） | synchronized块内调用远程服务 | 缩小锁范围或异步处理 |
| CON-004 | 金融计算中的计数器需使用原子类 | `count++`（多线程场景） | 改用`AtomicInteger` |

### 2.6 异常处理规则

**优先级：严重级（Critical）**

| 规则ID | 规则描述 | 检查要点 | 修正建议 |
|--------|----------|----------|----------|
| EXC-001 | 禁止捕获异常后不做任何处理（空catch块） | `catch(Exception e){}` | 记录日志或向上抛出 |
| EXC-002 | 禁止使用`e.printStackTrace()` | printStackTrace输出 | 改用日志框架记录（如log.error） |
| EXC-003 | 禁止抛出RuntimeException而不携带业务上下文 | `throw new RuntimeException()` | 使用自定义业务异常并携带错误码 |
| EXC-004 | finally块中不应使用return | finally中包含return语句 | 移除finally中的return |
| EXC-005 | 金融交易异常必须记录完整的事务上下文（交易流水号、金额、时间等） | 异常日志缺少关键字段 | 补充交易上下文信息 |

### 2.7 代码质量与最佳实践规则

**优先级：建议级（Info）至严重级（Critical）**

| 规则ID | 规则描述 | 检查要点 | 修正建议 |
|--------|----------|----------|----------|
| QLT-001 | 圈复杂度不超过15 | 方法if/switch/for嵌套过深 | 拆分方法，提取子函数 |
| QLT-002 | 方法行数不超过80行 | 单一方法代码过长 | 按职责拆分 |
| QLT-003 | 类文件行数不超过2000行 | 类文件过长 | 按单一职责拆分 |
| QLT-004 | 禁止存在未使用的局部变量、参数和private方法 | 声明的变量未使用 | 删除未使用的代码 |
| QLT-005 | 禁止出现重复代码片段超过5行 | 相同或相似的代码块重复出现 | 提取公共方法或工具类 |
| QLT-006 | 魔法值（magic number）必须定义为常量 | `if(status == 1)` | 定义为`public static final int STATUS_ACTIVE = 1` |
| QLT-007 | 三元运算符禁止嵌套超过两层 | `a ? b ? c : d : e` | 改用if-else |
| QLT-008 | 枚举类命名以Enum后缀结尾，枚举成员全大写，单词间用下划线 | `enum Color {RED, GREEN}` | 改为`ColorEnum`，但成员命名规范可保持 |

### 2.8 数据安全专项规则（依据监管要求）

**优先级：阻断级（Block）**

根据《中国人民银行业务领域数据安全管理办法》，业务数据应根据敏感性进行分类分级：

| 规则ID | 规则描述 | 检查要点 | 修正建议 |
|--------|----------|----------|----------|
| DSA-001 | 个人信息（姓名+身份证号组合）禁止以明文形式在代码中传递 | 日志、缓存、内存中包含个人敏感信息组合 | 脱敏或加密处理 |
| DSA-002 | 敏感级及以上数据必须标识数据分级标签 | 无数据分级注解 | 添加`@DataLevel`注解标识级别 |
| DSA-003 | 数据传输必须采用TLS 1.2及以上加密 | HTTP明文传输 | 强制使用HTTPS |
| DSA-004 | 涉及个人信息处理的功能必须有对应的授权校验 | 无权限校验直接访问 | 添加鉴权逻辑 |

### 2.9 金融业务逻辑专项规则

**优先级：阻断级（Block）**

| 规则ID | 规则描述 | 检查要点 | 修正建议 |
|--------|----------|----------|----------|
| BIZ-001 | 金额计算必须使用BigDecimal，并指定舍入模式 | `amount1.subtract(amount2)`未指定舍入模式 | 使用`setScale(2, RoundingMode.HALF_UP)` |
| BIZ-002 | 金融交易必须具有幂等性设计 | 无防重复提交机制 | 添加幂等校验（如唯一流水号） |
| BIZ-003 | 资金类操作必须有事务完整性保证 | 分布式事务缺少补偿机制 | 使用TCC或SAGA模式 |
| BIZ-004 | 交易流水号、订单号等必须保证全局唯一 | 本地生成无唯一性保证 | 使用分布式ID生成器（如雪花算法） |
