
## 零售银行POC演示脚本：AI辅助编码能力展示

### 一、工程背景介绍（约3分钟）

各位领导、专家，下午好。

我们面前这个零售银行POC工程，不是一段简单的“教学代码”，而是一个**严格按照银行技术标准构建的可演进的数字化银行底座**。

从业务视角看，它已经具备了个人客户管理和账户管理的核心能力：
- 客户可以在任一渠道完成身份核验，创建唯一客户编号；
- 可以开立个人活期账户，支持人民币/外币、Ⅰ/Ⅱ/Ⅲ类账户的基本属性；
- 可以查询单笔账户详情，用于柜面或手机银行的账户信息展示。

从技术视角看，这个工程体现了**银行级软件工程的三个关键追求**：

1. **分层清晰，职责归位**  
   Controller、Service、Repository、Mapper四层架构，每一层只做自己该做的事。事务边界、异常处理、日志规范都严格遵循行内研发手册。这不是临时堆砌的Demo，而是可以直接纳入持续集成流水线的“半成品”。

2. **数据库设计兼顾性能与扩展**  
   我们采用了H2作为嵌入式演示数据库，但表结构、索引设计、字段类型完全对标生产级MySQL/Oracle规范。例如`t_account`表针对`customer_id`建立了索引，支持高频的客户维度查询；`created_time`和`account_no`的组合排序可以保证分页结果稳定，避免高并发下数据重复或遗漏。

3. **统一契约，易于集成**  
   所有接口返回统一的`ApiResponse`，所有业务异常通过`BizException`和全局异常处理器统一收敛。无论是前端还是上下游系统，对接成本极低。

更重要的是，这个工程不是“写死”的——它从一开始就为**AI辅助开发**做好了准备。代码注释完整、命名语义化、设计模式一致，使得AI工具能够像一位熟悉项目的资深开发一样，理解现有逻辑，并**增量式地生成高质量代码**。

接下来，我们就用一个真实的业务需求，来展示AI如何在不破坏现有功能的前提下，快速完成一次“银行级”的增量开发。

### 二、场景引入：自然语言驱动代码生成（约5-7分钟）

#### 2.1 业务需求提出（讲故事）

今天，业务部门提出了一个新的需求：

> “我们的手机银行近期要上线‘账户总览’功能，客户登录后需要看到自己名下所有账户的列表。现在核心系统只有按账号查询单笔账户详情的接口，没有按客户号分页查询账户列表的能力。请开发团队在两周内完成这个功能，要求：支持分页、按创建时间倒序排列、不返回客户证件等敏感信息，并且不能影响现有的开户和账户详情查询。”

这个需求非常典型：**新增查询能力，不改造现有流程**。以前，开发人员需要：设计DTO → 写Controller → 写Service → 写Repository → 写Mapper XML → 写SQL → 写单元测试，至少半天工作量，还要反复核对字段、调试分页SQL、处理空指针和边界条件。

现在，我们试试用AI来完成。

#### 2.2 自然语言输入（现场演示准备）

我打开AI编码助手（比如Copilot / Codeium / 自研插件），输入以下自然语言描述：

> “基于当前零售银行POC工程，新增客户账户分页查询接口。  
> 接口路径 POST /api/v1/accounts/query-page，请求参数包含 customerId、pageNo、pageSize，其中pageNo从1开始，pageSize最大50。  
> 返回统一ApiResponse包装的分页对象，分页对象包含当前页列表、总记录数、页码、每页大小。  
> 列表项字段为：accountNo, customerId, accountName, accountType, productCode, currency, balance, branchCode, openDate, createdTime。  
> 按created_time降序、account_no降序排序，使用数据库物理分页。  
> 客户不存在时返回错误码CUST404。  
> 请生成所有分层代码：Controller、DTO、Service、Repository、Mapper、Mapper XML、单元测试，保持现有工程命名和注释风格。SQL中禁止使用SELECT *。”

注意：我没有告诉AI任何包名、方法名、具体的SQL写法——它需要自己去理解现有代码的结构。

#### 2.3 AI生成代码（演示过程）

AI在几十秒内返回了以下文件（可以现场展示差异对比）：

- `QueryCustomerAccountsRequest.java`：使用Lombok，添加了`@NotNull`、`@Min`等校验注解。
- `AccountListItemResponse.java`：只包含需要的字段，没有证件号、手机号。
- `PageResponse.java`：泛型分页类，与现有返回结构完美契合。
- `AccountController`中新增了`queryAccountPage`方法，路径、参数、返回值全部正确。
- `AccountService`接口和实现类中新增了分页查询方法，调用了Repository。
- `AccountRepository`接口和实现类中新增了分页查询和总数统计方法。
- `AccountMapper`中新增了`selectAccountPage`和`countAccountPage`方法。
- `AccountMapper.xml`中新增了两个SQL：一个分页查询，显式列出了15个字段；一个总数统计，使用`count(*)`。动态SQL只用了`where customer_id = #{customerId}`（因为当前需求没有账户状态过滤）。
- 单元测试`AccountControllerTest.java`：包含了客户不存在、无账户、正常分页、边界pageSize等5个测试用例，全部使用Mockito模拟依赖。

更让我惊喜的是，AI在生成的SQL旁边加了一段注释：

```sql
-- 索引使用：customer_id 命中 idx_account_customer_id，排序字段 created_time, account_no 可考虑复合索引进一步优化
```

它甚至主动提示了性能优化的方向。

#### 2.4 集成验证（简短演示）

我将生成的代码直接复制到工程中（不修改任何现有文件），启动Spring Boot应用，用Postman调用新接口：

- 请求：`{"customerId":"C2026041012301234","pageNo":1,"pageSize":10}`
- 响应：正确返回了该客户名下的2个账户，按创建时间倒序排列。
- 请求一个不存在的客户号：返回`{"code":"CUST404","message":"客户不存在"}`

原有接口`POST /api/v1/customers`和`POST /api/v1/accounts/open`仍然正常工作——零影响。

### 三、总结与价值升华（约1-2分钟）

各位可以看到，AI并不是在“写玩具代码”，而是真正理解了我们的分层架构、命名规范、异常处理机制，甚至考虑了索引和分页性能。它完成了一个**可直接上线的银行级查询接口**，而我们只做了一件事：**用自然语言描述业务需求**。

这对银行研发意味着什么？

- **效率提升**：类似的分页查询功能，从半天缩短到5分钟。
- **质量保障**：AI生成的代码天然符合统一异常、统一返回、SQL规范，减少了低级bug。
- **知识沉淀**：新人不需要反复问“Repository该怎么写”，AI就是最好的结对编程伙伴。
- **风险可控**：AI只做增量生成，不改动现有逻辑，兼容性由工具保证。

下一步，我们可以将这个能力推广到更复杂的场景：开户幂等控制、账户升降级规则校验、甚至批量交易的反洗钱筛查。AI不是替代开发人员，而是让我们的精力从“写重复代码”转移到“设计业务规则”和“保障系统安全”上。
