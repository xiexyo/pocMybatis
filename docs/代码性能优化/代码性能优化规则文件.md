# 代码性能优化规则文件（银行系统专用）

> 适用于高并发、高可用银行系统的代码审查与性能优化。本规则旨在指导开发人员提交的代码在保证正确性、安全性和可读性的前提下，实现显著的性能提升。

## 一、通用性能原则

1. **可读性与性能平衡**  
   优化不得以严重牺牲代码可读性为代价。复杂优化需添加注释说明原理与收益。

2. **银行系统特殊约束**
    - 事务边界清晰，避免长事务。
    - 保证数据最终一致性，慎用异步/缓存导致的数据延迟。
    - 所有 SQL 必须防注入，优化后仍需满足安全规范。

## 二、循环与逻辑简化

| 规则 ID | 规则描述 | 示例（反例 vs 正例） |
|--------|----------|----------------------|
| LOOP-01 | **避免在循环中调用远程服务或数据库**。应将批量查询/更新移到循环外部。 | 反例：`for(id:ids){ dao.selectById(id); }`<br>正例：`dao.selectBatch(ids);` |
| LOOP-02 | **减少循环内重复计算**。将不依赖循环变量的表达式提取到循环前。 | 反例：`for(i:list){ double rate = getRate() * Math.PI; }`<br>正例：`double rate = getRate() * Math.PI; for(...)` |
| LOOP-03 | **使用增强 for 或 Stream 时注意性能开销**。对大数据量优先使用传统 for 循环。 | Stream 并行流需评估线程安全及上下文切换成本，小数据量勿用。 |
| LOOP-04 | **避免在循环内拼接字符串**。应使用 `StringBuilder` 或 `String.join`。 | 反例：`String s=""; for(...){ s+=item; }`<br>正例：`StringBuilder sb = new StringBuilder(); for(...){ sb.append(item); }` |

## 三、数据库操作与 SQL 优化

| 规则 ID | 规则描述 | 示例 / 要点 |
|--------|----------|-------------|
| SQL-01 | **禁止 SELECT \***。只查询需要的字段，减少网络传输和内存开销。 | `SELECT col1, col2 FROM table WHERE ...` |
| SQL-02 | **使用预编译语句（PreparedStatement）**。防注入的同时可复用执行计划。 | MyBatis 中 `#{}` 即为预编译，避免 `${}` 拼接用户输入。 |
| SQL-03 | **避免隐式类型转换**。字段类型与参数类型不一致会导致索引失效。 | 反例：`WHERE varchar_col = 123` → 正例：`WHERE varchar_col = '123'` |
| SQL-04 | **分页查询优化**。大 offset 时改用游标或子查询/延迟关联。 | 反例：`LIMIT 100000, 20`<br>正例：`WHERE id > 上次最大id ORDER BY id LIMIT 20` |
| SQL-05 | **慎用 OR 和 IN 过大集合**。IN 超过 1000 个值建议拆分或使用临时表。 | 可用 `EXISTS` 替代部分 `OR`，或拆成多条查询业务合并。 |
| SQL-06 | **避免在 WHERE 条件中对字段使用函数**。如 `DATE(create_time)=?` 应改为范围查询。 | 反例：`WHERE DATE(create_time)='2025-01-01'`<br>正例：`WHERE create_time >= '2025-01-01' AND create_time < '2025-01-02'` |
| SQL-07 | **批量操作使用 JDBC batch**。插入/更新多条数据时使用 `addBatch()`。 | MyBatis 的 `foreach` 需结合 `executorType=BATCH`。 |
| SQL-08 | **事务内避免冗余查询**。同一事务中多次读取同一数据，考虑使用本地缓存或调整逻辑。 | 读取后未变更则可复用，或使用 `@Transactional(readOnly=true)` 标记只读事务。 |

## 四、索引设计与优化

| 规则 ID | 规则描述 | 检查要点 |
|--------|----------|----------|
| IDX-01 | **为高频 WHERE、JOIN、ORDER BY 列创建索引**。但避免过多索引（每表建议 ≤ 5 个）。 | 使用 `EXPLAIN` 检查 type 是否为 `ref`/`range`，避免 `ALL` 全表扫描。 |
| IDX-02 | **遵循最左前缀原则**。复合索引顺序按等值查询 → 范围查询 → 排序字段。 | 索引 `(a,b,c)` 支持 `WHERE a=? AND b=?`，不支持 `WHERE b=?`。 |
| IDX-03 | **避免对低基数列单独建索引**（如性别、状态）。可结合其他列建复合索引。 | 状态列若值极少，索引可能不被使用。 |
| IDX-04 | **覆盖索引**。若查询只需索引中的列，则避免回表，显著提升性能。 | `SELECT col1,col2 FROM t WHERE col1=?` 建索引 `(col1,col2)`。 |
| IDX-05 | **索引列避免可为空（NULL）**。`NULL` 值存储和查询效率较低，且复合索引中完全忽略 `NULL`。 | 设计表时尽量设置 `NOT NULL DEFAULT`。 |
| IDX-06 | **定期分析索引使用情况**。删除未使用或重复索引，减少写入开销。 | MySQL 可查 `sys.schema_unused_indexes`。 |

## 五、接口响应速度优化

| 规则 ID | 规则描述 | 实现方式 |
|--------|----------|----------|
| RES-01 | **合理使用缓存**。对不经常变化且读远大于写的数据使用本地缓存（Caffeine）或分布式缓存（Redis）。 | 注意缓存穿透、雪崩、击穿，设置合理过期时间。 |
| RES-02 | **并行调用非依赖的外部服务**。使用 `CompletableFuture` 或批量聚合。 | 示例：查询账户信息、交易流水、优惠活动可并行获取。 |
| RES-03 | **避免大对象及深拷贝**。尤其循环内创建大量临时对象，会加重 GC。 | 使用对象池（如连接池、线程池）复用资源。 |
| RES-04 | **异步处理非核心流程**。如日志记录、消息通知等使用 MQ 或 `@Async`。 | 保证主流程快速返回，但需确保异步任务异常不影响主流程。 |
| RES-05 | **限制接口超时与熔断**。对依赖的下游服务设置合理超时（如 200ms）并配置线程池隔离。 | 使用 Resilience4j、Sentinel 等组件。 |

## 六、冗余代码消除

| 规则 ID | 规则描述 |
|--------|----------|
| RED-01 | **提取重复代码为公共方法**。减少代码体积，也便于统一优化（如增加缓存）。 |
| RED-02 | **删除无用 import、变量、日志打印**。生产环境避免高频打印 debug 日志。 |
| RED-03 | **合并多次数据库调用**。同一个事务内对同一表的多次查询/更新应合并为一次。 |
| RED-04 | **避免使用不必要的包装类型**。优先使用 `int` 而非 `Integer` 在循环累加场景。 |
