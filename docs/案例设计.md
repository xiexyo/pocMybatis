
# 零售银行 AICoding 插件 POC 案例设计文档

## 1. 文档目的

本文档用于基于当前已完成的零售银行 POC 工程，设计一组能够全面体现 AICoding 插件能力的演示案例，覆盖：

- 自然语言转编码
- 代码智能补全
- 指定技术栈生成
- 特色编码模板
- 自定义模板生成与记忆
- 多文件协同生成
- 语法与规范纠错
- 逻辑漏洞与安全隐患排查
- 代码性能优化

本文档重点面向以下场景：

1. 作为后续 AI 辅助编码的案例输入材料
2. 作为 POC 汇报和演示的脚本基础
3. 作为后续新功能开发时的设计依据

---

## 2. POC 背景

当前项目是一个基于 **Java 17 + Spring Boot 3 + JPA + H2** 的零售银行账户服务 POC 工程，已完成以下基础能力：

- 创建客户
- 开立账户
- 查询账户详情
- 统一返回结构 ApiResponse
- 统一业务异常 BizException
- 全局异常处理 GlobalExceptionHandler
- 基础数据库表结构与数据初始化
- Swagger 接口文档
- H2 Console 数据查看

当前工程结构清晰，适合在此基础上继续演示 AI Coding 插件如何辅助完成新增功能、代码检查和优化工作。

---

## 3. POC 目标

本次案例设计目标不是单纯展示“AI 能写代码”，而是验证 AI Coding 插件是否能够在银行研发场景中支持一条完整的工程研发链路，包括：

### 3.1 研发效率目标
- 能根据自然语言快速生成接口骨架
- 能在现有工程中增量开发，而不是只适用于空白项目
- 能一次性生成多文件联动代码

### 3.2 工程规范目标
- 生成代码符合当前项目技术栈和包结构
- 符合统一返回、统一异常、统一命名规范
- 使用正确的 `jakarta.*` 包体系

### 3.3 银行业务适配目标
- 理解账户、交易、幂等、状态控制等银行典型规则
- 能补齐余额校验、状态校验、幂等控制等逻辑
- 能识别事务一致性和重复扣款风险

### 3.4 质量保障目标
- 能对已生成代码做规范纠错
- 能检查逻辑漏洞与安全隐患
- 能给出性能优化建议

# 7.1 案例1：自然语言生成账户入金接口

## 7.1.1 案例目标
验证 AICoding 插件能否基于自然语言需求，在当前工程中新增一个完整可运行的资金类接口。

## 7.1.2 业务背景
当前工程已支持开户与账户查询，但账户余额尚未发生真实变动。  
为形成后续转账场景的前置条件，需要新增“账户入金”能力。

## 7.1.3 业务规则
- requestId 必填
- accountNo 必须存在
- 账户状态必须为 ACTIVE
- amount 必须大于 0
- 入金成功后账户余额增加
- 必须写入交易流水
- 同一 requestId 重复请求不可重复入账

## 7.1.4 设计思路
本案例以“从需求到代码”的最短路径为目标，重点不是复杂逻辑，而是验证插件能否：

1. 理解资金类接口的基本语义
2. 自动识别需要修改哪些文件
3. 按现有工程结构新增代码
4. 生成可直接运行的入金接口

## 7.1.5 建议 AI 输入
```text
请基于当前零售银行账户服务工程新增账户入金接口，技术栈为 Java17 + Spring Boot 3 + Spring Data JPA + H2。
要求：
1. 新增 DepositRequest 和 DepositResponse
2. 在 TransactionController 中新增 POST /api/v1/transactions/deposit
3. 在 TransactionService / TransactionServiceImpl 中新增 deposit 方法
4. 校验账户存在、状态为 ACTIVE、金额大于 0
5. 入金成功后更新 t_account.balance
6. 写入一笔 t_transaction 流水，txnType = DEPOSIT
7. 使用 requestId 做幂等控制，若已成功处理则返回首次结果
8. 使用 ApiResponse 返回，异常使用 BizException
9. 保持当前包结构和代码风格，使用 jakarta 包
```

## 7.1.6 预期涉及文件
新增或修改：

- `DepositRequest.java`
- `DepositResponse.java`
- `TransactionController.java`
- `TransactionService.java`
- `TransactionServiceImpl.java`
- `TransactionEntity.java`
- `TransactionRepository.java`
- `AccountRepository.java`
- `IdempotentRecordRepository.java`
- `schema.sql`（如补余额快照字段）

## 7.1.7 执行步骤
### 步骤1：展示当前工程基线
展示当前已有：
- 创建客户
- 开户
- 查询账户详情
- H2 表结构
- Swagger 页面

### 步骤2：向插件输入自然语言需求
使用上面的需求描述作为输入。

### 步骤3：查看生成结果
重点检查：
- 是否沿用现有 `ApiResponse`
- 是否沿用 `BizException`
- 是否使用 `jakarta.validation`
- 是否按 `controller/service/repository/dto` 分层生成

### 步骤4：快速review关键逻辑
确认生成代码中包含：
- 金额校验
- 账户状态校验
- 更新余额
- 写入流水
- requestId 幂等判断

### 步骤5：接口联调
按以下顺序演示：
1. 查询账户详情，余额为 0
2. 调用入金接口
3. 再次查询账户详情，余额增加
4. 查看 `t_transaction` 表中新增的 DEPOSIT 流水
5. 使用相同 requestId 再次调用，验证不重复入账

## 7.1.8 演示输出
- Swagger 调用成功
- 账户余额变化正确
- 交易流水落表
- 重复 requestId 不重复入账

## 7.1.9 案例价值
该案例证明：
- AI 可直接将自然语言需求转化为工程代码
- AI 不只是生成 Controller，而是能联动 Service、Repository、Entity 和 SQL
- 适合展示“从0到1快速新增银行接口”的能力

# 7.2 案例2：多文件协同生成行内转账与幂等控制

## 7.2.1 案例目标
验证 AICoding 插件能否完成一个多文件、多规则、带事务和幂等的复杂银行接口开发任务。

## 7.2.2 业务背景
在入金场景完成后，需要支持两个本行账户之间的转账。  
该场景比入金更复杂，涉及：

- 转出账户扣款
- 转入账户加款
- 双账户状态校验
- 币种校验
- 余额校验
- 幂等控制
- 流水记录
- 事务一致性

## 7.2.3 业务规则
- requestId 必填
- 转出、转入账户必须存在
- 两个账户状态必须为 ACTIVE
- 两个账户不能相同
- 转出账户余额必须充足
- 账户币种必须一致
- 必须写入一笔 INTERNAL_TRANSFER 流水
- 同一 requestId 重复请求不可重复扣款

## 7.2.4 设计思路
本案例是整个 POC 的核心亮点，用于验证插件能否：

1. 识别这是一个跨文件、跨层次的新增需求
2. 自动拆解出 DTO、Controller、Service、幂等服务、Repository 增强和 SQL 变更
3. 理解双账户更新和事务控制的业务含义
4. 自动补全典型银行规则

## 7.2.5 建议 AI 输入
```text
请基于当前工程新增行内转账接口 POST /api/v1/transactions/internal-transfer。
要求：
1. 新增 InternalTransferRequest 和 InternalTransferResponse
2. 在 TransactionController 中新增转账接口
3. 在 TransactionService / TransactionServiceImpl 中新增 internalTransfer 方法
4. 校验转出账户和转入账户存在、状态正常、币种一致、账户不同、余额充足
5. 使用 t_idempotent_record 做 requestId 幂等控制
6. 转账成功后更新两个账户余额
7. 写入一笔 t_transaction 流水，txnType = INTERNAL_TRANSFER
8. 若重复请求已成功，则返回首次成功结果
9. 使用事务保证数据一致性
10. 保持当前工程风格和包结构
```

## 7.2.6 预期涉及文件
新增或修改：

- `InternalTransferRequest.java`
- `InternalTransferResponse.java`
- `TransactionController.java`
- `TransactionService.java`
- `TransactionServiceImpl.java`
- `IdempotentService.java`
- `IdempotentServiceImpl.java`
- `TransactionEntity.java`
- `TransactionRepository.java`
- `AccountRepository.java`
- `IdempotentRecordRepository.java`
- `schema.sql`

## 7.2.7 执行步骤
### 步骤1：展示需求文档
打开：
- `docs/02-需求说明.md`
- `docs/03-详细设计.md`

说明已经有明确需求和详细设计，不是随意生成。

### 步骤2：让插件生成多文件代码
把需求喂给插件。

### 步骤3：查看多文件协同结果
重点展示：
- DTO 已生成
- Service 已实现
- Controller 已新增
- 幂等服务已新增
- Repository 查询已增强
- SQL 有同步变更

### 步骤4：接口联调
1. 创建两个客户和两个账户
2. 给 A 账户入金
3. 发起 A -> B 转账
4. 查询 A 账户详情
5. 查询 B 账户详情
6. 查看交易流水
7. 重复提交相同 requestId

### 步骤5：展示幂等结果
重复请求时：
- 不重复扣款
- 返回首次处理结果或幂等提示

## 7.2.8 演示输出
- 转账成功
- 转出余额减少
- 转入余额增加
- 流水落表
- 幂等生效
- 无重复扣款

## 7.2.9 案例价值
该案例是 POC 最重要的复杂案例，能体现：
- 多文件协同生成能力
- 银行业务规则理解能力
- 工程级事务与幂等处理能力

# 7.3 案例3：基于模板生成账户冻结接口

## 7.3.1 案例目标
验证 AICoding 插件是否支持将银行通用接口模式沉淀成模板，并快速复用到新接口开发中。

## 7.3.2 业务背景
账户冻结是银行系统中的标准接口之一，结构固定、规则清晰，适合作为模板化场景。

冻结后要求：
- 账户状态更新为 FROZEN
- 记录冻结原因
- 查询账户详情时返回冻结原因
- 冻结账户不可参与入金和转账

## 7.3.3 设计思路
本案例的重点不是业务复杂度，而是展示：

1. 银行标准写接口可以模板化
2. AI 可基于模板而不是从零生成
3. 模板化开发适合大量重复接口场景

## 7.3.4 建议模板内容
建议定义一个“银行写接口标准模板”，包括：

- Request DTO
- Response DTO
- Controller
- Service
- 参数校验
- 状态校验
- BizException
- ApiResponse
- Swagger 注解
- 审计日志占位

## 7.3.5 建议 AI 输入
```text
请使用“银行写接口标准模板”生成账户冻结接口。
要求：
1. 新增 POST /api/v1/accounts/{accountNo}/freeze
2. 新增 FreezeAccountRequest 和 FreezeAccountResponse
3. 在 t_account 表增加 freeze_reason 字段
4. 冻结后 account_status = FROZEN
5. 账户详情接口返回 freezeReason
6. 后续入金和转账对 FROZEN 账户返回业务异常
7. 保持当前工程风格
```

## 7.3.6 预期涉及文件
- `FreezeAccountRequest.java`
- `FreezeAccountResponse.java`
- `AccountController.java`
- `AccountService.java`
- `AccountServiceImpl.java`
- `AccountEntity.java`
- `AccountDetailResponse.java`
- `schema.sql`

## 7.3.7 执行步骤
1. 先展示模板定义
2. 输入冻结接口需求
3. 让插件基于模板生成代码
4. 查看生成结果
5. 调用冻结接口
6. 再调用转账或入金接口，验证冻结账户不可操作

## 7.3.8 演示输出
- 冻结接口快速生成
- 账户状态成功更新
- 冻结原因可查
- 冻结账户不可继续交易

## 7.3.9 案例价值
该案例适合展示：
- AI 能沉淀重复经验
- 模板能力适合银行大量标准接口研发
- 能形成团队级复用资产

# 7.4 案例4：代码规范纠错与逻辑漏洞排查

## 7.4.1 案例目标
验证插件不仅能生成代码，还能对已有代码进行规范检查、逻辑审查和风险提示。

## 7.4.2 业务背景
银行系统对代码质量要求高，开发痛点不只是“写得慢”，更是：

- 事务遗漏
- 幂等缺失
- 敏感日志输出
- 账户状态校验遗漏
- 技术栈不规范

因此需要验证插件是否具备“代码审查能力”。

## 7.4.3 建议预置问题
可故意准备一版有缺陷的转账代码，包含：

### 规范问题
- 使用 `javax.validation` 而不是 `jakarta.validation`
- Controller 直接返回对象而非 `ApiResponse`
- 使用 `RuntimeException` 而非 `BizException`
- 命名不符合当前工程风格

### 逻辑问题
- 未校验借贷账号不能相同
- 未校验余额不足
- 未加 `@Transactional`
- 更新余额后未落流水

### 安全问题
- 日志打印完整账号、证件号
- requestId 未校验幂等
- 冻结账户仍允许转账

## 7.4.4 建议 AI 输入
```text
请检查以下行内转账代码是否存在语法错误、工程规范问题、逻辑漏洞和安全隐患，并输出问题清单和修复建议。
检查维度包括：
1. 是否符合当前 Spring Boot 3 + jakarta 规范
2. 是否符合当前 ApiResponse / BizException / GlobalExceptionHandler 风格
3. 是否存在事务一致性问题
4. 是否存在重复扣款风险
5. 是否存在敏感信息日志输出问题
6. 如可修复，请输出修正后的代码
```

## 7.4.5 执行步骤
1. 打开预置的“问题代码”
2. 让插件进行扫描
3. 展示问题列表
4. 展示修复建议
5. 选择自动修复或半自动修复
6. 再次编译和运行验证

## 7.4.6 演示输出
插件应输出：
- 问题类型
- 问题位置
- 风险说明
- 修复建议
- 修复后的代码

## 7.4.7 案例价值
这是非常适合银行评审的能力点，能证明：
- AI 不只是写样板代码
- 还能辅助做开发质量控制
- 能降低上线风险

# 7.5 案例5：交易流水查询性能优化

## 7.5.1 案例目标
验证插件在“已有代码可运行”的前提下，是否能进一步帮助开发人员做性能优化。

## 7.5.2 业务背景
交易流水查询是银行高频查询接口。  
初版代码往往存在以下问题：

- `findAll()` 后在内存中过滤
- 未分页
- 未限制 pageSize
- 未加排序
- 未加索引
- 查询条件组合不合理

## 7.5.3 设计思路
本案例重点展示 AI 从“代码生成”走向“代码优化”的能力，验证其是否能：

1. 识别低效实现
2. 给出 JPA 查询优化方案
3. 建议索引
4. 保持现有接口风格不变

## 7.5.4 建议 AI 输入
```text
请优化当前交易流水查询实现。
要求：
1. 支持按 accountNo、txnType、startTime、endTime 分页查询
2. 改为数据库分页，不允许全量查出后内存过滤
3. 按 txnTime 倒序
4. pageSize 最大 100
5. 补充必要的索引建议
6. 保持当前 Spring Data JPA 风格和返回结构不变
```

## 7.5.5 预期涉及文件
- `TransactionQueryRequest.java`
- `TransactionItemResponse.java`
- `TransactionQueryResponse.java`
- `TransactionRepository.java`
- `TransactionServiceImpl.java`
- `schema.sql`

## 7.5.6 执行步骤
1. 展示普通版查询代码
2. 让插件给出性能优化方案
3. 对比前后代码差异
4. 运行查询接口
5. 展示分页结果和查询效果
6. 说明索引与分页优化思路

## 7.5.7 演示输出
- 查询接口支持分页和过滤
- 查询逻辑从内存过滤变为数据库过滤
- schema 中有索引增强建议
- 响应结构保持稳定

## 7.5.8 案例价值
适合证明：
- AI 不只是“能生成”，还能“帮助优化”
- 对银行高频查询接口有现实意义

# 7.6 案例6：需求变更下的代码智能补全

## 7.6.1 案例目标
验证插件在已有实现基础上，是否能够快速响应需求变更并完成局部增量改造。

## 7.6.2 变更示例
建议选择一个简单且能体现业务规则变更的需求，例如：

### 变更场景A
冻结账户后，禁止入金和转账

### 变更场景B
单笔转账金额超过 50,000 时，需要增加风险等级校验

## 7.6.3 设计思路
该案例重点展示：
- AI 对现有上下文的理解
- AI 对局部代码补全和规则插入的能力
- 需求变更时的效率价值

## 7.6.4 建议 AI 输入
```text
请基于当前工程补充新规则：
冻结账户不允许入金和转账。
请在不改变现有接口结构的前提下，修改相关 Service 逻辑、补充业务异常，并说明需要修改的文件。
```

## 7.6.5 演示步骤
1. 展示当前可正常入金/转账
2. 提出变更需求
3. 让插件补全代码
4. 重新调用接口验证新规则

## 7.6.6 案例价值
该案例更贴近真实研发，因为企业开发中大量时间消耗在“变更开发”而不是首版开发。
