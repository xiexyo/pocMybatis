# 当前工程标准查询接口模板

## 1. 文档目的

本文档用于基于当前真实工程代码，沉淀适用于本项目的“标准查询接口模板”，指导后续 AI 或开发人员按照统一模式实现：

- 详情查询接口
- 列表查询接口
- 分页查询接口
- 统计查询接口

本模板强调：

- 查询接口统一返回风格
- 查询对象与响应对象解耦
- 分页、排序、过滤规范化
- 动态 SQL 由数据库完成查询处理
- 不使用内存分页和全量过滤

---

## 2. 适用范围

本模板适用于：

- 账户详情查询
- 交易流水查询
- 账户列表查询（后续扩展）
- 交易统计查询（后续扩展）
- 冻结历史查询（后续扩展）

---

## 3. 模板来源

## 3.1 当前真实工程直接来源

### 详情查询模板来源
- `AccountController#getAccountDetail`
- `AccountServiceImpl#getAccountDetail`

其特点包括：

1. 先查主对象
2. 再查关联对象
3. 最终组装响应 DTO
4. 查询不到抛 `BizException`
5. 不直接返回 Entity

---

## 3.2 扩展来源

分页查询模板可参考后续交易流水查询场景扩展。

## 4. 查询接口分类

建议将查询接口划分为两类：

### 4.1 详情查询
特点：

- 返回单个对象
- 常基于主键、编号或唯一标识查询
- 可能联动多个表或多个 Repository 查询

### 4.2 列表 / 分页查询
特点：

- 返回多条记录
- 支持条件过滤
- 支持排序
- 支持分页
- 由数据库完成过滤与分页

---

## 5. 查询接口标准结构

建议统一包含：

```text
dto.request.XXXQueryRequest
dto.response.XXXItemResponse
dto.response.XXXQueryResponse
controller.XXXController
service.XXXService
service.impl.XXXServiceImpl
repository.XXXRepository
repository.impl.XXXRepositoryImpl
mapper.XXXMapper
resources/mapper/XXXMapper.xml
```

说明：

- 若为详情查询，可不一定需要 `ItemResponse`
- 若为简单单对象查询，也可只返回单个 `XXXResponse`

---

## 6. 查询请求 DTO 模板

## 6.1 基本要求

1. 放在 `dto.request`
2. 使用 `jakarta.validation.*`
3. 时间范围字段使用 `LocalDateTime`
4. 分页字段使用 `pageNo`、`pageSize`
5. 条件字段保持业务语义清晰

---

## 6.2 分页字段建议

推荐统一使用：

- `pageNo`
- `pageSize`

处理规则建议：

- `pageNo` 默认 1
- `pageSize` 默认 20
- `pageSize` 最大 100

---

## 6.3 时间范围字段建议

若查询涉及时间区间，建议包含：

- `startTime`
- `endTime`

并在 Service 中校验：

```java
startTime <= endTime
```

---

## 6.4 示例

```java
@Data
public class TransactionQueryRequest {

    @NotBlank(message = "账户号不能为空")
    private String accountNo;

    private String txnType;

    @DateTimeFormat(iso = DateTimeFormat.ISO.DATE_TIME)
    private LocalDateTime startTime;

    @DateTimeFormat(iso = DateTimeFormat.ISO.DATE_TIME)
    private LocalDateTime endTime;

    private Integer pageNo;
    private Integer pageSize;
}
```

---

## 7. 查询响应 DTO 模板

## 7.1 详情查询模板

详情查询返回单个 Response DTO，例如：

```java
@Data
@Builder
public class AccountDetailResponse {
    private String accountNo;
    private String customerId;
    private String customerName;
    private String productCode;
    private String productName;
    private BigDecimal balance;
    private String currency;
    private String accountStatus;
}
```

---

## 7.2 列表项模板

列表查询通常使用 `ItemResponse`：

```java
@Data
@Builder
public class TransactionItemResponse {
    private String txnId;
    private String requestId;
    private String direction;
    private String counterpartyAccountNo;
    private String txnType;
    private BigDecimal amount;
    private String txnStatus;
    private String remark;
    private LocalDateTime txnTime;
    private BigDecimal balanceAfter;
}
```

---

## 7.3 分页响应模板

分页查询建议统一返回：

- `total`
- `pageNo`
- `pageSize`
- `records`

示例：

```java
@Data
@Builder
public class TransactionQueryResponse {
    private Long total;
    private Integer pageNo;
    private Integer pageSize;
    private List<TransactionItemResponse> records;
}
```

## 8. Controller 模板要求

Controller 必须：

- 使用 `@RestController`
- 统一返回 `ApiResponse<T>`
- 只负责参数接收和调用 Service

### 详情查询示例

```java
@GetMapping("/{accountNo}")
public ApiResponse<AccountDetailResponse> getAccountDetail(@PathVariable String accountNo)
```

### 分页查询示例

```java
@GetMapping
public ApiResponse<TransactionQueryResponse> queryTransactions(@Valid @ModelAttribute TransactionQueryRequest request)
```

说明：

- GET 查询接口优先使用 `@ModelAttribute`
- 不建议 GET 使用 `@RequestBody`

---

## 9. Service 模板要求

## 9.1 详情查询模板

标准流程建议：

```text
接收查询条件
-> 查询主对象
-> 若不存在抛BizException
-> 查询关联对象
-> 组装Response
-> 返回结果
```

---

## 9.2 分页查询模板

标准流程建议：

```text
接收请求
-> 校验主对象是否存在（如适用）
-> 处理分页参数默认值
-> 校验时间范围
-> count 查询
-> list 查询
-> 结果转换
-> 组装分页结果
-> 返回结果
```

---

## 10. 详情查询模板要求

## 10.1 典型模式

例如账户详情查询，推荐通过多个 Repository 查询后组装结果：

1. 查账户
2. 查客户
3. 查产品
4. 组装 `AccountDetailResponse`

说明：

- 当前真实工程已采用此模式
- 不强制使用复杂联表 SQL

---

## 10.2 错误处理

- 主对象不存在：对应 `404` 类错误码
- 关联对象不存在：根据业务语义抛出对应错误码

例如：

- `ACCT404`
- `CUST404`
- `PROD404`

---

## 11. 分页查询模板要求

## 11.1 分页原则

必须由数据库完成：

- 过滤
- 排序
- 分页

不允许：

- 全量查询后 Java 过滤
- 全量查询后 Java 分页

---

## 11.2 分页参数处理建议

```java
int pageNo = request.getPageNo() == null || request.getPageNo() < 1 ? 1 : request.getPageNo();
int pageSize = request.getPageSize() == null || request.getPageSize() < 1 ? 20 : request.getPageSize();
if (pageSize > 100) {
    pageSize = 100;
}
int offset = (pageNo - 1) * pageSize;
```

---

## 11.3 时间范围校验

```java
if (request.getStartTime() != null
        && request.getEndTime() != null
        && request.getStartTime().isAfter(request.getEndTime())) {
    throw new BizException("TXN002", "开始时间不能大于结束时间");
}
```

---

## 12. Repository 模板要求

列表查询建议由 Repository 向 Service 提供以下能力：

- `countXxx(...)`
- `queryXxxList(...)`

详情查询继续复用：

- `findById(...)`
- `findByNo(...)`
- `findByCode(...)`

说明：

- Service 不建议直接访问 Mapper
- Repository 对底层动态 SQL 能力做轻量封装

---

## 13. Mapper / XML 模板要求

## 13.1 基本要求

1. SQL 写在 XML 中
2. 动态 SQL 使用 `<if> <where> <trim>`
3. 多参数必须 `@Param`
4. 分页 SQL 使用数据库 `LIMIT / OFFSET`
5. 排序字段必须明确

---

## 13.2 动态 where 模板

```xml
<where>
    <if test="txnType != null and txnType != ''">
        AND txn_type = #{txnType}
    </if>
    <if test="startTime != null">
        AND txn_time <![CDATA[>=]]> #{startTime}
    </if>
    <if test="endTime != null">
        AND txn_time <![CDATA[<=]]> #{endTime}
    </if>
</where>
```

---

## 13.3 分页 SQL 模板

```sql
ORDER BY txn_time DESC
LIMIT #{pageSize} OFFSET #{offset}
```

---

## 14. 查询结果转换模板

若查询结果需要按“当前视角”转换，应在 Service 中完成 DTO 转换，不建议直接将 Entity 原样返回。

例如交易流水查询中：

- `direction`
- `counterpartyAccountNo`
- `balanceAfter`

都应在 Service 层按业务语义计算。

---

## 15. 性能优化模板要求

查询接口必须考虑以下优化要求：

1. 必须数据库分页
2. 必须数据库过滤
3. 必须数据库排序
4. 根据查询条件补齐索引
5. 避免不必要的重复查库
6. 动态 SQL 必须清晰可维护

---

## 16. AI 生成时必须遵循的规则

AI 使用本模板生成查询接口时必须：

1. 返回 `ApiResponse<T>`
2. 使用 `BizException`
3. 分页查询使用数据库分页
4. 使用 Repository + Mapper + XML
5. 不直接返回 Entity
6. 不在 Java 内存中过滤和分页
7. 输出完整多文件代码

## 19. 结论

标准查询接口模板是当前工程后续扩展中最容易高频复用的一类模板。其重点在于统一接口结构和查询实现质量，尤其要避免低质量的全量查询、Java 内存过滤和分页实现。
