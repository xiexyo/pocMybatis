# 当前工程资金交易接口模板

## 1. 文档目的

本文档用于基于当前真实工程基线和资金类扩展设计，沉淀适用于本项目的“资金交易接口模板”，用于指导后续 AI 或开发人员按统一模式实现：

- 存入
- 转账
- 扣款
- 冲正 / 撤销
- 代扣 / 代发等后续扩展能力

本模板强调：

- 事务一致性
- requestId 幂等
- 账户加锁
- 余额更新
- 交易流水记录
- 成功结果重放

说明：

- 当前真实工程尚未完整落地资金交易接口
- 本模板是基于当前工程结构和后续详细设计沉淀的标准实现模式
- 后续新增资金接口时必须优先复用该模板

---

## 2. 适用范围

适用于以下资金类接口：

- 账户存入
- 行内转账
- 扣款
- 账户间划转
- 冲正 / 撤销
- 代扣 / 代发（后续扩展）

---

## 3. 模板前提

在当前工程中使用本模板时，应默认遵循以下前提：

1. 当前工程仍采用单体结构
2. 所有数据库访问仍通过 Repository + Mapper + XML
3. 使用 `ApiResponse`
4. 使用 `BizException`
5. 使用 `@Transactional`
6. 若涉及交易记录，统一写入 `t_transaction`
7. 若涉及幂等控制，统一使用 `t_idempotent_record`

## 7. 资金交易标准处理流程

## 7.1 单边资金交易模板

例如：存入

```text
参数校验
-> 金额校验
-> 幂等校验
-> 初始化幂等记录
-> 锁定账户
-> 账户存在校验
-> 账户状态校验
-> 计算余额
-> 更新账户余额
-> 写交易流水
-> 更新幂等成功状态
-> 返回结果
```

---

## 7.2 双边资金交易模板

例如：行内转账

```text

-> 金额校验
-> 借贷账户不同校验
-> 幂等校验
-> 初始化幂等记录
-> 按固定顺序锁定双账户
-> 账户存在校验
-> 账户状态校验
-> 币种一致校验
-> 余额充足校验
-> 可选风控校验
-> 计算借贷双方余额
-> 更新双方账户余额
-> 写交易流水
-> 更新幂等成功状态
-> 返回结果
```

## 8. Request DTO 模板要求

## 8.1 基本要求

1. 放在 `dto.request`
2. 使用 `jakarta.validation.*`
3. `requestId` 必须可校验
4. `amount` 使用 `BigDecimal`
5. 不做复杂业务校验，只做基础格式校验

---

## 8.2 示例：转账请求

```java
@Data
public class InternalTransferRequest {

    @NotBlank(message = "requestId不能为空")
    private String requestId;

    @NotBlank(message = "转出账户不能为空")
    private String debitAccountNo;

    @NotBlank(message = "转入账户不能为空")
    private String creditAccountNo;

    @NotNull(message = "金额不能为空")
    private BigDecimal amount;

    @Size(max = 255, message = "备注长度不能超过255")
    private String remark;

    private String channelCode;
    private String branchCode;
    private String operatorId;
}
```

## 12. 资金类校验模板

## 12.1 金额校验

必须使用：

```java
amount.compareTo(BigDecimal.ZERO) > 0
```

若不满足：

```java
throw new BizException("TXN001", "交易金额必须大于0");
```

---

## 12.2 账户存在校验

若账户不存在：

```java
throw new BizException("ACCT404", "账户不存在");
```

---

## 12.3 状态校验

若账户冻结：

```java
throw new BizException("ACCT005", "账户已冻结");
```

若状态不可用：

```java
throw new BizException("ACCT001", "账户状态不可用");
```

---

## 12.4 双账户相同校验

若借贷账户相同：

```java
throw new BizException("ACCT003", "转出账户和转入账户不能相同");
```

---

## 12.5 币种一致校验

若币种不一致：

```java
throw new BizException("ACCT004", "账户币种不一致");
```

---

## 12.6 余额充足校验

若余额不足：

```java
throw new BizException("ACCT002", "账户余额不足");
```

## 13. 幂等模板要求

## 13.1 幂等处理原则

资金类接口建议统一支持 `requestId` 幂等。

必须实现：

1. 先查幂等记录
2. 若 `SUCCESS`，返回首次成功结果
3. 若 `PROCESSING`，返回处理中
4. 若 `FAILED`，返回已失败
5. 若不存在，则初始化 `PROCESSING`

---

## 13.2 幂等状态建议

- `PROCESSING`
- `SUCCESS`
- `FAILED`

---

## 13.3 常见错误码

- `IDEMP001`：请求正在处理中
- `IDEMP002`：请求已成功处理
- `IDEMP003`：请求已失败，请更换 requestId 重试

---

## 13.4 幂等服务模板

建议统一封装：

```java
void init(String requestId, String businessType, String businessKey);

void markSuccess(String requestId, String businessType, String responseCode, String responseMessage);

void markFailed(String requestId, String businessType, String responseCode, String responseMessage);

IdempotentRecordEntity getExistingRecord(String requestId, String businessType);
```

---

## 14. 账户加锁模板

## 14.1 单账户交易

建议通过：

```java
accountRepository.lockByAccountNo(accountNo)
```

底层使用：

```sql
SELECT ... FROM t_account WHERE account_no = #{accountNo} FOR UPDATE
```

---

## 14.2 双账户交易

必须采用**固定顺序加锁**，避免死锁：

1. 比较两个账号字符串大小
2. 较小账号先锁
3. 较大账号后锁

---

## 15. 余额更新模板

余额更新必须通过 XML SQL 完成。

示例：

```java
accountRepository.updateBalance(accountNo, balanceAfter, now);
```

底层 SQL 示例：

```sql
UPDATE t_account
SET balance = #{balance},
    updated_time = #{updatedTime}
WHERE account_no = #{accountNo}
```

## 17. 成功结果重放模板

当幂等记录状态为 `SUCCESS` 时，应通过：

```java
transactionRepository.findByRequestId(requestId)
```

查询首次成功的交易流水，并返回同样结果。

说明：

- 不允许重复扣款
- 不允许重复入账
- 不允许重复写流水

---

## 18. Repository 模板要求

资金接口建议至少具备以下 Repository：

### AccountRepository
- `findByAccountNo`
- `lockByAccountNo`
- `updateBalance`

### TransactionRepository
- `save`
- `findByRequestId`

### IdempotentRecordRepository
- `save`
- `findByRequestIdAndBusinessType`
- `updateStatus`

---

## 19. Mapper / XML 模板要求

必须遵循：

1. SQL 全部写 XML
2. 多参数必须 `@Param`
3. 锁定查询使用 `FOR UPDATE`
4. 更新余额和写流水全部通过 XML 完成
5. 查询首次成功结果必须能按 `requestId` 查询流水

---

方式扩展。

## 21. AI 生成时必须遵循的规则

AI 使用本模板生成代码时必须：

1. 使用 `@Transactional`
2. 使用 `requestId` 幂等
3. 使用 Repository + Mapper + XML
4. 使用账户锁定查询
5. 使用 `BizException`
6. 记录交易流水
7. 成功结果支持重放
8. 不引入 JPA / MyBatis-Plus / Redis / MQ
9. 输出完整多文件代码
