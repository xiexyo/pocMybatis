# 银行核心系统逻辑漏洞与安全隐患排查规则文件

## 版本信息
- 版本号：1.0
- 适用场景：银行核心系统

## 规则说明
本文件定义了针对银行核心系统的常见逻辑漏洞及安全隐患的检测规则，每项规则包含：
- **漏洞类型**：问题分类
- **检测要点**：AI需关注的代码模式、关键字或上下文
- **风险影响**：可能造成的业务后果（如账务错误、数据泄露、权限绕过）
- **修复方案**：推荐的代码修改或配置调整建议
- **严重等级**：高/中/低

---

## 一、空指针异常

### 规则编号：BANK-NULL-001
- **漏洞类型**：未校验对象/集合/数组为空直接调用方法或访问属性
- **检测要点**：
  - 从数据库查询、RPC调用、缓存、外部输入获得的对象未判空即调用方法（如 `getXxx()`、`toString()`）
  - 集合（List/Map/Set）未检查 `isEmpty()` 或 `!= null` 直接遍历或取元素
  - 数组未检查 `length` 或 `!= null` 直接使用下标
- **风险影响**：
  - 导致程序抛出 `NullPointerException`，引发交易中断、接口返回500错误
  - 在批量处理或异步任务中可能造成部分数据未处理，影响账务一致性
- **修复方案**：
  - 在调用前增加判空校验：`if (obj != null) { ... }`
  - 使用Java 8 `Optional` 或工具类（如 `Objects.requireNonNull`）
  - 对集合/数组使用 `CollectionUtils.isEmpty()` 或 `== null` 联合判断
- **严重等级**：高

## 二、边界值错误

### 规则编号：BANK-BOUND-001
- **漏洞类型**：数值范围校验缺失或错误（金额、利率、日期、次数等）
- **检测要点**：
  - 金额运算未检查负数、零值、超出单笔/单日限额
  - 循环或递归未定义正确的终止边界（如 `i<=limit` 误写为 `i<limit`）
  - 字符串截取、数组索引使用外部传入下标未校验是否在 `[0, length-1]` 范围
  - 日期计算未考虑闰年、月末、跨年边界（如 `02-29`、`12-31` 过渡到 `01-01`）
- **风险影响**：
  - 可能产生账务金额错误（如允许负值交易、超出限额扣款）
  - 数组越界导致程序崩溃或内存访问异常
  - 日期计算错误影响利息计算、定期到期、对账等核心业务
- **修复方案**：
  - 对所有数值输入进行范围校验（最小值、最大值、精度）
  - 使用 `BigDecimal` 并定义舍入模式，禁止直接使用 `float/double`
  - 循环边界采用 `<=` 或 `<` 结合业务逻辑明确，优先使用增强for循环或Stream
  - 日期处理使用 `java.time` 包，避免手工计算
- **严重等级**：高

## 三、事务未提交 / 回滚

### 规则编号：BANK-TX-001
- **漏洞类型**：数据库事务提交或回滚逻辑缺失、顺序错误、异常未捕获
- **检测要点**：
  - 手动事务管理（如 JDBC `setAutoCommit(false)`）但 `commit()` 只在正常路径调用，异常路径未调用 `rollback()`
  - 使用声明式事务（如 Spring `@Transactional`）但传播级别设置不当（如 REQUIRES_NEW 内层异常导致外层无法回滚）
  - 分布式事务未处理 TCC 或 SAGA 的补偿/回滚逻辑
  - 事务内包含非事务操作（如发送消息、文件写入）导致数据不一致
- **风险影响**：
  - 部分数据已提交而另一部分回滚，造成账务不平、重复记账、漏记账
  - 资源未释放，连接池耗尽
  - 跨系统交易可能出现资金已扣但未发货/未入账
- **修复方案**：
  - 优先使用声明式事务（`@Transactional`），明确 `rollbackFor = Exception.class`
  - 手工事务必须在 `finally` 块中根据异常标志决定 `commit` 或 `rollback`
  - 避免在事务中调用远程服务，将非事务操作移到事务外或使用最终一致性方案
  - 对分布式事务采用可靠消息或事件表模式
- **严重等级**：高

## 四、权限校验缺失

### 规则编号：BANK-AUTH-001
- **漏洞类型**：接口或方法未校验用户身份/角色/操作权限
- **检测要点**：
  - Controller 或 Service 方法上无权限注解（如 `@PreAuthorize`、`@RequiresRoles`）
  - 未从安全上下文中获取当前用户ID，或直接信任前端传入的用户标识
  - 批量查询接口未校验当前用户是否有权查看其他用户的数据（越权）
  - 内部调用（同一JVM内的方法调用）绕过了权限切面
- **风险影响**：
  - 普通用户可以执行管理员操作（如修改利率、审批贷款）
  - 用户可查询/修改他人账户信息、交易记录，造成严重数据泄露和资金风险
  - 越权操作可能导致对账失败、伪造交易
- **修复方案**：
  - 统一使用权限框架（如 Spring Security、Shiro），所有对外接口必须显式声明权限要求
  - 对于数据级权限，在 SQL 或业务逻辑中增加 `user_id = 当前用户ID` 条件
  - 内部调用若需权限校验，应通过代理对象或重新进入安全上下文
- **严重等级**：高

## 五、敏感数据未加密

### 规则编号：BANK-SEC-001
- **漏洞类型**：敏感信息（密码、身份证、卡号、CVV、PIN、密钥）以明文存储或传输
- **检测要点**：
  - 数据库表中密码字段为 `varchar` 且无 `encrypted` 标记或未使用哈希+盐
  - 日志中打印了身份证、卡号、手机号等个人信息（未脱敏）
  - HTTP 响应中返回明文敏感字段
  - 配置文件（如 `application.properties`）包含明文数据库密码、API密钥
  - 缓存（Redis、Memcached）中存储未加密的敏感数据
- **风险影响**：
  - 数据库被拖库或日志泄露导致大量客户隐私泄露
  - 内部人员可非法查询客户敏感信息
  - 违反金融行业合规要求（PCI-DSS、个人信息保护法）
- **修复方案**：
  - 密码使用 `bcrypt`、`scrypt` 或 `PBKDF2` 加盐哈希存储
  - 身份证、卡号等使用 AES-256 加密存储，并在展示时脱敏（如 `6222****1234`）
  - 禁止在日志中输出敏感字段；配置日志脱敏过滤器
  - 敏感配置使用密钥管理服务（KMS）或环境变量注入
  - HTTPS 强制传输，禁止 HTTP 明文
- **严重等级**：高

## 六、权限管控不足（细粒度）

### 规则编号：BANK-AUTH-002
- **漏洞类型**：角色权限划分不清晰、最小权限原则未落实、动态权限缺失
- **检测要点**：
  - 存在超级管理员角色拥有全部权限，无操作审计
  - 普通柜员可执行复核/授权操作（职责未分离）
  - 未对接口的请求频率、IP白名单、时间段等做限制
  - 功能权限与数据权限混用，导致高权限角色可查看未授权的数据
  - 未限制批量操作的数量（如一次性查询全部用户交易流水）
- **风险影响**：
  - 内部人员滥用权限进行非法转账、修改数据
  - 恶意用户通过高频接口进行暴力破解、撞库、短信轰炸
  - 批量导出数据造成大规模泄露
- **修复方案**：
  - 基于 RBAC 模型，遵循最小权限原则，定期复审角色权限
  - 实现操作审计日志，对敏感操作强制二次授权
  - 增加接口限流（令牌桶、漏桶）、IP黑白名单、操作时段控制
  - 批量操作增加分页限制和总数上限（如最大1000条）
- **严重等级**：高

## 七、其他常见银行系统漏洞（补充）

### 规则编号：BANK-OTHER-001
- **漏洞类型**：并发竞争条件（Race Condition）
- **检测要点**：
  - 对同一账户或公共资源的扣款、积分增减等操作未加锁或未使用乐观锁
  - 检查-然后-执行（check-then-act）模式未原子化
- **风险影响**：导致账户余额被透支、重复扣款、奖品超发
- **修复方案**：使用数据库行锁（`SELECT ... FOR UPDATE`）、乐观锁（版本号）、分布式锁（Redis/Redisson）

### 规则编号：BANK-OTHER-002
- **漏洞类型**：资源未释放（连接、流、锁）
- **检测要点**：
  - 数据库连接、文件流、网络连接未在 finally 或 try-with-resources 中关闭
  - 线程池未正确 shutdown
- **风险影响**：资源耗尽，系统不可用
- **修复方案**：使用 try-with-resources 或 finally 块释放；监控连接池状态

### 规则编号：BANK-OTHER-003
- **漏洞类型**：日志注入 / 敏感信息泄露
- **检测要点**：
  - 用户输入直接拼接到日志中（如 `log.info("input:" + userInput)`）
  - 异常堆栈直接返回给前端
- **风险影响**：攻击者可伪造日志、植入恶意内容；堆栈信息暴露内部路径
- **修复方案**：对日志参数进行转义；全局异常处理返回统一错误码

## 八、漏洞位置标注与报告输出要求

AI 在检测到漏洞时，应按以下格式输出：

```
[漏洞报告]
规则编号: BANK-XXX-XXX
文件路径: src/main/java/com/bank/.../AccountService.java
行号范围: 45-52
代码片段:
BigDecimal amount = request.getAmount();
account.setBalance(account.getBalance().subtract(amount));
风险说明: 未校验 amount 是否为负数或超过余额，可能导致余额变为负值或负数交易
修复建议: 在扣款前增加 if (amount.compareTo(BigDecimal.ZERO) <= 0) throw ...; 并且校验余额是否足够


## 九、适用范围与排除项
- 适用语言：Java（银行核心系统主流），可扩展至 C++/Go
- 排除项：前端代码、第三方库内部逻辑、测试代码（除非明确要求）
- 假阳性处理：若使用框架自带防御机制（如 Hibernate Validation、Spring Security 全局配置），需结合上下文判断是否已覆盖
