
# 零售银行 AICoding POC - 后续接口需求说明

## 1. 文档目的

本文档用于说明当前 POC 工程在已实现“创建客户、开户、账户详情查询”基础上，后续需要新增的业务能力与接口需求，作为 AI Coding 插件生成代码、补充数据库结构、完善分层逻辑的输入依据。

本文档聚焦以下 4 个功能：

1. 账户入金
2. 行内转账
3. 交易流水查询
4. 幂等控制

---

## 2. 当前项目背景

当前项目为一个基于 Java 17 + Spring Boot 3 + JPA + H2 的零售银行接口研发 POC 工程。

已完成内容：

- 客户创建
- 账户开立
- 账户详情查询
- 统一返回结构 ApiResponse
- 业务异常 BizException
- 全局异常处理 GlobalExceptionHandler
- 数据表：
  - t_customer
  - t_product
  - t_account
  - t_transaction
  - t_idempotent_record

当前项目定位为：

- 提供一个银行接口研发的最小可运行样板
- 支持 AI 插件基于现有工程上下文继续生成代码
- 重点展示“需求 -> 设计 -> 代码 -> 联调”的完整链路

## 4. 业务对象说明

### 4.1 客户
客户是账户的持有人，当前已支持客户创建。

### 4.2 产品
产品定义账户属性，如币种、账户等级、可售状态。

### 4.3 账户
账户是资金承载主体，后续入金和转账均基于账户进行。

### 4.4 交易
交易记录入金、转账等资金变化过程，要求可查询、可按 requestId 追踪。

### 4.5 幂等记录
用于避免客户端因超时、重试、网络抖动导致的重复交易请求。

## 5. 功能需求一：账户入金

### 5.1 功能描述
支持对指定账户执行入金操作，增加账户余额，并记录一笔交易流水。

### 5.2 使用场景
- 柜面存现
- 外部清算资金入账（POC 简化为直接入金）
- 模拟测试账户余额

### 5.3 输入要素
- requestId：请求唯一流水号，用于幂等控制
- accountNo：目标账户号
- amount：入金金额
- remark：备注，可选

### 5.4 输出结果
返回：
- requestId
- txnId
- accountNo
- amount
- balanceAfter
- txnStatus
- txnTime

### 5.5 业务规则
1. requestId 必填，且在同一业务类型下唯一
2. accountNo 必须存在
3. 账户状态必须为 ACTIVE
4. amount 必须大于 0
5. amount 保留 2 位小数
6. 入金成功后账户余额 = 原余额 + 入金金额
7. 必须写入交易流水表
8. 若同一 requestId 重复请求：
   - 若已成功，返回首次成功结果
   - 若处理中，返回“请求处理中”
   - 若已失败，返回首次失败信息或提示更换 requestId

### 5.6 非功能要求
- 入金与余额更新必须在一个事务中完成
- 不允许出现“余额已改但流水未落”或“流水已落但余额未改”
- 接口应与现有工程风格保持一致

---

## 6. 功能需求二：行内转账

### 6.1 功能描述
支持银行内部两个账户之间的转账，扣减转出账户余额，增加转入账户余额，并记录一笔交易流水。

### 6.2 使用场景
- 同客户账户互转
- 行内客户之间转账
- POC 演示借贷双边记账逻辑

### 6.3 输入要素
- requestId：请求唯一流水号
- debitAccountNo：转出账户
- creditAccountNo：转入账户
- amount：转账金额
- remark：备注，可选

### 6.4 输出结果
返回：
- requestId
- txnId
- debitAccountNo
- creditAccountNo
- amount
- debitBalanceAfter
- creditBalanceAfter
- txnStatus
- txnTime

### 6.5 业务规则
1. requestId 必填，且在同一业务类型下唯一
2. debitAccountNo 和 creditAccountNo 均必须存在
3. 转出账户和转入账户都必须为 ACTIVE 状态
4. debitAccountNo 与 creditAccountNo 不能相同
5. amount 必须大于 0
6. 转出账户余额必须充足
7. 两个账户币种必须一致
8. 转账成功后：
   - 转出账户余额 = 原余额 - amount
   - 转入账户余额 = 原余额 + amount
9. 必须写入一笔交易流水
10. 若同一 requestId 重复请求：
    - 若已成功，返回首次成功结果
    - 若处理中，返回“请求处理中”
    - 若已失败，返回首次失败信息或提示更换 requestId

### 6.6 非功能要求
- 转出、转入、落流水必须在同一事务中完成
- 要考虑并发场景下的余额一致性
- 需要避免双账户更新时死锁

## 7. 功能需求三：交易流水查询

### 7.1 功能描述
支持按账户号及其他可选条件查询交易流水列表。

### 7.2 使用场景
- 查询某账户的入金记录
- 查询某账户的转账历史
- 按时间范围查看交易

### 7.3 查询条件
- accountNo：账户号，必填
- txnType：交易类型，可选
- startTime：开始时间，可选
- endTime：结束时间，可选
- pageNo：页码，可选，默认 1
- pageSize：每页条数，可选，默认 20，最大 100

### 7.4 返回结果
返回分页结构，包含：
- total
- pageNo
- pageSize
- records 列表

每条记录包含：
- txnId
- requestId
- direction（DEBIT / CREDIT）
- counterpartyAccountNo
- txnType
- amount
- txnStatus
- remark
- txnTime
- balanceAfter（针对当前查询账户视角）

### 7.5 业务规则
1. accountNo 必填
2. accountNo 必须存在
3. startTime、endTime 若同时存在，则 startTime <= endTime
4. 默认按 txnTime 倒序返回
5. 查询账户作为借方或贷方的交易都应返回
6. direction 根据查询账户视角判断：
   - 查询账户为借方 -> DEBIT
   - 查询账户为贷方 -> CREDIT
7. 入金交易无对手账号时，counterpartyAccountNo 允许为空

## 8. 功能需求四：幂等控制

### 8.1 功能描述
对入金与转账接口增加幂等控制，避免重复请求导致重复入账或重复扣款。

### 8.2 适用范围
- 账户入金
- 行内转账

### 8.3 幂等主键
- requestId + businessType

其中：
- 入金业务类型：DEPOSIT
- 行内转账业务类型：INTERNAL_TRANSFER

### 8.4 幂等状态
建议至少支持：
- PROCESSING：处理中
- SUCCESS：成功
- FAILED：失败

### 8.5 幂等处理规则
1. 首次请求：
   - 创建幂等记录，状态为 PROCESSING
   - 执行业务
   - 成功则更新为 SUCCESS
   - 失败则更新为 FAILED
2. 重复请求：
   - 若状态为 SUCCESS，则直接返回原处理结果
   - 若状态为 PROCESSING，则返回“处理中”
   - 若状态为 FAILED，则返回原错误或提示更换 requestId

## 10. 返回码建议

### 10.1 成功
- B0000：成功

### 10.2 参数类
- P0001：参数校验失败

### 10.3 账户类
- ACCT404：账户不存在
- ACCT001：账户状态不可用
- ACCT002：账户余额不足
- ACCT003：转出与转入账户不能相同
- ACCT004：账户币种不一致

### 10.4 交易类
- TXN404：交易不存在
- TXN001：交易查询条件非法

### 10.5 幂等类
- IDEMP001：请求正在处理中
- IDEMP002：重复请求已成功处理
- IDEMP003：请求已失败，请更换 requestId 重试

### 10.6 产品/客户扩展类
保留沿用现有规则，不在本轮新增

## 11. 与现有工程的关系

本轮新增功能必须遵循当前工程已有规范：

1. 包结构不变
2. 继续使用 ApiResponse 统一返回
3. 继续使用 BizException 处理业务异常

8. 数据库继续使用 H2
