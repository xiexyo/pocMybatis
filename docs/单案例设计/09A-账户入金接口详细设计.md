# 案例A - 账户入金接口详细设计

## 1. 文档目的

本文档用于指导 AI 基于当前工程新增“账户入金接口”，用于演示银行资金类接口的新增开发能力。

实现本案例前，AI 必须先读取：
- `docs/09-00-全局约束与工程基线.md`

---

## 2. 目标

在当前工程基础上新增账户入金能力，要求：

1. 支持对指定账户入金
2. 更新账户余额
3. 写入交易流水
4. 支持 requestId 幂等控制
5. 返回首次成功结果重放
6. 保持当前工程风格不变
7. 所有数据库访问统一通过 MyBatis Mapper + XML 实现

---

## 3. 业务规则

1. `requestId` 必填
2. `accountNo` 必填
3. `amount` 必须大于 0
4. 账户必须存在
5. 账户状态必须为 `ACTIVE`
6. 若账户为 `FROZEN`，返回 `ACCT005`
7. 入金成功后账户余额增加
8. 必须写入一笔 `DEPOSIT` 交易流水
9. 同一 `requestId` 重复请求不可重复入账
10. 重复请求若已成功，需返回首次成功结果
11. 重复请求若处于处理中，返回 `IDEMP001`
12. 重复请求若历史已失败，返回 `IDEMP003`

## 8.3 TransactionServiceImpl#deposit 详细流程

### 方法签名
```java
@Transactional
public DepositResponse deposit(DepositRequest request)
```

### 处理流程
1. 校验 `amount.compareTo(BigDecimal.ZERO) > 0`
   - 否则抛 `BizException("TXN001", "交易金额必须大于0")`

2. 查询幂等记录：
   - `businessType = "DEPOSIT"`
   - `businessKey = accountNo`

3. 若幂等记录存在：
   - `SUCCESS` -> 根据 requestId 查询交易流水并返回首次成功结果
   - `PROCESSING` -> 抛 `BizException("IDEMP001", "请求正在处理中")`
   - `FAILED` -> 抛 `BizException("IDEMP003", "请求已失败，请更换requestId重试")`

4. 若不存在幂等记录：
   - 调用 `idempotentService.init(requestId, "DEPOSIT", accountNo)`

5. 调用 `accountMapper.selectByAccountNoForUpdate(accountNo)` 获取账户锁定记录

6. 账户不存在：
   - 调用 `idempotentService.markFailed(...)`
   - 抛 `BizException("ACCT404", "账户不存在")`

7. 若账户状态为 `FROZEN`：
   - 调用 `idempotentService.markFailed(...)`
   - 抛 `BizException("ACCT005", "账户已冻结")`

8. 若账户状态不是 `ACTIVE`：
   - 调用 `idempotentService.markFailed(...)`
   - 抛 `BizException("ACCT001", "账户状态不可用")`

9. 计算：
   - `balanceAfter = currentBalance.add(amount)`

10. 更新账户余额和 `updatedTime`
- 调用 `accountMapper.updateBalanceByAccountNo(accountNo, balanceAfter, now)`

11. 生成 `txnId`

12. 保存交易流水：
- `txnType = DEPOSIT`
- `txnStatus = SUCCESS`
- `debitAccountNo = null`
- `creditAccountNo = accountNo`
- `debitBalanceAfter = null`
- `creditBalanceAfter = balanceAfter`
- `amount = request.amount`
- `remark = request.remark`
- `txnTime = now`

13. 调用 `idempotentService.markSuccess(requestId, "DEPOSIT", "B0000", "success")`

14. 返回 `DepositResponse`

### 异常处理
- `BizException`：标记幂等失败后继续抛
- `Exception`：标记幂等失败后继续抛出，由全局异常处理器统一处理

说明：
- 幂等失败标记建议在非重复成功重放场景下执行
- 若 `SUCCESS` 重放场景直接返回结果，则不应覆盖原幂等状态

## 9. 首次结果重放规则

当幂等记录状态为 `SUCCESS` 时：
- 使用 `transactionMapper.selectByRequestId(requestId)` 查询交易记录
- 构造 `DepositResponse`

### 字段映射规则

| 字段 | 取值 |
|---|---|
| requestId | transaction.requestId |
| txnId | transaction.txnId |
| accountNo | transaction.creditAccountNo |
| amount | transaction.amount |
| balanceAfter | transaction.creditBalanceAfter |
| txnStatus | transaction.txnStatus |
| txnTime | transaction.txnTime |

说明：
- 因此 `TransactionEntity` 必须增加：
   - `debitBalanceAfter`

                             @Param("balance") BigDecimal balance,

### AccountMapper.xml 要求

#### 锁定查询 SQL
```sql
SELECT
    account_no,
    customer_id,
    account_name,
    account_type,
    product_code,
    currency,
    balance,
    account_status,
    branch_code,
    open_date,
    created_time,
    updated_time
FROM t_account
WHERE account_no = #{accountNo}
FOR UPDATE
```

#### 更新余额 SQL
```sql
UPDATE t_account
SET balance = #{balance},
    updated_time = #{updatedTime}
WHERE account_no = #{accountNo}
```

## 12. IdempotentServiceImpl 设计要求

### 表记录使用规则
- requestId：请求唯一号
- businessType：`DEPOSIT`
- businessKey：accountNo
- processStatus：
   - `PROCESSING`
   - `SUCCESS`
   - `FAILED`

### 事务要求
以下方法建议使用：
```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
```
- `init`
- `markSuccess`
- `markFailed`

### 处理逻辑要求
- `getExistingRecord` 优先按 `requestId + businessType` 查询
- 若不存在则返回 null
- `init` 写入一条 PROCESSING 记录
- `markSuccess` 更新状态为 SUCCESS
- `markFailed` 更新状态为 FAILED
