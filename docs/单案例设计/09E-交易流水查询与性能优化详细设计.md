# 案例E - 交易流水查询与性能优化详细设计（基于当前真实工程）

## 1. 文档目的

本文档用于指导在**当前真实工程代码基线**上新增“交易流水查询接口”，并按企业级查询接口要求完成：

- 条件过滤
- 分页查询
- 排序控制
- 结果视角转换
- SQL 动态查询
- 索引优化

本案例用于演示 AI 编码助手在以下场景下的能力：

- 面向现有工程结构的增量开发能力
- 多文件协同生成能力
- 动态 SQL 生成能力
- 查询类接口分页与性能优化能力
- 代码理解与联动改造能力

实现本案例前，必须先读取：

- `docs/09-00-全局约束与工程基线.md`

建议先完成：

- `docs/09A-账户存入接口详细设计.md`
- `docs/09B-行内转账接口详细设计.md`

特别说明：

- 当前真实工程中**尚未**落地 `TransactionController`、`TransactionService`、`TransactionServiceImpl`
- 当前真实工程虽然已有：
   - `TransactionEntity`
   - `TransactionMapper`
   - `TransactionMapper.xml`
   - `t_transaction`
- 但目前仅具备底层基础表和基础查询能力，**尚未形成完整交易查询对外接口**

## 2. 目标

在当前工程基础上实现：

1. 交易流水查询接口
2. 按 `accountNo` 查询账户相关流水
3. 支持 `txnType`、`startTime`、`endTime` 过滤
4. 支持分页
5. 按 `txnTime` 倒序
6. 返回当前账户视角的：
   - `direction`
   - `counterpartyAccountNo`
   - `balanceAfter`
7. 避免低质量查询实现：
   - 不允许全量查出后 Java 内存过滤
   - 不允许 Java 内存分页
8. 所有数据库访问继续通过 **Repository + Mapper + XML** 实现
9. 通过本案例展示 AI 对“查询接口设计 + SQL 动态优化 + 多文件联动修改”的支持能力

---

## 3. 当前工程适配说明

## 3.1 必须遵循的真实工程风格

当前真实工程采用的分层结构为：

```text
Controller -> Service -> Repository -> Mapper -> MyBatis XML -> H2 Database
```

因此本案例必须满足：

- Controller 只接收参数、做参数校验、调用 Service
- Service 编排业务逻辑与参数合法性校验
- Repository 作为对 Mapper 的轻量封装层继续保留
- Mapper 只定义接口
- SQL 全部写在 XML 中
- 不允许 Controller 直接调用 Mapper
- 不允许在 Java 中手写 SQL
- 不允许引入 JPA / Hibernate / MyBatis-Plus / Specification 等新体系

## 4. 业务规则

1. `accountNo` 必填
2. `accountNo` 必须存在
3. `startTime` 和 `endTime` 若同时存在，则必须满足 `startTime <= endTime`
4. `pageNo` 默认 1
5. `pageSize` 默认 20，最大 100
6. 返回当前账户作为借方或贷方的所有交易
7. 默认按 `txnTime DESC` 排序
8. 存入交易允许无对手账号
9. 返回结果中必须计算当前账户视角的：
   - `direction`
   - `counterpartyAccountNo`
   - `balanceAfter`
10. 不允许数据库全量查出后在 Java 中做过滤和分页
11. 过滤、排序、分页必须由数据库完成

## 10.2 TransactionServiceImpl#queryTransactions 详细流程

### 方法签名

```java
@Override
public TransactionQueryResponse queryTransactions(TransactionQueryRequest request)
```

### 处理流程

#### 第1步：校验账户号必填
`accountNo` 已通过 DTO 注解校验，不再在 Service 中重复做空值注解式校验。

---

#### 第2步：校验账户是否存在

调用：

```java
accountRepository.findByAccountNo(request.getAccountNo())
```

若不存在，抛出：

```java
throw new BizException("ACCT404", "账户不存在");
```

---

#### 第3步：初始化分页参数

规则：

```java
pageNo = request.getPageNo() == null || request.getPageNo() < 1 ? 1 : request.getPageNo();
pageSize = request.getPageSize() == null || request.getPageSize() < 1 ? 20 : request.getPageSize();
if (pageSize > 100) {
    pageSize = 100;
}
```

---

#### 第4步：校验时间范围

若：

```java
request.getStartTime() != null
&& request.getEndTime() != null
&& request.getStartTime().isAfter(request.getEndTime())
```

则抛出：

```java
throw new BizException("TXN002", "开始时间不能大于结束时间");
```

---

#### 第5步：计算分页偏移量

```java
offset = (pageNo - 1) * pageSize;
```

---

#### 第6步：查询总数

调用：

```java
transactionRepository.countTransactionList(
    request.getAccountNo(),
    request.getTxnType(),
    request.getStartTime(),
    request.getEndTime()
)
```

---

#### 第7步：查询分页数据

调用：

```java
transactionRepository.queryTransactionList(
    request.getAccountNo(),
    request.getTxnType(),
    request.getStartTime(),
    request.getEndTime(),
    offset,
    pageSize
)
```

说明：

- 必须由数据库完成过滤、排序、分页
- 不允许先全量查出再 Java 内存处理

#### 第8步：结果转换

转换时必须依据**当前查询账户视角**计算：

- `direction`


字段包括：

- `total`
- `pageNo`
- `pageSize`
- `records`

## 11. 结果转换规则

对于每一条 `TransactionEntity`，根据当前 `accountNo` 判断返回视角。

## 11.1 若当前账户为借方

条件：

```text
accountNo == transaction.debitAccountNo
```

则返回：

- `direction = "DEBIT"`
- `counterpartyAccountNo = transaction.creditAccountNo`
- `balanceAfter = transaction.debitBalanceAfter`

---

## 11.2 若当前账户为贷方

条件：

```text
accountNo == transaction.creditAccountNo
```

则返回：

- `direction = "CREDIT"`
- `counterpartyAccountNo = transaction.debitAccountNo`
- `balanceAfter = transaction.creditBalanceAfter`

---

## 11.3 若是存入交易

由于存入场景通常：

```text
debitAccountNo = null
creditAccountNo = 当前账户
```

因此返回：

- `direction = "CREDIT"`
- `counterpartyAccountNo = null`
- `balanceAfter = creditBalanceAfter`

---

## 11.4 异常数据兜底建议

理论上查询条件已限定：

```sql
debit_account_no = accountNo OR credit_account_no = accountNo
```

但若出现脏数据导致当前账户既不是借方也不是贷方，建议：

- 跳过该条并记录内部日志（若后续补日志）
- 或根据实际治理要求抛出系统异常

POC 阶段建议优先保持简单，可直接按正常逻辑假设数据一致。

## 13. TransactionMapper.xml 设计要求

## 13.1 count SQL

必须支持动态条件：

- `accountNo`
- `txnType`
- `startTime`
- `endTime`

逻辑核心：

```sql
(debit_account_no = #{accountNo} OR credit_account_no = #{accountNo})
```

同时根据条件动态拼接：

- `txn_type = #{txnType}`（当有值时）
- `txn_time >= #{startTime}`（当有值时）
- `txn_time <= #{endTime}`（当有值时）

---

## 13.2 list SQL

必须支持：

- 与 count 相同的动态过滤条件
- `ORDER BY txn_time DESC, created_time DESC`
- `LIMIT #{pageSize} OFFSET #{offset}`

建议同时查询以下字段：

- `txn_id`
- `request_id`
- `debit_account_no`
- `credit_account_no`
- `txn_type`
- `txn_status`
- `amount`
- `remark`
- `txn_time`
- `created_time`
- `debit_balance_after`
- `credit_balance_after`

说明：

## 13.3 动态 SQL 建议写法

建议在 XML 中使用：

```xml
<where>
    (debit_account_no = #{accountNo} OR credit_account_no = #{accountNo})
    <if test="txnType != null and txnType != ''">
        AND txn_type = #{txnType}
    </if>
    <if test="startTime != null">
        AND txn_time <![CDATA[>=]]> #{startTime}
    </if>
    <if test="endTime != null">
        AND txn_time <![CDATA[<=]]> #{endTime}
    </if>
</where>
```

## 14. 性能优化要求

AI 在实现本接口时必须遵循以下要求：

1. 不允许全表查询后在内存中过滤
2. 必须使用数据库分页
3. 必须支持条件过滤
4. 必须按 `txnTime DESC` 排序
5. 必须保持响应结构稳定
6. 必须给出或落实索引增强
7. 不得在 Java 中拼接 SQL
