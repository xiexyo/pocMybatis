# 案例B - 行内转账接口详细设计

## 1. 文档目的

本文档用于指导 AI 基于当前工程新增“行内转账接口”，用于演示银行资金类双边账户变更场景下的新增开发能力。

实现本案例前，AI 必须先读取：
- `docs/09-00-全局约束与工程基线.md`
- `docs/09-01-案例A-账户入金接口详细设计.md`

---

## 2. 目标

在当前工程基础上新增行内转账能力，要求：

1. 支持本行两个账户之间转账
2. 扣减转出账户余额
3. 增加转入账户余额
4. 写入一笔转账交易流水
5. 支持 requestId 幂等控制
6. 支持首次成功结果重放
7. 保持当前工程风格不变
8. 所有数据库访问统一通过 MyBatis Mapper + XML 实现

---

## 3. 业务规则

1. `requestId` 必填
2. `debitAccountNo` 必填
3. `creditAccountNo` 必填
4. `amount` 必须大于 0
5. 转出账户必须存在
6. 转入账户必须存在
7. 转出账户与转入账户不能相同
8. 两个账户状态必须为 `ACTIVE`
9. 若任一账户为 `FROZEN`，返回 `ACCT005`
10. 转出账户余额必须充足，否则返回 `ACCT002`
11. 两个账户币种必须一致，否则返回 `ACCT004`
12. 转账成功后：
    - 转出账户余额减少
    - 转入账户余额增加
13. 必须写入一笔 `INTERNAL_TRANSFER` 交易流水
14. 同一 `requestId` 重复请求不可重复扣款或重复入账
15. 重复请求若已成功，需返回首次成功结果
16. 重复请求若处于处理中，返回 `IDEMP001`
17. 重复请求若历史已失败，返回 `IDEMP003`

## 7.2 TransactionServiceImpl#internalTransfer 详细流程

### 方法签名
```java
@Transactional
public InternalTransferResponse internalTransfer(InternalTransferRequest request)
```

### 处理流程
1. 校验 `amount.compareTo(BigDecimal.ZERO) > 0`
   - 否则抛 `BizException("TXN001", "交易金额必须大于0")`

2. 校验 `debitAccountNo` 与 `creditAccountNo` 不相同
   - 相同则抛 `BizException("ACCT003", "转出账户和转入账户不能相同")`

3. 查询幂等记录：
   - `businessType = "INTERNAL_TRANSFER"`
   - `businessKey = debitAccountNo + "->" + creditAccountNo`

4. 若幂等记录存在：
   - `SUCCESS` -> 根据 requestId 查询交易流水并返回首次成功结果
   - `PROCESSING` -> 抛 `BizException("IDEMP001", "请求正在处理中")`
   - `FAILED` -> 抛 `BizException("IDEMP003", "请求已失败，请更换requestId重试")`

5. 若不存在幂等记录：
   - 调用 `idempotentService.init(requestId, "INTERNAL_TRANSFER", debitAccountNo + "->" + creditAccountNo)`

6. 为避免双账户锁顺序不一致导致死锁：
   - 比较两个账号字符串大小
   - 较小的账号先加锁
   - 较大的账号后加锁

7. 分别调用 `accountMapper.selectByAccountNoForUpdate(accountNo)` 查询两个账户

8. 若任一账户不存在：
   - 调用 `idempotentService.markFailed(...)`
   - 抛 `BizException("ACCT404", "账户不存在")`

9. 若任一账户状态为 `FROZEN`：
   - 调用 `idempotentService.markFailed(...)`
   - 抛 `BizException("ACCT005", "账户已冻结")`

10. 若任一账户状态不是 `ACTIVE`：
- 调用 `idempotentService.markFailed(...)`
- 抛 `BizException("ACCT001", "账户状态不可用")`

11. 校验币种一致：
- 不一致则调用 `idempotentService.markFailed(...)`
- 抛 `BizException("ACCT004", "账户币种不一致")`

12. 校验转出账户余额充足：
- `debitBalance.compareTo(amount) < 0`
- 调用 `idempotentService.markFailed(...)`
- 抛 `BizException("ACCT002", "账户余额不足")`

13. 计算余额：
- `debitBalanceAfter = debitBalance.subtract(amount)`
- `creditBalanceAfter = creditBalance.add(amount)`

14. 更新两个账户余额和 `updatedTime`
- 调用 `accountMapper.updateBalanceByAccountNo(...)` 两次

15. 生成 `txnId`

16. 插入交易流水：
- `txnType = INTERNAL_TRANSFER`
- `txnStatus = SUCCESS`
- `debitAccountNo = 转出账户`
- `creditAccountNo = 转入账户`
- `debitBalanceAfter = 转出后余额`
- `creditBalanceAfter = 转入后余额`
- `amount = request.amount`
- `remark = request.remark`
- `txnTime = now`

17. 调用 `idempotentService.markSuccess(requestId, "INTERNAL_TRANSFER", "B0000", "success")`

18. 返回 `InternalTransferResponse`

### 异常处理
- `BizException`：标记幂等失败后继续抛
- `Exception`：标记幂等失败后继续抛出，由全局异常处理器统一处理

## 9. Mapper 设计要求

## 9.1 AccountMapper
必须具备以下方法：
```java
AccountEntity selectByAccountNoForUpdate(@Param("accountNo") String accountNo);

int updateBalanceByAccountNo(@Param("accountNo") String accountNo,
                             @Param("balance") BigDecimal balance,
                             @Param("updatedTime") LocalDateTime updatedTime);
```

### AccountMapper.xml 要求

#### 锁定查询 SQL
```sql
SELECT
    account_no,
    customer_id,
    account_name,
    account_type,
    product_code,
    currency,
    balance,
    account_status,
    branch_code,
    open_date,
    created_time,
    updated_time
FROM t_account
WHERE account_no = #{accountNo}
FOR UPDATE
```

#### 更新余额 SQL
```sql
UPDATE t_account
SET balance = #{balance},
    updated_time = #{updatedTime}
WHERE account_no = #{accountNo}
```
