# 案例B - 行内转账接口详细设计（基于当前真实工程）

## 1. 文档目的

本文档用于指导在**当前真实工程代码基线**上新增“行内转账接口”，用于演示银行资金类双边账户变更场景下的新增开发能力，同时支撑 AI 编码助手 POC 的以下能力验证：

- 自然语言转多文件代码生成
- 指定技术栈代码生成
- 事务与幂等控制生成
- 双账户加锁顺序控制逻辑生成
- SQL / Mapper / XML 协同生成
- 代码规范、逻辑漏洞与性能优化建议生成
- 单元测试场景生成

实现本案例前，必须先读取：

- `docs/09-00-全局约束与工程基线.md`
- `docs/09-01-案例A-账户存入接口详细设计.md`

特别说明：

- 当前真实工程采用的不是纯 `Controller -> Service -> Mapper` 结构
- 当前真实工程实际为：

```text
Controller -> Service -> Repository -> Mapper -> MyBatis XML -> H2 Database
```

因此本案例必须基于当前真实工程增量实现，不得脱离现有项目结构另起一套实现。

---

## 2. 目标

在当前工程基础上新增行内转账能力，要求：

1. 支持本行两个账户之间转账
2. 扣减转出账户余额
3. 增加转入账户余额
4. 写入一笔转账交易流水
5. 支持 `requestId` 幂等控制
6. 支持首次成功结果重放
7. 保持当前工程代码风格不变
8. 所有数据库访问继续通过 **Repository + Mapper + XML** 实现
9. 不引入 JPA / Hibernate / MyBatis-Plus
10. 满足 AI 编码插件 POC 的多文件协同生成与逻辑校验演示要求

---

## 3. 业务规则

1. `requestId` 必填
2. `debitAccountNo` 必填
3. `creditAccountNo` 必填
4. `amount` 必须大于 0
5. 转出账户必须存在
6. 转入账户必须存在
7. 转出账户与转入账户不能相同
8. 两个账户状态必须为 `ACTIVE`
9. 若任一账户状态为 `FROZEN`，返回 `ACCT005`
10. 若任一账户状态不是 `ACTIVE` 且不是 `FROZEN`，返回 `ACCT001`
11. 转出账户余额必须充足，否则返回 `ACCT002`
12. 两个账户币种必须一致，否则返回 `ACCT004`
13. 转账成功后：
   - 转出账户余额减少
   - 转入账户余额增加
14. 必须写入一笔 `INTERNAL_TRANSFER` 交易流水
15. 同一 `requestId` 重复请求不可重复扣款或重复入账
16. 重复请求若已成功，需返回首次成功结果
17. 重复请求若处于处理中，返回 `IDEMP001`
18. 重复请求若历史已失败，返回 `IDEMP003`

说明：

- 当前工程 `t_idempotent_record` 以 `request_id` 为主键，因此 `requestId` 在当前项目中应视为**全局唯一**
- 本案例中的幂等接口仍按 `requestId + businessType` 组织，以兼容后续扩展

## 11. TransactionServiceImpl#internalTransfer 详细流程

## 11.1 方法签名

```java
@Override
@Transactional
public InternalTransferResponse internalTransfer(InternalTransferRequest request)
```

---

## 11.2 处理流程

### 第1步：校验金额必须大于0

```java
request.getAmount().compareTo(BigDecimal.ZERO) > 0
```

否则抛出：

```java
throw new BizException("TXN001", "交易金额必须大于0");
```

---

### 第2步：校验转出和转入账户不能相同

若：

```java
request.getDebitAccountNo().equals(request.getCreditAccountNo())
```

则抛出：

```java
throw new BizException("ACCT003", "转出账户和转入账户不能相同");
```

---

### 第3步：查询幂等记录

设置：

```text
businessType = "INTERNAL_TRANSFER"
businessKey = debitAccountNo + "->" + creditAccountNo
```

调用：

```java
idempotentService.getExistingRecord(request.getRequestId(), "INTERNAL_TRANSFER")
```

---

### 第4步：处理重复请求场景

#### 4.1 若幂等记录存在且状态为 `SUCCESS`
- 查询交易流水：
  ```java
  transactionRepository.findByRequestId(requestId)
  ```
- 返回首次成功结果
- 不再重复扣款或入账
- 不覆盖原幂等状态

#### 4.2 若幂等记录存在且状态为 `PROCESSING`
抛出：

```java
throw new BizException("IDEMP001", "请求正在处理中");
```

#### 4.3 若幂等记录存在且状态为 `FAILED`
抛出：

```java
throw new BizException("IDEMP003", "请求已失败，请更换requestId重试");
```

---

### 第5步：初始化幂等记录

若幂等记录不存在，则调用：

```java
idempotentService.init(requestId, "INTERNAL_TRANSFER", debitAccountNo + "->" + creditAccountNo)
```

写入一条 `PROCESSING` 记录。

---

### 第6步：按固定顺序加锁，避免双账户死锁

为避免并发场景下两个线程锁定账户顺序不一致导致死锁，必须统一加锁顺序：

- 比较两个账号字符串大小
- 较小账号先锁
- 较大账号后锁

示意逻辑：

```java
String firstLockAccountNo;
String secondLockAccountNo;
```

然后依次调用：

```java
accountRepository.lockByAccountNo(firstLockAccountNo);
accountRepository.lockByAccountNo(secondLockAccountNo);
```

---

### 第7步：识别转出账户与转入账户对象

由于加锁顺序可能与业务方向不同，因此在获取两条账户记录后，需要按账号重新映射为：

- `debitAccount`
- `creditAccount`

不得直接假设“先锁的是转出账户”。

---

### 第8步：校验账户是否存在

#### 8.1 转出账户不存在
- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT404", "转出账户不存在");
```

#### 8.2 转入账户不存在
- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT404", "转入账户不存在");
```

说明：

- 当前错误码仍使用 `ACCT404`
- message 可区分具体哪一侧账户不存在

---

### 第9步：校验账户冻结状态

若任一账户状态为 `FROZEN`：

- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT005", "账户已冻结");
```

---

### 第10步：校验账户状态是否可用

若任一账户状态不是 `ACTIVE`：

- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT001", "账户状态不可用");
```

说明：

- 当前真实工程中 `ACCT001` 已在开立账户场景中用于“已存在一类户，不能重复开立”
- 本案例沿用原全局预留语义表示“账户状态不可用”
- 后续若要统一错误码语义，应在全局错误码文档中统一梳理

---

### 第11步：校验账户币种是否一致

若：

```java
!debitAccount.getCurrency().equals(creditAccount.getCurrency())
```

则：

- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT004", "账户币种不一致");
```

---

### 第12步：校验转出账户余额是否充足

若：

```java
debitAccount.getBalance().compareTo(request.getAmount()) < 0
```

则：

- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT002", "账户余额不足");
```

---

### 第13步：计算转账后余额

```java
debitBalanceAfter = debitAccount.getBalance().subtract(request.getAmount());
creditBalanceAfter = creditAccount.getBalance().add(request.getAmount());
```

---

### 第14步：更新两个账户余额

调用：

```java
accountRepository.updateBalance(debitAccountNo, debitBalanceAfter, now);
accountRepository.updateBalance(creditAccountNo, creditBalanceAfter, now);
```

说明：

- 两次更新都在同一个业务事务中完成
- 任一步失败应整体回滚

---

### 第15步：生成交易流水号

建议沿用当前工程 ID 风格，生成规则可定义为：

```text
T + yyyyMMddHHmmss + 6位随机数
```

例如：

```java
private String generateTxnId()
```

---

### 第16步：保存交易流水

保存字段要求：

- `txnId`
- `requestId`
- `debitAccountNo = 转出账户号`
- `creditAccountNo = 转入账户号`
- `txnType = INTERNAL_TRANSFER`
- `txnStatus = SUCCESS`
- `amount = request.amount`
- `remark = request.remark`
- `txnTime = now`
- `createdTime = now`
- `debitBalanceAfter = 转出后余额`
- `creditBalanceAfter = 转入后余额`

调用：

```java
transactionRepository.save(entity);
```

---

### 第17步：更新幂等状态为成功

调用：

```java
idempotentService.markSuccess(requestId, "INTERNAL_TRANSFER", "B0000", "success");
```

---

### 第18步：返回响应结果

返回：

```java
InternalTransferResponse
```

字段映射如下：

- `requestId = request.requestId`
- `txnId = entity.txnId`
- `debitAccountNo = request.debitAccountNo`
- `creditAccountNo = request.creditAccountNo`
- `amount = request.amount`
- `debitBalanceAfter = debitBalanceAfter`
- `creditBalanceAfter = creditBalanceAfter`
- `txnStatus = SUCCESS`
- `txnTime = now`

---

## 11.3 异常处理要求

### 业务异常
- `BizException`：若本次已初始化幂等记录，则先标记失败再继续抛出

### 系统异常
- `Exception`：若本次已初始化幂等记录，则先标记失败再继续抛出，由 `GlobalExceptionHandler` 统一处理

### 注意事项
- 成功重放场景不得覆盖已有幂等状态
- 建议通过 `boolean idempotentInitialized` 标识当前请求是否已完成幂等初始化，避免误更新状态

- 使用 `transactionRepository.findByRequestId(requestId)` 查询交易记录
- 构造 `InternalTransferResponse`
- 直接返回，不再重复执行扣款、入账和写流水

| debitAccountNo | transaction.debitAccountNo |
| creditAccountNo | transaction.creditAccountNo |
| amount | transaction.amount |
| debitBalanceAfter | transaction.debitBalanceAfter |
