# 案例B - 行内转账与幂等控制详细设计

## 1. 文档目的

本文档用于指导 AI 在当前工程中新增“行内转账接口”，并完整接入幂等控制、事务控制、双账户加锁和流水记录。

实现本案例前，AI 必须先读取：
- `docs/09-00-全局约束与工程基线.md`

建议先完成：
- `docs/09A-账户入金接口详细设计.md`

## 2. 目标

在当前工程基础上实现：
1. 行内转账接口
2. 双账户余额更新
3. 交易流水落表
4. requestId 幂等控制
5. 首次成功结果重放
6. 死锁规避
7. 事务一致性保障

---

## 3. 业务规则

1. `requestId` 必填
2. `debitAccountNo` 必填
3. `creditAccountNo` 必填
4. `amount` 必须大于 0
5. 转出账户和转入账户不能相同
6. 两个账户必须存在
7. 两个账户状态必须为 `ACTIVE`
8. 冻结账户不可参与转账
9. 两个账户币种必须一致
10. 转出账户余额必须充足
11. 必须写入一笔 `INTERNAL_TRANSFER` 流水
12. 重复 requestId 不得重复扣款
13. 已成功 requestId 要返回首次成功结果

## 8.2 TransactionServiceImpl#internalTransfer 详细流程

### 方法签名
```java
@Transactional
public InternalTransferResponse internalTransfer(InternalTransferRequest request)
```

### 处理流程
1. 校验 `amount.compareTo(BigDecimal.ZERO) > 0`
    - 否则抛 `BizException("TXN001", "交易金额必须大于0")`
2. 校验 `debitAccountNo` 与 `creditAccountNo` 不相同
    - 否则抛 `BizException("ACCT003", "转出账户和转入账户不能相同")`
3. 查询幂等记录：
    - businessType = `INTERNAL_TRANSFER`
    - businessKey = `debitAccountNo + "->" + creditAccountNo`
4. 若幂等记录存在：
    - `SUCCESS` -> 根据 `requestId` 查询交易流水并返回首次成功结果
    - `PROCESSING` -> 抛 `BizException("IDEMP001", "请求正在处理中")`
    - `FAILED` -> 抛 `BizException("IDEMP003", "请求已失败，请更换requestId重试")`
5. 若不存在幂等记录：
    - 调用 `idempotentService.init(requestId, "INTERNAL_TRANSFER", debitAccountNo + "->" + creditAccountNo)`
6. 为避免死锁，先对账号排序：
    - `first = min(debitAccountNo, creditAccountNo)`
    - `second = max(debitAccountNo, creditAccountNo)`
7. 依次使用 `accountRepository.lockByAccountNo(first)`、`lockByAccountNo(second)` 加锁
8. 再根据账号匹配出实际的：
    - 转出账户 `debitAccount`
    - 转入账户 `creditAccount`
9. 若任一账户不存在：
    - 调用 `idempotentService.markFailed(...)`
    - 抛 `BizException("ACCT404", "账户不存在")`
10. 校验转出账户状态：
- 若为 `FROZEN` -> `ACCT005`
- 若非 `ACTIVE` -> `ACCT001`
11. 校验转入账户状态：
- 若为 `FROZEN` -> `ACCT005`
- 若非 `ACTIVE` -> `ACCT001`
12. 校验币种一致：
- 不一致则抛 `BizException("ACCT004", "账户币种不一致")`
13. 校验余额充足：
- `debitAccount.getBalance().compareTo(amount) >= 0`
- 否则抛 `BizException("ACCT002", "账户余额不足")`
14. 计算余额：
- `debitBalanceAfter = debitBalance.subtract(amount)`
- `creditBalanceAfter = creditBalance.add(amount)`
15. 更新转出账户余额与 `updatedTime`
16. 更新转入账户余额与 `updatedTime`
17. 生成 `txnId`
18. 保存交易流水：
- `txnType = INTERNAL_TRANSFER`
- `txnStatus = SUCCESS`
- `debitAccountNo = request.debitAccountNo`
- `creditAccountNo = request.creditAccountNo`
- `debitBalanceAfter = debitBalanceAfter`
- `creditBalanceAfter = creditBalanceAfter`
- `amount = request.amount`
- `remark = request.remark`
- `txnTime = now`
19. 调用 `idempotentService.markSuccess(requestId, "INTERNAL_TRANSFER", "B0000", "success")`
20. 返回 `InternalTransferResponse`

### 异常处理
- `BizException`：标记幂等失败后继续抛出
- `Exception`：标记幂等失败后交给全局异常处理器

## 9. 首次成功结果重放规则

当 `requestId` 对应幂等记录为 `SUCCESS` 时：
- 查询 `transactionRepository.findByRequestId(requestId)`
- 构造 `InternalTransferResponse`：

| 字段 | 取值 |
|---|---|
| requestId | transaction.requestId |
| txnId | transaction.txnId |
| debitAccountNo | transaction.debitAccountNo |
| creditAccountNo | transaction.creditAccountNo |
| amount | transaction.amount |
| debitBalanceAfter | transaction.debitBalanceAfter |
| creditBalanceAfter | transaction.creditBalanceAfter |
| txnStatus | transaction.txnStatus |
| txnTime | transaction.txnTime |

## 11. 并发与事务要求

### 11.1 主业务事务
`internalTransfer` 必须加：
```java
@Transactional
```

### 11.2 幂等事务
`init` / `markSuccess` / `markFailed` 建议：
```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
```

### 11.3 死锁规避
对两个账户加锁前必须排序，再依次加锁，不允许直接按请求顺序锁定。
