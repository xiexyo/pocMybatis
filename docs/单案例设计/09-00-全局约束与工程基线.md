# 零售银行 AICoding POC - 全局约束与工程基线

## 1. 文档目的

本文档用于描述当前项目**基于真实工程代码**的工程基线、技术栈、分层结构、数据库现状、编码约束、Mapper/XML 规范和统一错误码。

后续所有功能扩展、文档设计、AI 生成代码，都必须以**当前实际代码实现**为准，禁止脱离现有工程结构另起一套实现。

## 2. 当前项目基线

## 2.1 项目定位

当前项目是一个基于 **Java 17 + Spring Boot 3 + MyBatis XML + H2** 的零售银行账户服务 POC 工程。

当前已实现的核心业务主线：

- 创建客户
- 开立账户
- 查询账户详情
- 健康检查

当前项目目标不是构建完整银行核心系统，而是提供一个：

- 可运行
- 可演示
- 可继续增量开发
- 便于 AI Coding 插件持续生成代码

的标准后端单体工程。

## 2.3 当前工程架构

当前项目实际采用的分层结构为：

```text
Controller -> Service -> Repository -> Mapper -> MyBatis XML -> H2 Database
```

### 各层职责

- **Controller**
    - 暴露 REST 接口
    - 接收请求参数
    - 触发参数校验
    - 返回统一响应对象 `ApiResponse<T>`（健康检查接口除外）

- **Service**
    - 编排业务逻辑
    - 执行业务校验
    - 控制事务边界
    - 抛出 `BizException`

- **Repository**
    - 对 Mapper 做轻量封装
    - 对 Service 提供更稳定的数据访问接口
    - 当前已存在 `CustomerRepository`、`AccountRepository`、`ProductRepository`

- **Mapper**
    - 定义数据库访问方法
    - 不写注解 SQL，仅保留接口定义

- **Mapper XML**
    - 编写实际 SQL
    - 位于 `src/main/resources/mapper`

- **Entity**
    - 用于承载表记录
    - 用于 MyBatis 查询结果映射
    - 不属于 JPA 实体

### 架构约束

- 保持当前单体工程结构，不改为微服务、多模块、DDD 重构
- Controller 不允许直接调用 Mapper
- Controller 不允许写 SQL 或拼接数据库逻辑
- Service 层优先通过现有 Repository 访问数据
- Repository 是**当前工程的自定义封装层**，**不是 Spring Data JPA Repository**
- 禁止引入 JPA / Hibernate / MyBatis-Plus / QueryDSL / Reactive 等替代当前实现体系

## 2.7 当前已实现业务规则

### 2.7.1 创建客户

当前 `CustomerServiceImpl#createCustomer` 的业务规则：

- 证件号码不可重复
- 若证件号码已存在，抛出：
    - `CUST001`：客户证件号码已存在
- 自动生成客户号 `customerId`
- 默认客户状态：
    - `ACTIVE`
- `riskLevel` 若为空或空白，默认赋值：
    - `R2`

### 2.7.2 开立账户

当前 `AccountServiceImpl#openAccount` 的业务规则：

- 校验客户是否存在
    - 不存在抛出 `CUST404`
- 校验产品是否存在
    - 不存在抛出 `PROD404`
- 校验产品是否在售
    - 非 `ON_SALE` 抛出 `PROD001`
- 若产品账户等级为 `CLASS_I`，则校验客户是否已存在**激活状态一类户**
    - 若存在则抛出 `ACCT001`
- 自动生成账号 `accountNo`
- 新开账户默认：
    - `balance = 0`
    - `accountStatus = ACTIVE`

### 2.7.3 查询账户详情

当前 `AccountServiceImpl#getAccountDetail` 的业务规则：

- 账户不存在：`ACCT404`
- 账户对应客户不存在：`CUST404`
- 账户对应产品不存在：`PROD404`

## 4.3 返回规范

### 当前真实情况

- 业务接口统一返回：
  ```java
  ApiResponse<T>
  ```
- 健康检查接口 `/health` 当前直接返回：

### 约束建议

- 新增业务接口应统一返回 `ApiResponse<T>`
- 非业务类探活接口可按当前 `/health` 风格保留简单返回

---

## 4.4 异常规范

业务异常统一使用：

```java
throw new BizException("错误码", "错误信息");
```

当前统一异常处理类为：

- `GlobalExceptionHandler`

处理规则如下：

- `BizException` → HTTP 400
- `MethodArgumentNotValidException` → HTTP 400，错误码 `P0001`
- 其他 `Exception` → HTTP 500，错误码 `S9999`

### 当前实现注意点
`GlobalExceptionHandler` 对系统异常直接返回 `ex.getMessage()`。

因此后续新增代码时必须注意：

- 不要在异常 message 中拼接敏感信息
- 不要输出完整证件号、完整账号等敏感字段

---

## 4.5 金额与时间规范

- 金额：`BigDecimal`
- 时间：`LocalDateTime`

金额比较必须使用：

```java
amount.compareTo(BigDecimal.ZERO)
```

禁止使用：

- `==`
- `>`
- `<`
- `equals()`

---

## 4.6 事务规范

当前写操作 Service 已使用：

```java
@Transactional
```

例如：

- `CustomerServiceImpl#createCustomer`
- `AccountServiceImpl#openAccount`

约束：

- 事务定义在 Service 层
- Controller 层不允许加事务
- 不允许在 XML 中表达事务逻辑

---

## 4.7 日志与敏感信息规范

- 不打印完整证件号
- 不打印完整账号
- 不打印敏感数据明文
- 不在异常 message 中暴露敏感字段

POC 阶段即使暂未实现统一脱敏工具，也不得主动输出敏感原文。

---

## 4.8 DTO 规范

- 请求参数放在 `dto.request`
- 响应对象放在 `dto.response`
- Controller 不允许直接返回 Entity
- Service 不允许直接将 Entity 暴露给外部接口

---

## 4.9 SQL 与数据访问规范

当前工程统一使用 **Repository + Mapper + MyBatis XML** 完成数据访问。

必须遵循以下规则：

1. 所有 SQL 写在 `src/main/resources/mapper/*.xml`
2. Mapper 接口只定义方法，不写注解 SQL
3. XML `namespace` 必须与 Mapper 全限定名一致
4. XML `id` 必须与 Mapper 方法名一致
5. 多参数方法必须使用 `@Param`
6. 不允许在 Java 代码中手写 SQL
7. 不允许使用 `${}` 拼接普通查询条件
8. 动态 SQL 优先使用：
    - `<if>`
    - `<where>`
    - `<trim>`
    - `<set>`
9. Service 层优先通过 Repository 访问数据
10. 如新增 Transaction / Idempotent 相关业务，建议延续现有风格补充对应 Repository 层

## 9. MyBatis XML 统一约束

## 9.1 XML 目录

所有 Mapper XML 统一放在：

```text
src/main/resources/mapper
```

## 9.2 namespace 规范

例如：

```xml
<mapper namespace="com.bank.poc.mapper.AccountMapper">
```

必须与 Mapper 接口全限定名完全一致。

## 9.3 参数规范

- 单参数可直接使用 `#{paramName}`
- 多参数必须使用 `@Param`
- 不允许依赖位置参数
- 不允许使用 `${}` 拼接普通查询条件

## 9.4 resultType / resultMap 规范

当前工程简单查询统一优先使用：

```xml
resultType="AccountEntity"
```

并依赖以下配置完成下划线转驼峰映射：

```yaml
mybatis.configuration.map-underscore-to-camel-case: true
```

如后续存在复杂联表查询，可按需引入 `resultMap`，但应保持清晰、简洁。

## 9.5 SQL 风格规范

- SQL 关键字大写
- 字段名使用数据库原始字段名
- 动态查询优先使用 `<where>`
- 更新语句优先使用 `<set>`

    - `entity`
    - `mapper`
    - `mapper xml`
    - 必要的 `repository`
    - `dto`
    - `service`
    - `controller`

## 12. AI 生成代码总原则

AI 在读取后续案例设计文档时，必须遵循以下原则：

1. 基于现有工程增量开发
2. 以真实代码结构为准，不推翻当前实现
3. 优先复用已有类和代码风格
4. 保持 Controller / Service / Repository / Mapper 分层清晰
5. 输出可编译、可运行代码
6. 所有数据库访问必须走 MyBatis XML
7. 不得引入 JPA / Hibernate / Spring Data Repository
8. 不得在 Controller 中直接访问数据库
9. 新增业务接口统一返回 `ApiResponse<T>`（健康检查类接口除外）
10. 所有表结构变更必须同步更新相关 Java / XML / SQL 文件
