# 零售银行 AICoding POC - 全局约束与工程基线

## 1. 文档目的

本文档定义当前项目的工程基线、技术栈、架构约束、编码规范、数据库基础、Mapper/XML 规范和统一错误码，供所有后续案例详细设计共同引用。

所有新增代码都必须基于本文档执行，禁止脱离当前工程另起一套实现。

## 2. 当前项目基线

## 2.1 项目定位
当前项目是一个基于 **Java 17 + Spring Boot 3 + MyBatis + XML + H2** 的零售银行账户服务 POC 工程。

业务主线为：
- 创建客户
- 开立账户
- 查询账户详情
- 后续扩展：存入、转账、冻结、查询、幂等、Mock联调、代码治理

本项目的核心目标不是构建完整银行核心系统，而是为 **AI Coding 插件** 提供一个：
- 可运行
- 可扩展
- 可持续增量生成代码
- 可演示完整研发链路

的标准后端工程。

## 2.3 当前工程架构

项目采用标准分层结构：

```text
Controller -> Service -> Mapper -> MyBatis XML -> H2 Database
```

### 各层职责
- Controller：暴露 REST 接口，接收参数，返回统一响应
- Service：编排业务逻辑、规则校验、事务控制
- Mapper：定义数据访问接口
- Mapper XML：编写自定义 SQL
- Entity：数据库对象承载类 / 查询结果映射对象
- Common/Config：公共响应、异常处理、配置

### 架构约束
- 不允许改回 JPA / Repository 模式
- 不允许绕过 Service 直接在 Controller 中调用 Mapper
- 不允许在 Controller 中直接写 SQL 或拼接数据库逻辑
- 所有数据库访问统一通过 Mapper + XML 实现

## 4. 全局编码规范

## 4.1 包与导包规范

### 必须使用
- `jakarta.validation.*`

### 禁止使用
- `javax.validation.*`

### 说明
当前工程已切换为 MyBatis，因此：
- **Entity 不是 JPA 实体**
- **禁止使用 `jakarta.persistence.*`**
- **禁止添加 `@Entity`、`@Table`、`@Column`、`@Id` 等 JPA 注解**

Entity 在当前工程中的角色是：
- 数据表映射载体
- MyBatis 查询结果承载对象
- Service 层内部领域数据对象

---

## 4.2 命名规范
- Entity：`XXXEntity`
- Mapper：`XXXMapper`
- Mapper XML：`XXXMapper.xml`
- Service：`XXXService`
- Service 实现：`XXXServiceImpl`
- 请求 DTO：`XXXRequest`
- 响应 DTO：`XXXResponse`
- Controller：`XXXController`

---

## 4.3 返回规范
所有 Controller 统一返回：

```java
ApiResponse<T>
```

不得直接返回裸对象，不得各接口自行定义不同返回壳。

---

## 4.4 异常规范
业务异常统一使用：

```java
throw new BizException("错误码", "错误信息");
```

不得在 Service 中直接抛 `RuntimeException` 作为业务异常。

Controller 不得手工捕获业务异常后拼装返回值，统一由 `GlobalExceptionHandler` 处理。

---

## 4.5 金额与时间规范
- 金额：`BigDecimal`
- 时间：`LocalDateTime`

金额比较必须使用：

```java
amount.compareTo(BigDecimal.ZERO)
```

不得使用：
- `==`
- `>`
- `<`
- `equals()` 来判断金额大小

---

## 4.6 事务规范
所有资金类写接口必须在 Service 层使用：

```java
@Transactional
```

幂等状态管理建议使用：

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
```

约束：
- 事务必须定义在 Service 层
- 不允许在 Controller 层加事务
- 不允许将事务逻辑下沉到 XML 中表达

---

## 4.7 日志与敏感信息规范
- 不打印完整证件号
- 不打印完整账号
- 不打印敏感数据明文
- 异常信息不得暴露敏感字段

POC 阶段如未实现脱敏工具类，也不得主动输出完整敏感数据。

---

## 4.8 DTO 规范
- 请求参数必须使用 `dto.request`
- 响应对象必须使用 `dto.response`
- 不允许 Controller 直接返回 Entity
- 不允许 Service 直接将 Entity 暴露给外部接口

---

## 4.9 SQL 编写规范
当前工程统一使用 **MyBatis XML** 编写 SQL，必须遵循以下规则：

1. 所有 SQL 写在 `resources/mapper/*.xml`
2. Mapper 接口只定义方法，不写注解 SQL
3. XML `namespace` 必须与 Mapper 接口全限定名一致
4. XML 中 `id` 必须与 Mapper 方法名一致
5. 优先使用参数名绑定，不允许字符串拼接 SQL
6. 动态查询优先使用：
    - `<if>`
    - `<where>`
    - `<trim>`
    - `<set>`
7. 不允许在 Java 代码中手写 SQL
8. 不允许引入 MyBatis-Plus 替代 XML

## 5. 全局错误码

### 通用
- `B0000`：成功

- `CUST404`：客户不存在

### 产品类
- `PROD404`：产品不存在
- `PROD001`：产品当前不可售

### 账户类
- `ACCT404`：账户不存在
- `ACCT001`：账户状态不可用
- `ACCT002`：账户余额不足

- `IDEMP001`：请求正在处理中
- `IDEMP002`：请求已成功处理
- `IDEMP003`：请求已失败，请更换requestId重试

### 风控类
- `RISK001`：风险校验未通过

## 8. Mapper 统一增强要求

## 8.1 AccountMapper
必须支持按账号查询账户：

```java
AccountEntity selectByAccountNo(String accountNo);
```

在资金类场景中，建议支持加锁查询方法，例如：

```java
AccountEntity lockByAccountNo(String accountNo);
```

对应 XML SQL 建议使用：

```sql
SELECT ...
FROM t_account
WHERE account_no = #{accountNo}
FOR UPDATE
```

说明：
- 锁查询必须写在 XML 中
- 不允许使用 JPA 的 `@Lock`
- 转账场景中需要考虑双账户加锁顺序一致，避免死锁

---

## 8.2 TransactionMapper
必须支持：
- 按 `requestId` 查询交易
- 按账户号查询交易
- 按时间范围查询交易

后续如实现分页和复杂条件过滤，优先在 XML 中使用动态 SQL。

建议保留或新增能力：
```java
TransactionEntity selectByRequestId(String requestId);
List<TransactionEntity> selectByAccountNo(String accountNo);
```

若支持复杂条件查询，可扩展：
```java
List<TransactionEntity> queryTransactionList(...);
Long countTransactionList(...);
```

---

## 8.3 IdempotentRecordMapper
必须支持：

```java
IdempotentRecordEntity selectByRequestId(String requestId);
```

如业务要求按 `requestId + businessType` 联合查询，可新增：

```java
IdempotentRecordEntity selectByRequestIdAndBusinessType(String requestId, String businessType);
```

若增加该能力，必须同步修改：
- Mapper 接口
- Mapper XML
- 若涉及唯一性约束，还需同步 `schema.sql`

---

## 9. MyBatis XML 统一约束

## 9.1 XML 目录
所有 Mapper XML 统一放在：

```text
src/main/resources/mapper
```

## 9.2 namespace 规范
例如：

```xml
<mapper namespace="com.bank.poc.mapper.AccountMapper">
```

必须与接口全路径完全一致。

## 9.3 参数规范
- 单参数可直接使用 `#{paramName}`
- 多参数必须在 Mapper 接口中使用 `@Param`
- 不允许依赖位置参数
- 不允许使用 `${}` 拼接普通查询值

## 9.4 resultType / resultMap 规范
当前简单查询默认优先使用：

```xml
resultType="AccountEntity"
```

若后续存在：
- 联表查询
- 字段别名复杂映射
- 嵌套对象映射

则允许引入 `resultMap`，但必须保持清晰简洁，不得过度复杂化。

## 9.5 SQL 风格规范
- SQL 关键字大写
- 字段名使用数据库原始字段名
- 结果映射依赖 `map-underscore-to-camel-case: true`
- 多条件查询优先使用 `<where>`
- 更新语句优先使用 `<set>`

## 10. AI 生成代码总原则

AI 在读取各案例文档时，必须遵循以下原则：

1. 基于现有工程增量开发
2. 优先复用已有类和风格
3. 不推翻当前项目结构
4. 不引入额外中间件
5. 保持代码职责清晰
6. 输出可编译、可运行代码
7. 所有表结构变更必须同步：
    - `schema.sql`
    - `entity`
    - `mapper`
    - `mapper xml`
    - `dto`
    - `service`
    - `controller`
8. 所有数据库访问必须走 Mapper + XML
9. 不得引入 JPA / Repository / Hibernate 实体注解
10. 若新增查询能力，优先通过 XML 动态 SQL 实现
