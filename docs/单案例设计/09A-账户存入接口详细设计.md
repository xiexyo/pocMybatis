# 案例A - 账户存入接口详细设计（基于当前真实工程）

## 1. 文档目的

本文档用于指导在**当前真实工程代码基线**上新增“账户存入接口”，用于演示银行资金类接口的新增开发能力，同时支撑 AI 编码助手 POC 的以下能力验证：

- 自然语言转多文件代码生成
- 指定技术栈代码生成（Java 17 / Spring Boot 3 / MyBatis XML / H2）
- SQL 与 Mapper XML 协同生成
- 事务控制与幂等控制生成
- 代码规范校验与优化建议
- 单元测试场景生成

实现本案例前，必须先读取：

- `docs/09-00-全局约束与工程基线.md`

特别说明：  
当前真实工程并非纯 `Controller -> Service -> Mapper` 结构，而是：

```text
Controller -> Service -> Repository -> Mapper -> MyBatis XML -> H2 Database
```

因此本案例详细设计必须遵循当前真实工程风格，不得脱离现有项目结构另起一套实现。

---

## 2. 目标

在当前工程基础上新增账户存入能力，要求：

1. 支持对指定账户存入
2. 更新账户余额
3. 写入交易流水
4. 支持 `requestId` 幂等控制
5. 对重复成功请求返回首次成功结果重放
6. 保持当前工程代码风格不变
7. 所有数据库访问继续通过 **Repository + Mapper + XML** 实现
8. 不引入 JPA / Hibernate / MyBatis-Plus
9. 满足 AI 编码插件 POC 的多文件协同生成演示要求

---

## 3. 业务规则

1. `requestId` 必填
2. `accountNo` 必填
3. `amount` 必须大于 0
4. 账户必须存在
5. 账户状态必须为 `ACTIVE`
6. 若账户状态为 `FROZEN`，返回 `ACCT005`
7. 若账户状态不是 `ACTIVE` 且不是 `FROZEN`，返回 `ACCT001`
8. 存入成功后账户余额增加
9. 必须写入一笔 `DEPOSIT` 交易流水
10. 同一 `requestId` 重复请求不可重复入账
11. 重复请求若已成功，需返回首次成功结果
12. 重复请求若处于处理中，返回 `IDEMP001`
13. 重复请求若历史已失败，返回 `IDEMP003`

说明：

- 当前工程 `t_idempotent_record` 以 `request_id` 为主键，因此 `requestId` 在当前项目中应视为**全局唯一**
- 本案例查询幂等记录时仍按 `requestId + businessType` 组织接口，以兼容后续扩展，但不调整当前表主键设计

## 6. 数据库变更要求

## 6.1 t_transaction 增加余额快照字段

由于当前工程数据库使用的是 **H2 File 模式**：

```yaml
spring.datasource.url: jdbc:h2:file:./data/retail-bank-poc
spring.sql.init.mode: always
```

因此本案例对 `schema.sql` 的修改要兼容：

- **新库初始化**
- **已有本地文件库增量升级**

建议同时做两类处理：

### 方式一：更新建表定义
在 `CREATE TABLE IF NOT EXISTS t_transaction` 中补充字段：

```sql
debit_balance_after  DECIMAL(18,2),
credit_balance_after DECIMAL(18,2)
```

### 方式二：补充增量 DDL
在 `schema.sql` 末尾追加：

```sql
ALTER TABLE t_transaction ADD COLUMN IF NOT EXISTS debit_balance_after DECIMAL(18,2);
ALTER TABLE t_transaction ADD COLUMN IF NOT EXISTS credit_balance_after DECIMAL(18,2);
```

说明：

- 存入场景中：
    - `debit_balance_after = null`
    - `credit_balance_after = 存入后余额`
- 该字段用于：
    1. 重复成功请求结果重放
    2. 后续交易流水查询展示余额快照

---

## 6.2 索引增强

在 `schema.sql` 中补充：

```sql
CREATE INDEX IF NOT EXISTS idx_transaction_credit_account_no
    ON t_transaction(credit_account_no);

CREATE INDEX IF NOT EXISTS idx_transaction_txn_time
    ON t_transaction(txn_time);
```

说明：

- 当前 `t_transaction` 已有 `request_id` 唯一索引
- 本次新增索引用于提升：
    - 按入账账户查询交易
    - 按交易时间查询交易
- 如后续扩展转账查询，可继续新增 `debit_account_no`、`txn_type` 索引

## 11. TransactionServiceImpl#deposit 详细流程

## 11.1 方法签名

```java
@Override
@Transactional
public DepositResponse deposit(DepositRequest request)
```

---

## 11.2 处理流程

### 第1步：校验金额必须大于0

```java
request.getAmount().compareTo(BigDecimal.ZERO) > 0
```

否则抛出：

```java
throw new BizException("TXN001", "交易金额必须大于0");
```

---

### 第2步：查询幂等记录

```text
businessType = "DEPOSIT"
businessKey = accountNo
```

调用：

```java
idempotentService.getExistingRecord(request.getRequestId(), "DEPOSIT")
```

---

### 第3步：处理重复请求场景

#### 3.1 若幂等记录存在且状态为 `SUCCESS`
- 通过 `transactionService` 内部查询或 `transactionRepository.findByRequestId(requestId)` 查询交易流水
- 直接返回首次成功结果
- 不再重复入账
- 不覆盖原幂等状态

#### 3.2 若幂等记录存在且状态为 `PROCESSING`
抛出：

```java
throw new BizException("IDEMP001", "请求正在处理中");
```

#### 3.3 若幂等记录存在且状态为 `FAILED`
抛出：

```java
throw new BizException("IDEMP003", "请求已失败，请更换requestId重试");
```

---

### 第4步：若幂等记录不存在，则初始化

调用：

```java
idempotentService.init(requestId, "DEPOSIT", accountNo);
```

写入一条 `PROCESSING` 状态记录。

---

### 第5步：锁定查询账户

调用：

```java
accountRepository.lockByAccountNo(accountNo)
```

底层 SQL 使用：

```sql
SELECT ... FROM t_account WHERE account_no = #{accountNo} FOR UPDATE
```

---

### 第6步：账户不存在

- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT404", "账户不存在");
```

---

### 第7步：账户已冻结

若 `accountStatus = FROZEN`：

- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT005", "账户已冻结");
```

---

### 第8步：账户状态不可用

若状态不是 `ACTIVE`：

- 调用 `idempotentService.markFailed(...)`
- 抛出：

```java
throw new BizException("ACCT001", "账户状态不可用");
```

说明：

- 当前真实工程中 `ACCT001` 已被开立账户场景使用为“该客户已存在一类户，不能重复开立”
- 本案例沿用原全局预留语义使用 `ACCT001` 表示“账户状态不可用”
- 若后续需要统一错误码语义，可在全局错误码文档中单独梳理

---

### 第9步：计算存入后余额

```java
balanceAfter = account.getBalance().add(request.getAmount());
```

---

### 第10步：更新账户余额

调用：

```java
accountRepository.updateBalance(accountNo, balanceAfter, now);
```

---

### 第11步：生成交易流水号

建议沿用当前工程 ID 生成风格，生成规则可定义为：

```text
T + yyyyMMddHHmmss + 6位随机数
```

例如：

```java
private String generateTxnId()
```

---

### 第12步：保存交易流水

保存字段要求：

- `txnId`
- `requestId`
- `debitAccountNo = null`
- `creditAccountNo = accountNo`
- `txnType = DEPOSIT`
- `txnStatus = SUCCESS`
- `amount = request.amount`
- `remark = request.remark`
- `txnTime = now`
- `createdTime = now`
- `debitBalanceAfter = null`
- `creditBalanceAfter = balanceAfter`

调用：

```java
transactionRepository.save(entity);
```

---

### 第13步：更新幂等状态为成功

调用：

```java
idempotentService.markSuccess(requestId, "DEPOSIT", "B0000", "success");
```

---

### 第14步：返回响应结果

返回：

```java
DepositResponse
```

字段映射：

- `requestId = request.requestId`
- `txnId = entity.txnId`
- `accountNo = accountNo`
- `amount = request.amount`
- `balanceAfter = balanceAfter`
- `txnStatus = SUCCESS`
- `txnTime = now`

---

## 11.3 异常处理要求

### 业务异常
- `BizException`：标记幂等失败后继续抛出

### 系统异常
- `Exception`：标记幂等失败后继续抛出，由 `GlobalExceptionHandler` 统一处理

### 注意事项
- 对于重复成功重放场景，不得覆盖已有幂等状态
- 只有本次首次进入处理流程、且已初始化幂等记录的场景，才执行失败标记
- 建议在代码中通过布尔变量标记 `idempotentInitialized`，避免重复状态覆盖

## 12. 首次成功结果重放规则

当幂等记录状态为 `SUCCESS` 时：

- 使用 `transactionRepository.findByRequestId(requestId)` 查询交易记录
- 构造 `DepositResponse`
- 直接返回，不再重复执行账户余额更新和交易入库

### 字段映射规则

| 响应字段 | 取值 |
|---|---|
| requestId | transaction.requestId |
| txnId | transaction.txnId |
| accountNo | transaction.creditAccountNo |
| amount | transaction.amount |
| balanceAfter | transaction.creditBalanceAfter |
| txnStatus | transaction.txnStatus |
| txnTime | transaction.txnTime |

说明：

- 因此 `TransactionEntity` 必须增加：
    - `debitBalanceAfter`
    - `creditBalanceAfter`

- `markSuccess` 更新状态为 `SUCCESS`
- `markFailed` 更新状态为 `FAILED`
