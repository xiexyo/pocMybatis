# 案例C - 账户冻结与需求变更联动详细设计（基于当前真实工程）

## 1. 文档目的

本文档用于指导在**当前真实工程代码基线**上新增“账户冻结接口”，并联动修改账户详情查询，以及在案例A/案例B已落地的前提下联动校验存入、转账逻辑，用于演示：

- 需求变更下的多文件联动改造能力
- 数据库字段新增后的 Entity / Mapper / XML / DTO / Service / Controller 同步变更能力
- AI 编码插件在“增量改造现有工程”场景下的代码生成、纠错、联动修改和回归校验能力

实现本案例前，必须先读取：

- `docs/09-00-全局约束与工程基线.md`

建议先完成：

- `docs/09A-账户存入接口详细设计.md`
- `docs/09B-行内转账接口详细设计.md`

特别说明：

- 当前真实工程基础能力仅包括：
    - 创建客户
    - 开立账户
    - 查询账户详情
    - 健康检查
- 当前真实工程**尚未包含**存入、转账、冻结接口代码
- 因此本案例分为两层落地目标：
    1. **当前真实工程必做范围**：冻结接口 + 账户详情查询联动
    2. **若案例A/B已落地的扩展范围**：冻结状态对存入、转账逻辑生效

---

## 2. 目标

在当前工程基础上实现以下能力：

1. 新增账户冻结接口
2. 在账户表新增冻结原因字段 `freeze_reason`
3. 账户详情查询返回冻结原因
4. 冻结后账户状态变更为 `FROZEN`
5. 重复冻结同一账户时，推荐直接返回当前冻结结果，不重复更新
6. 若案例A已实现，冻结账户禁止存入
7. 若案例B已实现，冻结账户禁止转账
8. 展示需求变更场景下的多文件联动改造能力
9. 满足 AI 编码插件 POC 中“多文件协同生成、纠错、联动修改、代码理解”的演示要求

## 3. 当前工程适配说明

当前真实工程采用的分层结构为：

```text

```

因此本案例必须遵循以下约束：

- 不允许绕过现有工程结构另起一套实现

## 4. 业务规则

1. `accountNo` 必须存在
2. `freezeReason` 必填
3. 冻结后账户状态改为 `FROZEN`
4. 冻结原因写入 `freeze_reason`
5. 若账户已是 `FROZEN`，推荐直接返回当前冻结结果，不报错、不重复更新
6. 查询账户详情时必须返回 `freezeReason`
7. 若案例A已落地，冻结账户在存入时不可操作
8. 若案例B已落地，冻结账户在转账时不可操作
9. 冻结接口本身不要求 `requestId` 幂等控制，重复调用通过“已冻结直接返回”实现幂等式表现

说明：

- 本案例不涉及解冻接口
- 本案例不要求落冻结操作流水
- `freezeReason` 仅记录原因，不做枚举值限制

## 6. 数据库变更要求

## 6.1 t_account 新增字段

由于当前工程使用的是 **H2 File Database**，并配置了：

```yaml
spring.sql.init.mode: always
```

因此需要同时兼容：

- 新库初始化
- 已存在本地文件库的增量升级

建议两类修改同时进行。

### 方式一：更新建表语句

在 `schema.sql` 中 `CREATE TABLE IF NOT EXISTS t_account` 的定义中新增字段：

```sql
freeze_reason VARCHAR(255)
```

建议放在：

```sql
account_status VARCHAR(16) NOT NULL,
branch_code    VARCHAR(16),
freeze_reason  VARCHAR(255),
open_date      TIMESTAMP NOT NULL,
```

### 方式二：追加增量 DDL

在 `schema.sql` 末尾追加：

```sql
ALTER TABLE t_account ADD COLUMN IF NOT EXISTS freeze_reason VARCHAR(255);
```

说明：

- `freeze_reason` 为可空字段
- 未冻结账户该字段为 `null`

## 11. Service 设计

## 11.1 AccountService 新增方法

```java
FreezeAccountResponse freezeAccount(String accountNo, FreezeAccountRequest request);
```

---

## 11.2 AccountServiceImpl#freezeAccount 详细流程

### 方法签名建议

```java
@Override
@Transactional
public FreezeAccountResponse freezeAccount(String accountNo, FreezeAccountRequest request)
```

### 处理流程

#### 第1步：查询并锁定账户

优先使用：

```java
accountRepository.lockByAccountNo(accountNo)
```

若当前工程尚未补齐锁定查询能力，则本案例应一并新增。

说明：

- 冻结操作属于账户状态变更，建议使用行级锁
- 可避免冻结与后续资金类操作并发冲突

---

#### 第2步：账户不存在

若账户不存在，抛出：

```java
throw new BizException("ACCT404", "账户不存在");
```

---

#### 第3步：账户已冻结

若账户当前状态已为 `FROZEN`：

- 直接返回当前冻结结果
- 不报错
- 不覆盖原冻结原因
- 不重复更新 `updatedTime`

返回字段建议取当前账户已有值：

- `accountNo`
- `accountStatus`
- `freezeReason`
- `updatedTime`

---

#### 第4步：更新账户状态和冻结原因

若账户当前不是 `FROZEN`，则更新：

- `accountStatus = "FROZEN"`
- `freezeReason = request.freezeReason`
- `updatedTime = now`

调用：

```java
accountRepository.updateStatusAndFreezeReason(accountNo, "FROZEN", request.getFreezeReason(), now);
```

---

#### 第5步：返回响应结果

返回：

```java
FreezeAccountResponse
```

字段映射：

- `accountNo = accountNo`
- `accountStatus = "FROZEN"`
- `freezeReason = request.freezeReason`
- `updatedTime = now`

---

## 11.3 设计说明

1. 冻结接口不要求 requestId 幂等
2. 通过“已冻结直接返回当前结果”实现幂等式效果
3. 不记录冻结流水
4. 不做解冻逻辑
5. 若后续增加审计要求，可再增量扩展冻结操作日志
