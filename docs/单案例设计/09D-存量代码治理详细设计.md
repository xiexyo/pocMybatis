# 案例D - 存量代码治理详细设计（基于当前真实工程）

## 1. 文档目的

本文档用于指导 AI 对**当前真实工程代码**或**预置问题代码样例**执行存量代码治理，包括：

- 技术栈与依赖规范检查
- 工程分层与编码风格检查
- 事务一致性与并发风险检查
- 业务逻辑漏洞排查
- 安全与敏感信息风险提示
- 输出修复建议与修复后代码

本案例**不是新增业务接口**，而是面向银行客户演示 AI 编码助手在“**已有代码治理**”场景下的能力，重点体现：

- 对现有工程的理解能力
- 对规范与风险的识别能力
- 对多文件联动修复的建议能力
- 对符合当前工程风格的修复代码生成能力

实现本案例前，必须先读取：

- `docs/09-00-全局约束与工程基线.md`

特别说明：

- 当前真实工程是一个基于 **Java 17 + Spring Boot 3 + MyBatis XML + H2** 的单体工程
- 当前真实工程采用的分层结构为：

```text
Controller -> Service -> Repository -> Mapper -> MyBatis XML -> H2 Database
```

因此本案例所有治理结论和修复建议，都必须以**当前真实工程风格**为准，而不是套用其他项目的 JPA、DDD、MyBatis-Plus 或微服务标准。

验证 AI 是否能够对已有代码完成以下工作：

1. 技术栈与依赖规范检查

3. 统一返回与异常风格检查

6. 安全风险与敏感信息风险提示
7. SQL / Mapper XML 规范检查
8. 输出修复建议和修复后代码
9. 在需求变更或代码不规范场景下，给出符合当前工程风格的联动修复方案

## 6. 必须检查的维度

## 6.1 技术栈规范检查

检查项：

1. 是否使用 `jakarta.validation.*`
2. 是否误用 `javax.validation.*`
3. 是否误用 `jakarta.persistence.*`
4. 是否误用 JPA 注解：
    - `@Entity`
    - `@Table`
    - `@Column`
    - `@Id`
5. 是否符合 Spring Boot 3 / Jakarta 体系
6. 是否与当前 MyBatis XML 工程基线一致

### 发现问题后输出要求

必须输出：

- 具体文件路径
- 具体 import 或注解位置
- 不符合当前工程基线的原因
- 修复建议
- 修复后代码

说明：

- 当前真实工程**不使用 JPA**
- 因此若扫描到任何 JPA 注解或 `jakarta.persistence.*`，应直接标记为规范问题

---

## 6.2 工程风格与分层检查

检查项：

1. Controller 是否统一返回 `ApiResponse<T>`
    - 说明：`/health` 为特例，可保留简单返回
2. Controller 是否直接写业务逻辑
3. Controller 是否直接访问 Mapper / Repository
4. Service 是否通过 `BizException` 抛业务异常
5. 是否存在 `RuntimeException` 直接表示业务错误
6. Repository 是否只是对 Mapper 的轻量封装
7. Mapper 是否只定义接口，不写注解 SQL
8. SQL 是否全部写在 XML 中
9. DTO、Service、Repository、Mapper 命名是否符合当前工程规范
10. 是否存在职责不清的问题，如：
    - Controller 中大量业务编排
    - Service 中拼接 SQL
    - Mapper 中写业务判断

---

## 6.3 返回与异常处理规范检查

检查项：

1. 业务接口是否统一使用 `ApiResponse<T>`
2. 是否存在直接返回 Entity 的情况
3. 是否存在返回裸对象、不符合统一响应规范的情况
4. 是否通过 `BizException` 抛业务异常
5. 是否绕过 `GlobalExceptionHandler` 自行拼装错误返回
6. 参数校验失败是否能统一映射为 `P0001`
7. 系统异常是否被统一处理为 `S9999`
8. 异常 message 是否可能暴露敏感数据

---

## 6.4 事务一致性检查

检查项：

1. 写操作 Service 方法是否加 `@Transactional`
2. 事务是否定义在 Service 层，而不是 Controller 层
3. 是否存在“部分成功”风险，例如：
    - 先改余额后落流水
    - 先写交易再更新账户，异常时不一致
4. 幂等状态更新是否存在回滚丢失风险
    - 若案例A/B已落地，应重点检查 `REQUIRES_NEW`
5. 是否遗漏悲观锁 / 锁定查询，导致并发覆盖余额风险
6. 是否存在多账户更新时锁顺序不一致问题
    - 若案例B已落地，应重点检查死锁风险

---

## 6.5 业务逻辑检查

检查项：

1. 是否校验金额 `> 0`
2. 是否校验账户存在
3. 是否校验客户存在
4. 是否校验产品存在
5. 是否校验产品是否可售
6. 是否校验账户状态
7. 是否校验冻结账户不可操作
    - 若案例C或冻结逻辑已落地
8. 是否校验借贷账户不能相同
    - 若案例B已落地
9. 是否校验余额充足
    - 若案例B已落地
10. 是否校验币种一致
    - 若案例B已落地
11. 是否校验 requestId 幂等
    - 若案例A/B已落地
12. 查询接口是否校验时间范围合法
    - 若后续交易查询已落地
13. 是否存在空指针风险、边界值遗漏、默认值处理不一致等问题

---

## 6.6 SQL 与 Mapper XML 规范检查

检查项：

1. SQL 是否全部写在 XML 中
2. XML `namespace` 是否与 Mapper 全限定名一致
3. XML `id` 是否与 Mapper 方法名一致
4. 多参数方法是否使用 `@Param`
5. 是否错误使用 `${}` 拼接普通查询条件
6. 查询字段是否与 Entity 保持一致
7. 新增字段后是否遗漏 XML 联动修改
8. 是否存在 SQL 风格不统一问题：
    - 关键字未大写
    - 字段名不规范
    - 动态 SQL 不清晰
9. 是否存在索引缺失导致的性能风险
10. 是否存在表结构、Entity、Mapper、XML 不一致的问题

---

## 6.7 安全与日志检查

检查项：

1. 是否打印完整账号
2. 是否打印完整证件号
3. 是否打印敏感数据明文
4. 是否在异常信息中暴露敏感字段
5. 是否存在缺少幂等保护导致重复扣款或重复入账风险
    - 若案例A/B已落地
6. 是否存在状态校验遗漏导致冻结账户可交易风险
    - 若案例C已落地
7. 是否存在未校验输入边界导致异常或污染数据风险
8. 是否存在返回过多内部实现细节的风险

---

## 6.8 性能与可维护性检查

检查项：

1. 是否存在重复查库可合并的情况
2. 是否存在不必要的对象转换或冗余代码
3. 是否存在生成 ID 逻辑重复、可抽取的问题
4. 是否存在 SQL 可优化问题
5. 是否缺少必要索引
6. 是否存在可读性差、方法过长、职责混乱问题
7. 是否存在多文件联动修改时遗漏同步的问题

---

## 7. 治理输出格式要求

AI 在执行本案例时，应按以下结构输出，不能只给笼统建议。

---

## 7.1 问题清单

每个问题至少包含：

- 问题编号
- 问题标题
- 问题类型：
    - 规范
    - 逻辑
    - 安全
    - 性能
    - 可维护性
- 风险等级：
    - 高
    - 中
    - 低
- 文件路径
- 代码位置
- 问题描述
- 风险说明

建议输出格式示例：

```text
问题编号：D-001
问题类型：规范
风险等级：中
文件路径：src/main/java/com/bank/poc/service/impl/XXXServiceImpl.java
代码位置：createXXX 方法
问题描述：直接使用 RuntimeException 表示业务异常
风险说明：不符合当前工程统一异常规范，可能导致错误码不一致
```

---

## 7.2 修复建议

每个问题的修复建议至少包含：

- 最佳修复方式
- 修复原因
- 是否影响现有接口行为
- 是否需要联动修改其他文件
- 是否建议立即修复或后续优化

---

## 7.3 修复后代码

若问题适合自动修复，AI 应输出：

- 修复后的完整方法
- 或修复后的完整类
- 或修复后的完整 XML / SQL 片段

不得只输出一句：

- “建议增加校验”
- “建议补事务”
- “建议统一异常”

而必须尽量给出可直接落地的代码。

---

## 7.4 汇总结论

扫描结束后，应输出总体结论，包括：

1. 当前代码整体健康度评价
2. 高风险问题数量
3. 中低风险问题数量
4. 是否建议立即整改
5. 本次治理影响范围概述

## 9. 基于当前真实工程的推荐治理基线

为了让 AI 输出更贴合实际，建议在治理时先以当前真实工程作为“基线标准”。

---

## 9.1 当前真实工程中应被识别为“正确”的做法

AI 应能识别出当前工程中的一些正向实践，例如：

1. 使用 `jakarta.validation.*`
2. 使用 `BizException` 表达业务异常
3. 使用 `GlobalExceptionHandler` 统一异常处理
4. Controller 统一返回 `ApiResponse<T>`（`/health` 除外）
5. 使用 `Repository -> Mapper -> XML`
6. SQL 统一落在 XML 中
7. 写操作 Service 使用 `@Transactional`
8. 金额字段使用 `BigDecimal`
9. 时间字段使用 `LocalDateTime`

---

## 9.2 当前真实工程中可被识别为“待优化”的方向

AI 也应能发现当前真实工程中仍有可优化项，例如：

1. `GlobalExceptionHandler` 对系统异常直接返回 `ex.getMessage()`，存在暴露内部异常信息风险
2. 错误码 `ACCT001` 在当前真实代码与预留全局语义之间存在语义复用/冲突风险
3. 客户号、账号生成逻辑分散在不同 Service 中，可考虑统一抽取
4. 当前尚未落地资金类接口，交易与幂等相关底层表结构存在但未形成完整闭环
5. 健康检查接口未使用统一响应壳，虽可接受，但应在文档中明确为特例

说明：

- 这些“待优化项”不一定都是必须整改问题
- 可作为 AI 治理分析中的“优化建议”输出，体现理解深度

## 12. AI 输出与实现要求

AI 在本案例中不得只说“建议增加校验”之类笼统话术，必须尽量输出：

- 问题定位
- 风险分类
- 修复方案
- 联动影响文件

同时必须遵守以下要求：

5. 修复代码不得引入新的架构体系
6. 若涉及数据库字段新增，必须同步提示修改：
    - `schema.sql`

    - `mapper xml`
    - `repository`
    - `service`
    - `controller`
7. 若扫描对象为案例A/B/C扩展代码，必须同时检查幂等、事务、锁、余额快照、冻结联动等资金类关键点

## 13. 与 AI 编码插件 POC 要求的对应关系

本案例可直接用于验证 AI 编码插件以下能力：

### 13.1 编码纠错与优化功能
