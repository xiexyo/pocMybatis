# 案例D - 幂等服务详细设计

## 1. 文档目的

本文档用于指导 AI 基于当前工程实现统一的幂等服务能力，用于支撑账户入金、行内转账等写接口，避免重复请求导致重复入账、重复扣款等问题。

实现本案例前，AI 必须先读取：
- `docs/09-00-全局约束与工程基线.md`

---

## 2. 目标

在当前工程基础上新增统一幂等服务能力，要求：

1. 支持按 `requestId + businessType` 判断是否重复请求
2. 支持状态：
    - `PROCESSING`
    - `SUCCESS`
    - `FAILED`
3. 支持写接口首次请求初始化
4. 支持成功后更新为 `SUCCESS`
5. 支持失败后更新为 `FAILED`
6. 支持被入金、转账等业务复用
7. 保持当前工程风格不变
8. 所有数据库访问统一通过 MyBatis Mapper + XML 实现

---

## 3. 业务规则

1. 每个写请求必须携带 `requestId`
2. 幂等服务按 `requestId + businessType` 判断业务唯一性
3. 首次请求先写入 `PROCESSING`
4. 业务执行成功后更新为 `SUCCESS`
5. 业务执行失败后更新为 `FAILED`
6. 若已存在 `SUCCESS` 记录，由业务层决定返回首次成功结果
7. 若已存在 `PROCESSING` 记录，返回 `IDEMP001`
8. 若已存在 `FAILED` 记录，返回 `IDEMP003`

说明：
- 当前 POC 阶段，`requestId` 可默认视为全局唯一
- 但业务代码仍建议按 `requestId + businessType` 查询，提高语义清晰度

## 5. Entity 设计

## 5.1 IdempotentRecordEntity 字段
当前应至少包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| requestId | String | 请求唯一号 |
| businessType | String | 业务类型 |
| businessKey | String | 业务主键，如 accountNo 或 转账路径 |
| processStatus | String | PROCESSING / SUCCESS / FAILED |
| responseCode | String | 响应码 |
| responseMessage | String | 响应说明 |
| createdTime | LocalDateTime | 创建时间 |
| updatedTime | LocalDateTime | 更新时间 |

---

## 6. Service 设计

## 6.1 IdempotentService
建议定义方法：

```java
void init(String requestId, String businessType, String businessKey);

void markSuccess(String requestId, String businessType, String responseCode, String responseMessage);

void markFailed(String requestId, String businessType, String responseCode, String responseMessage);

IdempotentRecordEntity getExistingRecord(String requestId, String businessType);
```

---

## 6.2 IdempotentServiceImpl 设计要求

### 事务要求
以下方法建议使用：

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
```

- `init`
- `markSuccess`
- `markFailed`

### 方法职责

#### getExistingRecord
- 根据 `requestId + businessType` 查询已有幂等记录
- 若不存在返回 `null`

#### init
- 写入一条状态为 `PROCESSING` 的记录
- `responseCode`、`responseMessage` 可先为空
- 若已存在记录，不建议盲目重复插入，应由业务层先调用 `getExistingRecord`

#### markSuccess
- 更新记录为：
    - `processStatus = SUCCESS`
    - `responseCode = B0000`
    - `responseMessage = success`

#### markFailed
- 更新记录为：
    - `processStatus = FAILED`
    - `responseCode = 业务失败码`
    - `responseMessage = 失败说明`

## 7. Mapper 设计要求

## 7.1 IdempotentRecordMapper
建议定义方法：

```java
IdempotentRecordEntity selectByRequestIdAndBusinessType(@Param("requestId") String requestId,
                                                        @Param("businessType") String businessType);

int insert(IdempotentRecordEntity entity);

int updateStatusByRequestIdAndBusinessType(@Param("requestId") String requestId,
                                           @Param("businessType") String businessType,
                                           @Param("processStatus") String processStatus,
                                           @Param("responseCode") String responseCode,
                                           @Param("responseMessage") String responseMessage,
                                           @Param("updatedTime") LocalDateTime updatedTime);
```

可保留已有方法：
```java
IdempotentRecordEntity selectByRequestId(@Param("requestId") String requestId);
List<IdempotentRecordEntity> selectByBusinessType(@Param("businessType") String businessType);
```

---

## 7.2 IdempotentRecordMapper.xml 设计要求

### 查询 SQL
按 `request_id + business_type` 查询：

```sql
SELECT
    request_id,
    business_type,
    business_key,
    process_status,
    response_code,
    response_message,
    created_time,
    updated_time
FROM t_idempotent_record
WHERE request_id = #{requestId}
  AND business_type = #{businessType}
```

### 插入 SQL
插入字段至少包括：
- `request_id`
- `business_type`
- `business_key`
- `process_status`
- `response_code`
- `response_message`
- `created_time`
- `updated_time`

### 更新 SQL
根据 `request_id + business_type` 更新：
- `process_status`
- `response_code`
- `response_message`
- `updated_time`

## 8. 表结构建议

当前表 `t_idempotent_record` 已可支撑本案例。

若后续希望严格保证 `requestId + businessType` 唯一，可增量补充唯一索引，例如：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uk_idempotent_request_business
    ON t_idempotent_record(request_id, business_type);
```

说明：
- 若当前 `request_id` 已是主键，则该索引并非必须
- POC 阶段可保持 requestId 全局唯一，不强制改主键结构

---

## 9. 业务接入规范

幂等服务不直接对外暴露 Controller，由业务 Service 调用。

### 9.1 入金接入方式
1. 查询已有记录
2. 若无记录则 init
3. 业务成功 markSuccess
4. 业务失败 markFailed

### 9.2 转账接入方式
1. 查询已有记录
2. 若无记录则 init
3. 业务成功 markSuccess
4. 业务失败 markFailed

---

## 10. 测试场景

1. 首次请求初始化 PROCESSING
2. 成功后更新 SUCCESS
3. 失败后更新 FAILED
4. 根据 requestId + businessType 查询已有记录
5. 已有 SUCCESS 时业务层重放首次结果
6. 已有 PROCESSING 时业务层返回 `IDEMP001`
7. 已有 FAILED 时业务层返回 `IDEMP003`

---

## 11. AI 输出要求

AI 在实现本案例时必须：
1. 输出完整代码，不只输出片段
2. 保持现有包结构
3. 使用 `BizException`、`GlobalExceptionHandler`
4. 使用 Spring 事务注解
5. 使用 MyBatis `Mapper + XML`
6. 所有 SQL 写入 `IdempotentRecordMapper.xml`
7. 输出可直接编译运行代码
8. 不得引入 JPA Repository 或 Hibernate 注解
9. 不得将幂等逻辑直接写进 Controller
