
# 案例G - 银行标准模板沉淀与复用设计

## 1. 文档目的

本文档不是直接新增业务接口，而是用于指导 AI 总结当前工程中已经形成的标准开发模式，将其沉淀为可复用的银行接口模板。

该文档适合在以下场景使用：

1. 向 AI 说明后续生成代码必须复用当前风格
2. 向领导/客户说明插件具备“模板沉淀与复用”能力
3. 后续新增接口时，统一生成风格和结构

---

## 2. 目标

沉淀以下 3 类模板：

1. 银行标准写接口模板
2. 银行资金交易接口模板
3. 银行标准查询接口模板

---

## 3. 模板1：银行标准写接口模板

## 3.1 适用场景
适用于以下接口：
- 账户冻结
- 账户状态变更
- 账户解冻（若后续扩展）
- 非资金类标准写接口

## 3.2 模板结构
- Request DTO
- Response DTO
- Controller
- Service
- ServiceImpl
- Mapper
- Mapper XML
- 参数校验
- 状态校验
- 统一返回 ApiResponse
- BizException
- Swagger 注解
- 可选 requestId / operatorId / branchCode

## 3.3 模板要求

### Request DTO
必须：
- 使用 `jakarta.validation`
- 字段命名与现有风格一致

### Controller
必须：
- 使用 `@RestController`
- 使用 `@RequestMapping`
- 返回 `ApiResponse<T>`

### Service
必须：
- 核心逻辑在 Service
- 不在 Controller 中写业务逻辑

### 持久层
必须：
- 使用 `Mapper + XML`
- 不使用 JPA Repository
- 不在 Java 中拼接 SQL

### 异常
必须：
- 使用 `BizException`

---

## 4. 模板2：银行资金交易接口模板

## 4.1 适用场景
适用于以下接口：
- 账户入金
- 行内转账
- 冲正/撤销（后续扩展）
- 代扣（后续扩展）

## 4.2 模板结构
- Request DTO
- Response DTO
- Controller
- TransactionService
- TransactionServiceImpl
- Mapper
- Mapper XML
- 幂等控制
- 事务控制
- 金额校验
- 账户状态校验
- 余额校验
- 流水记录
- 成功结果重放

## 4.3 模板要求

### 必备字段
请求中建议包含：
- requestId
- amount
- remark
- channelCode
- branchCode
- operatorId

### 事务要求
资金类接口必须：
```java
@Transactional
```

### 幂等要求
必须：
- 先查幂等
- 成功结果可重放
- 处理中和失败状态可区分

### 账户处理要求
必须：
- 通过 Mapper 锁定账户记录
- 余额更新通过 XML SQL 完成
- 双账户交易考虑锁顺序一致

### 流水要求
必须：
- 落 `t_transaction`
- 记录余额快照
- 使用 Mapper + XML 插入流水

---

## 5. 模板3：银行标准查询接口模板

## 5.1 适用场景
适用于以下接口：
- 交易流水查询
- 账户列表查询（后续扩展）
- 冻结历史查询（后续扩展）

## 5.2 模板结构
- QueryRequest DTO
- ItemResponse DTO
- QueryResponse DTO
- Controller
- Service
- Mapper
- Mapper XML 动态查询
- 分页
- 排序
- 条件校验

## 5.3 模板要求

### 请求
必须支持：
- 分页参数
- 条件参数
- 时间范围参数（如适用）

### 查询实现
必须：
- 数据库分页
- 不允许内存全量过滤
- 不允许 Java 内存分页
- 复杂查询优先使用 XML 动态 SQL

### 响应
统一返回：
- `total`
- `pageNo`
- `pageSize`
- `records`

---

## 6. 模板沉淀方式建议

AI 在读取本项目时，应识别并记忆以下工程习惯：

1. 返回统一使用 `ApiResponse`
2. 业务异常统一使用 `BizException`
3. DTO 按 request/response 分包
4. Service 与 ServiceImpl 分离
5. MyBatis Mapper + XML 风格优先
6. 金额使用 `BigDecimal`
7. 时间使用 `LocalDateTime`
8. Controller 只负责接收与返回，不写业务逻辑
9. 写接口要考虑 `requestId`
10. 资金接口要考虑幂等与事务
11. 查询接口要考虑数据库动态 SQL、分页、排序
12. 所有表结构变更要同步 XML SQL

---

## 7. AI 复用方式建议

后续新增接口时，可直接给 AI 这样的指令：

### 写接口模板指令
```text
请基于当前项目已有的“银行标准写接口模板”生成账户解冻接口，保持与冻结接口相同的风格和结构，并使用 Mapper + XML 实现数据库访问。
```

### 资金交易模板指令
```text
请基于当前项目已有的“银行资金交易接口模板”生成账户扣款接口，保留 requestId、事务、幂等、流水记录、账户锁定等模式，并使用 Mapper + XML 实现。
```

### 查询模板指令
```text
请基于当前项目已有的“银行标准查询接口模板”生成账户交易统计查询接口，保持分页、排序、统一响应结构，并通过 MyBatis XML 动态 SQL 实现查询。
```

---

## 8. AI 输出要求

AI 在使用模板时必须：

1. 复用当前工程风格，而不是重新设计一套
2. 保持统一返回结构
3. 保持统一异常结构
4. 保持现有包结构
5. 不引入额外复杂框架
6. 输出完整代码而不是只给模板描述
7. 所有数据库访问统一使用 Mapper + XML
8. 不得引入 JPA Repository、Specification、MyBatis-Plus 等替代方案
```
