# 案例G - 银行标准模板沉淀与复用设计（基于当前真实工程）

## 1. 文档目的

本文档不是直接新增某一个业务接口，而是用于指导 AI 基于**当前真实工程代码**，总结已经形成的标准开发模式，并沉淀为可复用的银行接口模板。

本案例适用于以下场景：

1. 向 AI 明确：后续生成代码必须复用当前项目风格，而不是另起一套实现
2. 向领导/客户展示：插件具备“模板沉淀与复用”能力
3. 为后续新增接口提供统一模板，降低风格漂移和重复设计成本
4. 支撑 AI 编码助手 POC 中“场景化模板、团队规范记忆、多文件协同生成”的能力验证

特别说明：

- 当前真实工程是一个基于 **Java 17 + Spring Boot 3 + MyBatis XML + H2** 的单体工程
- 当前真实工程采用的分层结构为：

```text
Controller -> Service -> Repository -> Mapper -> MyBatis XML -> H2 Database
```

因此本案例沉淀的模板，必须以**当前真实工程结构**为准，不能脱离现有项目风格抽象出一套与实际不一致的模板。

## 2.1 当前真实工程中已存在的可复用模式

当前真实工程中，已经可以直接提炼出以下开发模式：

### 业务写接口模式
来源于：
- `CustomerController` / `CustomerServiceImpl#createCustomer`
- `AccountController` / `AccountServiceImpl#openAccount`

体现出的特征包括：

- Controller 统一接收 DTO
- 使用 `@Valid` 做参数校验
- 返回 `ApiResponse<T>`
- Service 编排业务逻辑
- Service 使用 `BizException`
- 写操作使用 `@Transactional`
- 数据访问通过 `Repository -> Mapper -> XML`
- Response DTO 与 Entity 解耦

---

### 详情查询接口模式
来源于：
- `AccountController#getAccountDetail`
- `AccountServiceImpl#getAccountDetail`

体现出的特征包括：

- 先做主对象查询
- 再做关联对象查询
- 最终组装响应 DTO
- 查询不到时统一抛 `BizException`
- 不直接返回 Entity

---

### 公共规范模式
来源于：
- `ApiResponse`
- `BizException`
- `GlobalExceptionHandler`

体现出的特征包括：

- 统一返回壳 `ApiResponse<T>`
- 统一业务异常 `BizException`
- 统一全局异常处理
- 参数校验异常统一映射为 `P0001`
- 系统异常统一映射为 `S9999`

---

### MyBatis XML 数据访问模式
来源于：
- `repository/*.java`
- `mapper/*.java`
- `resources/mapper/*.xml`

体现出的特征包括：

- Mapper 接口只定义方法
- SQL 全部写在 XML 中
- Repository 对 Mapper 做轻量封装
- Entity 为普通 POJO，不使用 JPA 注解
- XML 使用下划线转驼峰映射

## 4. 模板沉淀总原则

AI 在沉淀和复用模板时，必须遵循以下原则：

1. 基于当前真实工程抽象模板，不得脱离当前项目另起架构
2. 模板必须体现当前项目的真实分层：
  - Controller
  - Service
  - Repository
  - Mapper
  - XML
3. 模板必须体现当前项目的真实公共规范：
  - `ApiResponse`
  - `BizException`
  - `GlobalExceptionHandler`
4. 模板必须兼容当前技术栈：
  - Java 17
  - Spring Boot 3
  - Jakarta Validation
  - MyBatis XML
  - H2
5. 模板必须能指导 AI 一次生成多文件代码，而不是只生成单个类
6. 模板必须适合“AI 生成框架 + 人工补细节”的协作模式
7. 若模板涉及数据库字段变更，必须同步提示修改：
  - `schema.sql`
  - `entity`
  - `repository`
  - `mapper`
  - `mapper xml`
  - `dto`
  - `service`
  - `controller`

- `冻结/解冻/状态变更` 可作为后续扩展场景复用

---

## 5.2 当前真实工程中的模板来源

### 来源一：创建客户
- `CustomerController`
- `CustomerService`
- `CustomerServiceImpl`
- `CustomerRepository`
- `CustomerMapper`
- `CustomerMapper.xml`

### 来源二：开立账户
- `AccountController`
- `AccountService`
- `AccountServiceImpl`
- `AccountRepository`
- `AccountMapper`
- `AccountMapper.xml`

---

## 5.3 标准结构

银行标准写接口模板建议统一包含：

- Request DTO
- Response DTO
- Controller
- Service
- ServiceImpl
- Repository
- RepositoryImpl
- Mapper
- Mapper XML
- 参数校验
- 业务校验
- 统一返回 `ApiResponse`
- 统一异常 `BizException`
- Swagger 注解
- 必要时增加 `@Transactional`

---

## 5.4 模板要求

### Request DTO
必须：

- 放在 `dto.request`
- 使用 `jakarta.validation.*`
- 字段命名与当前工程风格一致
- 不直接复用 Entity

---

### Response DTO
必须：

- 放在 `dto.response`
- 不直接暴露 Entity
- 与接口返回语义对应
- 建议使用 `@Data + @Builder`

---

### Controller
必须：

- 使用 `@RestController`
- 使用 `@RequestMapping`
- 使用 `@RequiredArgsConstructor`
- 使用 `@Tag`、`@Operation`
- 返回 `ApiResponse<T>`
- 不写业务逻辑
- 不直接调用 Repository / Mapper

---

### Service
必须：

- 核心业务逻辑写在 Service 中
- 业务校验在 Service 中完成
- 业务异常统一使用 `BizException`
- 写操作根据需要加 `@Transactional`

---

### Repository
必须：

- 作为对 Mapper 的轻量封装层保留
- 不在 Service 中大面积直接访问 Mapper
- 接口能力围绕业务语义封装，而不是暴露过多数据库细节

---

### Mapper / XML
必须：

- 使用 `Mapper + XML`
- Mapper 接口只定义方法
- SQL 全部写在 XML 中
- 不使用注解 SQL
- 不在 Java 中拼接 SQL

---

### 异常与返回
必须：

- 成功返回 `ApiResponse.success(...)`
- 业务失败使用 `BizException`
- 统一走 `GlobalExceptionHandler`

---

## 5.5 标准写接口模板示意

```text
Request DTO
   ↓
Controller 接收参数并校验
   ↓
Service 编排业务逻辑
   ↓
Repository 调用 Mapper
   ↓
Mapper XML 执行 SQL
   ↓
Service 组装 Response DTO
   ↓
Controller 返回 ApiResponse<Response>
```

---

## 5.6 当前可直接复用的模板特征

AI 在后续生成类似接口时，应优先复用当前真实工程中的以下习惯：

1. Controller 层只保留一层薄封装
2. Service 层负责默认值处理与业务校验
3. `BizException("错误码", "错误信息")`
4. 写操作放 Service 层事务
5. 通过 Repository 统一访问数据
6. Response 使用 Builder 组装

## 6. 模板2：银行资金交易接口模板

## 6.1 适用场景

适用于以下接口：

- 账户存入
- 行内转账
- 扣款
- 冲正 / 撤销（后续扩展）
- 代扣 / 代发（后续扩展）

说明：

- 当前真实工程中，该模板**尚未完全落地**
- 但已在案例A、案例B详细设计中形成清晰标准
- 可作为 AI 的“标准资金接口模板”沉淀

---

## 6.2 标准结构

银行资金交易接口模板建议统一包含：

- Request DTO
- Response DTO
- Controller
- `TransactionService`
- `TransactionServiceImpl`
- `IdempotentService`
- Repository
- Mapper
- Mapper XML
- 幂等控制
- 事务控制
- 金额校验
- 账户状态校验
- 余额校验
- 账户加锁
- 流水记录
- 成功结果重放

---

## 6.3 模板要求

### 请求字段建议
请求中建议包含：

- `requestId`
- `amount`
- `remark`
- `channelCode`
- `branchCode`
- `operatorId`

根据场景增加：

- `accountNo`
- `debitAccountNo`
- `creditAccountNo`

---

### 事务要求
资金类接口必须：

```java
@Transactional
```

说明：

- 事务必须定义在 Service 层
- 不允许放在 Controller 层

---

### 幂等要求
必须：

1. 先查幂等记录
2. `SUCCESS` 支持结果重放
3. `PROCESSING` 与 `FAILED` 状态可区分
4. 成功与失败状态显式落库
5. 幂等状态建议独立事务维护

---

### 账户处理要求
必须：

- 账户查询支持锁定
- 余额更新通过 XML SQL 完成
- 双账户交易要考虑锁顺序一致
- 状态校验在更新前完成

---

### 资金校验要求
必须：

- 金额大于 0
- 账户存在
- 账户状态可用
- 冻结账户不可操作
- 借贷账户不能相同（如适用）
- 币种一致（如适用）
- 余额充足（如适用）

---

### 流水要求
必须：

- 落 `t_transaction`
- 记录交易类型与交易状态
- 记录余额快照
- 使用 `Mapper + XML` 插入流水
- 支持后续按 `requestId` 重放首次成功结果

---

## 6.4 模板流程示意

### 单边资金交易模板（如存入）
```text
参数校验
→ 幂等校验
→ 锁定账户
→ 状态校验
→ 更新余额
→ 写流水
→ 更新幂等成功
→ 返回结果
```

### 双边资金交易模板（如转账）
```text
参数校验
→ 幂等校验
→ 双账户按固定顺序加锁
→ 账户与状态校验
→ 币种与余额校验
→ 可选风控校验
→ 更新转出/转入余额
→ 写流水
→ 更新幂等成功
→ 返回结果
```

- `AccountServiceImpl#getAccountDetail`

### 可扩展来源
- 案例E：交易流水分页查询

---

## 7.3 标准结构

银行标准查询接口模板建议统一包含：

- QueryRequest DTO
- ItemResponse DTO（如列表查询）
- QueryResponse DTO
- Controller
- Service
- Repository
- Mapper
- Mapper XML 动态查询
- 条件校验
- 分页
- 排序
- 结果组装

---

## 7.4 模板要求

### 请求要求
必须支持：

- 条件参数
- 分页参数（如适用）
- 时间范围参数（如适用）
- 默认值处理

---

### 查询实现要求
必须：

- 通过数据库完成过滤
- 通过数据库完成分页
- 通过数据库完成排序
- 不允许内存全量过滤
- 不允许 Java 内存分页
- 复杂查询优先使用 XML 动态 SQL

---

### 响应要求
标准分页查询统一建议返回：

- `total`
- `pageNo`
- `pageSize`
- `records`

详情查询则直接返回业务详情 DTO。

---

### 错误处理要求
必须：

- 主对象不存在时抛 `BizException`
- 查询条件非法时抛 `BizException`
- 不直接把数据库对象原样返回

---

## 7.5 标准查询接口模板示意

### 详情查询模板
```text
接收查询参数
→ 校验主键/编号
→ 查询主对象
→ 查询关联对象
→ 组装详情 Response
→ 返回 ApiResponse
```

### 分页查询模板
```text
接收条件参数
→ 校验时间范围/分页参数
→ 校验主对象是否存在（如适用）
→ count 查询
→ list 查询
→ 结果转换
→ 组装分页 Response
→ 返回 ApiResponse
```

## 10. 模板沉淀方式建议

AI 在读取本项目时，应识别并记忆以下工程习惯：

1. 统一返回使用 `ApiResponse`
2. 业务异常统一使用 `BizException`
3. DTO 按 `request / response` 分包
4. Service 与 ServiceImpl 分离
5. Repository 作为轻量封装层保留
6. MyBatis Mapper + XML 风格优先
7. Entity 为普通 POJO，不使用 JPA 注解
8. 金额使用 `BigDecimal`
9. 时间使用 `LocalDateTime`
10. Controller 只负责接收与返回，不写业务逻辑
11. 写接口要考虑参数校验、业务校验、事务
12. 资金接口要考虑幂等、锁、流水、余额快照
13. 查询接口要考虑数据库分页、排序、动态 SQL
14. 所有表结构变更要同步到 XML / Entity / DTO / Service
15. 健康检查 `/health` 是非业务接口特例，不作为业务模板标准

## 12. AI 使用模板时的约束要求

AI 在使用模板时必须：

1. 复用当前真实工程风格，而不是重新设计一套
2. 保持统一返回结构
3. 保持统一异常结构
4. 保持现有包结构
5. 保留 Repository 层
6. 所有数据库访问统一使用 Mapper + XML
7. 不引入额外复杂框架
8. 不得引入 JPA Repository、Specification、MyBatis-Plus 等替代方案
9. 输出完整代码，而不是只给模板描述
10. 若为增量改造，必须明确列出涉及哪些文件变更
11. 若模板涉及数据库字段新增，必须同步提示修改 `schema.sql`
12. 若模板涉及已有查询或写流程联动，必须提示联动影响范围
