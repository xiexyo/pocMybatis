## 1. 文档目的

本文档用于指导 AI Coding 插件基于当前已完成的零售银行账户服务工程，按照既定案例逐步增量生成完整代码。

本文档覆盖以下案例：

### 核心案例
1. 案例A：账户入金接口
2. 案例B：行内转账接口 + 幂等控制

5. 案例E：交易流水查询接口 + 查询优化

### 增强案例

7. 案例G：银行标准接口模板沉淀与复用

本文档设计目标是：**让 AI 在读取本文档后，能够按照当前项目现有风格，增量生成高质量、可编译、可运行的代码。**

## 4. 全局错误码定义

建议 AI 统一使用以下错误码。

### 通用
- `B0000`：成功
- `P0001`：参数校验失败
- `S9999`：系统异常

### 客户类
- `CUST404`：客户不存在
- `CUST001`：客户证件号码已存在

### 产品类
- `PROD404`：产品不存在
- `PROD001`：产品当前不可售

### 账户类
- `ACCT404`：账户不存在
- `ACCT001`：账户状态不可用
- `ACCT002`：账户余额不足
- `ACCT003`：转出账户和转入账户不能相同
- `ACCT004`：账户币种不一致
- `ACCT005`：账户已冻结
- `ACCT006`：账户已存在一类户，不能重复开立

### 交易类
- `TXN001`：交易金额必须大于0
- `TXN002`：开始时间不能大于结束时间
- `TXN003`：交易查询条件非法

### 幂等类
- `IDEMP001`：请求正在处理中
- `IDEMP002`：请求已成功处理
- `IDEMP003`：请求已失败，请更换requestId重试

### 风控类（增强案例）
- `RISK001`：风险校验未通过

# 10. 案例A详细设计：账户入金接口

## 10.1 目标
在当前工程基础上新增资金类接口，支持账户入金、交易落表、幂等控制。

---

## 10.2 API 设计

### 10.2.1 路径
```http
POST /api/v1/transactions/deposit
```

### 10.2.2 请求 DTO
类名：
```java
DepositRequest
```

字段定义：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| requestId | String | 是 | 请求唯一号 |
| accountNo | String | 是 | 入金账户号 |
| amount | BigDecimal | 是 | 入金金额 |
| remark | String | 否 | 备注 |
| channelCode | String | 否 | 渠道号，可选增强字段 |
| branchCode | String | 否 | 机构号，可选增强字段 |
| operatorId | String | 否 | 操作员号，可选增强字段 |

### 10.2.3 请求校验
- `requestId`：`@NotBlank`
- `accountNo`：`@NotBlank`
- `amount`：`@NotNull`
- `remark`：可为空，但长度不得超过 255（可在 Service 或注解约束）

### 10.2.4 响应 DTO
类名：
```java
DepositResponse
```

字段定义：

| 字段 | 类型 | 说明 |
|---|---|---|
| requestId | String | 请求唯一号 |
| txnId | String | 交易流水号 |
| accountNo | String | 入金账户号 |
| amount | BigDecimal | 入金金额 |
| balanceAfter | BigDecimal | 入金后余额 |
| txnStatus | String | 交易状态 |
| txnTime | LocalDateTime | 交易时间 |

---

## 10.3 Controller 设计

### 类名
```java
TransactionController
```

### 方法签名建议
```java
@PostMapping("/deposit")
public ApiResponse<DepositResponse> deposit(@Valid @RequestBody DepositRequest request)
```

### 职责
- 接收请求
- 参数校验
- 调用 `transactionService.deposit`
- 返回 `ApiResponse.success(...)`

---

## 10.4 Service 设计

### 方法签名
```java
@Transactional
public DepositResponse deposit(DepositRequest request)
```

### 业务处理步骤
1. 校验 `amount > 0`
2. 查询幂等记录：
    - 若存在且 `SUCCESS`：直接根据 `requestId` 查询交易流水并返回首次结果
    - 若存在且 `PROCESSING`：抛 `BizException("IDEMP001", "请求正在处理中")`
    - 若存在且 `FAILED`：抛 `BizException("IDEMP003", "请求已失败，请更换requestId重试")`
3. 若幂等记录不存在，则调用 `idempotentService.init(requestId, "DEPOSIT", accountNo)`
4. 使用 `accountRepository.lockByAccountNo(accountNo)` 加锁查询账户
5. 账户不存在则：
    - `idempotentService.markFailed(...)`
    - 抛 `BizException("ACCT404", "账户不存在")`
6. 账户状态必须为 `ACTIVE`
7. 若账户状态为 `FROZEN` 或非 `ACTIVE`，抛 `BizException("ACCT001", "账户状态不可用")`
8. 计算新余额：
    - `balanceAfter = account.balance + amount`
9. 更新账户：
    - `balance = balanceAfter`
    - `updatedTime = now`
10. 生成交易流水：
- `txnType = "DEPOSIT"`
- `txnStatus = "SUCCESS"`
- `debitAccountNo = null`
- `creditAccountNo = accountNo`
- `debitBalanceAfter = null`
- `creditBalanceAfter = balanceAfter`
11. 保存交易流水
12. 调用 `idempotentService.markSuccess(...)`
13. 返回 `DepositResponse`

### 异常处理
- 业务异常：在抛出前标记幂等失败
- 系统异常：catch 后标记幂等失败，再抛出

---

## 10.5 重复请求结果重放规则

若 `requestId` 已成功处理：
- 通过 `transactionRepository.findByRequestId(requestId)` 查询交易流水
- 返回：

```text
requestId = transaction.requestId
txnId = transaction.txnId
accountNo = transaction.creditAccountNo
amount = transaction.amount
balanceAfter = transaction.creditBalanceAfter
txnStatus = transaction.txnStatus
txnTime = transaction.txnTime
```

---

## 10.6 Repository 使用要求
- `AccountRepository.lockByAccountNo`
- `TransactionRepository.findByRequestId`
- `IdempotentRecordRepository.findByRequestIdAndBusinessType`

---

2. 账户不存在
3. 账户冻结
4. amount <= 0
5. 相同 requestId 重复提交
6. requestId 正在处理中

---

## 11.3 Controller 设计

### 方法签名建议
```java
@PostMapping("/internal-transfer")
public ApiResponse<InternalTransferResponse> internalTransfer(@Valid @RequestBody InternalTransferRequest request)
```

---

## 11.4 Service 设计

### 方法签名
```java
@Transactional
public InternalTransferResponse internalTransfer(InternalTransferRequest request)
```

### 业务处理步骤
1. 校验 `amount > 0`
2. 校验 `debitAccountNo != creditAccountNo`
3. 查询幂等记录：
    - `SUCCESS`：返回首次成功结果
    - `PROCESSING`：抛 `IDEMP001`
    - `FAILED`：抛 `IDEMP003`
4. 若幂等记录不存在，则初始化：
    - `idempotentService.init(requestId, "INTERNAL_TRANSFER", debitAccountNo + "->" + creditAccountNo)`
5. 为避免死锁，按账户号字典序排序：
    - `firstAccountNo = min(debitAccountNo, creditAccountNo)`
    - `secondAccountNo = max(debitAccountNo, creditAccountNo)`
6. 依次调用 `accountRepository.lockByAccountNo` 获取两个账户
7. 获取实际转出账户、转入账户对象
8. 校验两个账户都存在
9. 校验两个账户状态均为 `ACTIVE`
10. 若任一账户为 `FROZEN`，抛 `ACCT005`
11. 校验两个账户币种一致
12. 校验转出账户余额充足：
- `debit.balance.compareTo(amount) >= 0`
13. 计算：
- `debitBalanceAfter = debit.balance - amount`
- `creditBalanceAfter = credit.balance + amount`
14. 更新转出账户余额与更新时间
15. 更新转入账户余额与更新时间
16. 生成交易流水：
- `txnType = "INTERNAL_TRANSFER"`
- `txnStatus = "SUCCESS"`
- `debitAccountNo = debitAccountNo`
- `creditAccountNo = creditAccountNo`
- `debitBalanceAfter = debitBalanceAfter`
- `creditBalanceAfter = creditBalanceAfter`
17. 保存交易流水
18. 调用 `idempotentService.markSuccess(...)`
19. 返回 `InternalTransferResponse`

### 异常处理
任何业务异常或系统异常发生时：
- 调用 `idempotentService.markFailed(...)`
- 再抛出异常

### 方法签名建议
```java

```

---

## 14.4 Service 设计

### 方法签名
```java
public TransactionQueryResponse queryTransactions(TransactionQueryRequest request)
```

### 业务处理步骤
1. 校验 `accountNo` 必填
2. 校验账户存在
3. 校验 `pageNo` 默认值为 1
4. 校验 `pageSize` 默认值为 20，若大于 100 则取 100 或抛异常
5. 若 `startTime` 和 `endTime` 同时存在，则要求 `startTime <= endTime`
6. 使用 `Specification<TransactionEntity>` 动态构造查询条件：
    - 当前账户作为借方或贷方
    - 若 `txnType` 非空则按类型过滤
    - 若 `startTime` 非空则 `txnTime >= startTime`
    - 若 `endTime` 非空则 `txnTime <= endTime`
7. 使用 `PageRequest.of(pageNo - 1, pageSize, Sort.by(DESC, "txnTime"))`
8. 查询结果转换为 `TransactionItemResponse`
9. 返回分页结构

### 视角转换规则
若 `accountNo == debitAccountNo`
- `direction = "DEBIT"`
- `counterpartyAccountNo = creditAccountNo`
- `balanceAfter = debitBalanceAfter`

若 `accountNo == creditAccountNo`
- `direction = "CREDIT"`
- `counterpartyAccountNo = debitAccountNo`
- `balanceAfter = creditBalanceAfter`

若入金交易：
- `debitAccountNo = null`
- `direction = "CREDIT"`
- `counterpartyAccountNo = null`
- `balanceAfter = creditBalanceAfter`
